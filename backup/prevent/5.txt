부서: DS N/W SRE그룹(클라우드N/W서비스)
내용: DS부문 네트워크 운영 자동화 기능 개발

1. DS 네트워크 자동화의 필요성
- 최근 SDN(Software Defined Network)으로 분류되어 판매되는 글로벌 제조사(CISCO, Arista, HPE, Extreme 등)들의 제품에는 기본적으로 Controller 솔루션이 포함되어 제공됨
- 해당 Controller 솔루션은 초기 구축부터 운영 단계에서의 다양한 자동화 기능(신규 장비 자동 인식 및 초기 설정 자동 배포, 다수 장비 일괄 설정 배포 등)을 제공하고 있으나 벤더 종속적임
- 현재 그룹 관계사 네트워크에는 SDN 기술(논리적 망분리, IP mobility 등)을 필요로 하는 오피스 영역 위주로 적용 중에 있음
  -> 전자 DX부문 수원 R3(완료)/우면 캠퍼스(확산 적용중), 디스플레이 기흥연구소(SDR)(완료), 삼성SDS 판교IT캠퍼스(완료), 금융권 오피스망(적용 검토중)
- Cloud(SDS SCP, Public Cloud 등)로 전환되지 않은 그룹 관계사의 중요 업무(DS MES 포함)는 Legacy 방식의 기존 네트워크 환경을 유지하고 있음
- DS 네트워크 운영 규모는 매년 지속적인 증가를 유지해왔고, 이제는 자동화 없이는 운영이 어려운 대규모 환경으로 성장하였음.

* 2010년 12월: 장비수 1490대, 2018년 12월: 장비수 8105대, 2025년 6월: 9893대

- 오픈소스 기반의 솔루션들을 활용하여 벤더에 종속되지 않는 운영 업무에 필수적인 아래 기능들 위주로 자동화 기능을 개발하여 활용하고 있음
|항목|필요기능|
|---|---|
|장비정보조회|다수 네트워크 장비에 대한 CLI 명령어기반 일괄정보조회, NMS자동수집불가항목에 대한 명령어기반의 정보조회 및 추이분석, 다양한 정보(단말정보, 네트워크연결정보 등)를 연계하여 단일화면에서 통합조회, NMS등 수집된 네트워크 장비 연결 정보기반으로 구성도를 자동으로 작성/업데이트|
|변경작업수행|다수의 네트워크장비에 대한 설정 동시 배포기능, OS 업그레이드 등 정형화된 변경작업 수행(복잡도가 높고 영향도가 큰작업에 대해서는 보조기능으로만 활용), 변경작업전후 네트워크 장비 상태정보와 Config 비교및 차이점 분석|
|이벤트 감지/자동알람|NMS Critical 이베느에 대한 챗봇/이벤트봇을 통한 자동메신저 및 메일링 기능, 이벤트 분석에 필요한 1차정보 자동 수집 및 공유|

2. DS 네트워크 자동화 소개
가. 네트워크 자동화 개발/구현 현황
|명칭|개발또는적용배경|적용시효과|
|---|---|---|
|작업/조회 자동화(Pretty Ansible)|장비정보조회, 변경작업 등 단순 반복업무 효율화 필요, 점검자동화를 통한안정적인 변경작업수행 필요|단순조회작업(표 자동변환 또는 raw text), 작업전후 자동점검 및 비교(표준설정, 운영상태, 포트상태), 작업자동화(같은 작업에서 동일명령어, 장비별 명령어등 모두 지원)|
|데이터분석시스템(Splunk)|데이터저장 및 분석/시각화 솔루션 필요|데이터분석플랫폼/모든데이터기반으로 다양한 분서가능, 장애발생시 네트워크 One-view를 통한 정보 확인, 그외 daily 점검 및 정보조회를 위한 70여개 대시보드 활용중|
|구성도 관리솔루션(Diagram Viewer)|구성도가 수동관리되어 정합성이 낮음|Web 기반으로 상시 구성도 조회 및 편집 가능, NMS CI정보 자동연계를 통한 정합성관리|
|이벤트봇/챗봇|야간 및 주말 등 취약시간대에도 빠른 2선 지원 필요|메신저를 통한 상시 장비정보, NMS, 구성도 등 조회, 메신저를 통한 단순조회작업 수행, 이벤트 발생시 장비상태 및 관련 정보 메일로 즉시 확인|

그외 솔루션
- IRIS: 장비라우팅 정보 기반 장확한 IP 대역관리, 예약/반납기능을 통한 IP 중복장애발생 방지
- NCV: 계획서 내 명령어 작성 오류 검증 (예: interface, address 등)
- Flow 분석솔루션(Elastiflow): 주요장비에서 사용량 기준으로 Flow 분석 가능, 사용량 기반의 모니터링 및 문제분석 가능
- 네트워크 속도측정(Speedtest): 접속PC <-> 서버간 NW 속도측정
- RestAPI Gateway: API 리스트 및 테스트 수행
- 컨테이너 관리(Portainer CE): 개발된 솔루션들의 이미지 및 컨테이너를 GUI 기반으로 관리 가능
- 서비스 모니터링 솔루션(Uptime kuma): 자체 서비스(Ansible, Splunk 등) 모니터링 및 메일/메신저 알림, 고객 중요 서비스 모니터링 및 메일/메신저 알림
- 데이터 동기화 솔루션(Syncthing): 안정적인 자체 개발 서비스 운영
- 데이터베이스(PostgreSQL): 고성능/고가용성 기반의 안정적인 서비스 운영가능, 스트리밍 복제를 통한 준실시간 데이터 백업 가능
- LowCode 앱(Buildbase): 복잡한 코딩을 최대한 줄이면서 필요한 App 개발 가능, GBIC 자산관리 자동화 App 개발
- LowCode 워크플로(n8n): Buildbase/Splunk와 연계하여 Workflow 구현가능, 다양한 App 개발환경 구성

나. Pretty Ansible (작업 자동화 솔루션) 개발 배경
- 기존에서 CLI 기반의 Ansible을 Web 기반으로 사용하기 위한 솔루션은 존재하였음 (Ansbile AWX/Tower)
- 단, 기존 솔루션은 사용자별 권한관리, Workflow, 스케줄링, API 등에 집중된 솔루션으로 스크립트나 자동화, 코드 작성에 익숙하지 않은 네트워크 운영자들이 사용하기에는 부적합한 솔루션으로 판단하여 자체 개발
|주요기능|Ansible AWX/Tower|Pretty Ansible|비고|
|---|---|---|---|
|신규작업생성|미지원|지원|Pretty Ansible 핵심기능<br>AWS/Tower: 스크립트를 CLI 등 별도 환경에서 YML로 직접 작성-> 문제발생가능성 매우높음<br>Pretty Ansible: 사전 구성된 Workflow 기반으로 작업명/접속방식/접속계정/수행명령어 등만 입력|
|사용자 관리|지원|지원|Pretty Ansible 내 통합 인증 구현|
|스케줄링|지원|미지원(외부)|별도 Python기반으로 운영 -> Pretty Ansible의 복잡성 감소|
|동적 인벤토리 관리|지원|지원|NMS 서버에서 장비정보 자동 연동|
|Workflow 생성|지원|미지원(사전구성)|필요한 workflow는 미리작성, Pretty Ansible에서는 검증된 Workflow를 미리 구성후 대상 장비/명령어만 GUI로 입력받아 신규 작업을 생성|
|REST API|지원|지원|조회 기능만으로 제한|

다. Pretty Ansible 핵심 동작 구조
1) 아키텍처 구조
작업요청: 운영자-> WEB/챗봇(API)/스케줄러(CLI)-> 단순조회/변경, 자동점검 -> Ansible 자동화 작업영역
Ansible 자동화 작업영역: 장비접속모듈/점검모듈/가공모듈1/가공모듈2/비교모듈 -> 결과확인
결과확인: 파일기반 결과확인, RDBMS 데이터 연동, 분석솔루션데이터연동(One-View)

2) 장비접속모듈: Raw data -> 데이터 추출 모듈
3) 점검모듈: 운영상태(PIR), 표준설정(가용성) 자동 점검 모듈
4) 가공모듈: JSON -> CSV 단순 변환, 전체 결과 병합(CSV/XLSX)
5) 비교모듈: 사전/사후 점검결과를 비교, 다른 부분만 강조

라. Pretty Ansible 작업 활용
1) 사전 점검(PIR/가용성/백업/포트맵)
PIR: 불량모듈은 없는지, 라우팅 연동은 잘되어있는지?
가용성: 표준설정대로 적용되었는지, SSI 보안기준을 충족하는지?
백업: 이슈분석을 위해 필요한 다양한 점검 결과
포트맵: 연결속도가 변경된게 있는지, 사라진IP가 있는지?

2) 변경작업(동일명령어/장비별 명령어)
동일명령어: 다수의 장비에 동일한 명령어를 적용하는 기능
장비별 명령어: 다수의 장비에 다른 명령어를 적용(포트명, IP 등)하는 기능

마. Splunk 활용 - 네트워크 One-View
-네트워크 전체 현황과 연결장비 및 서버 정보를 한눈에 볼 수 있는 대시보드

바. 구성도 관리솔루션
- 구성도 디자인은 각 부서별로 원하는 형식으로 자유롭게 작성 가능
- 장비정보는 장비명, 회선정보는 포트명만 작성, 그외의 모든 정보는 CI기준 자동 업데이트

사. 이벤트봇/챗봇
- 이벤트봇: 특이로그 감지 시 장비에 즉시 접속, 공통적으로 확인하는 명령어를 수행하여 자동 메일링
- 챗봇: 공통 명령어외 추가 확인이 필요한 확인 명령어, 장비 관련정보, IP차단여부, 구성도등 확인
  . 최근 1시간내 이슈발생장비는 Dropdown 형태로 제공
  . 위의 시스템(Ansible/Splunk/구성도 관리솔루션 등)들이 모두 API를 활용하여 챗봇과 연계되어있음
