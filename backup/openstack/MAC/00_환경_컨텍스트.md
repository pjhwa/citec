# 환경 컨텍스트 (Canonical Context)

> 본 문서는 본 교육 시리즈 전체의 **고정 가정** 을 정의한다. 모든 편(1편 ~ 10편)은 별도 명시가 없는 한 본 문서의 환경을 전제로 작성·검증된다.
>
> **문서 갱신 정책**: 환경 변경(예: OpenStack 마이너 업그레이드, 컨테이너 이미지 태그 갱신)이 발생하면 본 문서를 먼저 갱신하고, 이후 작성되는 편은 갱신 시점 이후 컨텍스트를 따른다. 과거 편은 작성 시점의 컨텍스트를 보존한다.

## 2. 환경 항목별 상세

### 2.1 운영체제 — Ubuntu 24.04 LTS (Noble Numbat)

- 컴퓨트 노드 호스트 OS는 **Ubuntu 24.04 LTS** 로 통일.
- 커널은 24.04 기본 GA 커널 또는 HWE 커널을 가정. LSM은 빌트인 AppArmor가 primary.
- AppArmor 사용자 공간 도구는 패키지 `apparmor` / `apparmor-utils` / `apparmor-profiles-extra`. 버전대는 `4.0.1-0ubuntu0.24.04.x` 계열.
- `kernel.apparmor_restrict_unprivileged_userns = 1` 이 기본값. 시리즈 6·7편에서 이 값을 어떻게 다룰지 명시한다.
- libvirt 컨테이너 *내부* 에도 Ubuntu 24.04(noble) 베이스가 들어있다(이미지 태그 명시) — 즉 호스트와 컨테이너 양쪽 모두 noble. 그러나 LSM은 호스트 커널에서 작동하므로 **정책 적재의 주체는 항상 호스트 커널** 이라는 점을 잊지 말 것.

### 2.2 OpenStack 2025.1 (Epoxy)

- 2025년 4월 릴리스. SLURP(Skip Level Upgrade Release Process) 비대상 일반 릴리스.
- Nova/Cinder/Neutron/Glance 등 본 시리즈에서 다루는 핵심 서비스가 모두 Epoxy 기준.
- libvirt API 버전·QEMU 버전은 컨테이너 이미지에 동봉된 패키지에 의해 결정. 이미지 빌드 시점의 noble 패키지 셋과 일치한다고 가정.
- nova-compute의 `[libvirt]` 섹션에서 `virt_type = kvm`, `images_type = rbd` (Ceph RBD ephemeral) 가정.

### 2.3 배포 도구 — Kustomize + ArgoCD + OpenStack-Helm 차트

본 시리즈는 **"OpenStack-Helm 차트의 values를 Helm CLI로 직접 적용하지 않는다"** 를 전제로 한다. 흐름은 다음 두 가지 중 하나로 정형화된다.

- **방식 A (Argo CD가 Helm + Kustomize 결합 직접 처리)**
  - ArgoCD Application의 source가 Helm chart이고, 그 결과를 `kustomize` 후처리.
  - 또는 두 개의 Application을 둠: 하나는 Helm 렌더링, 하나는 Kustomize 오버레이 + 정책 patch.
- **방식 B (사전 렌더링)**
  - CI에서 `helm template openstack-helm/<chart> -f values.yaml > base/`
  - 그 결과 위에 Kustomize 오버레이를 두고 Git 커밋
  - ArgoCD는 Kustomize 소스만 본다.

**시리즈에서의 정책**: 별도 명시가 없는 한 **방식 B(사전 렌더링)** 를 기본 설명 모델로 채택한다. 사이트별로 방식 A를 쓰는 경우 7편에서 별도 절로 다룬다.

| 디렉터리(예시) | 역할 |
|---|---|
| `charts/` | OpenStack-Helm chart 서브모듈 또는 vendored 사본 |
| `values/<env>/` | 환경별 Helm values (dev / staging / prod) |
| `base/<chart>/` | `helm template`으로 사전 렌더링된 매니페스트 |
| `overlays/<env>/<chart>/` | Kustomize 오버레이 (patches + ConfigMap·Secret 추가) |
| `policies/apparmor/` | AppArmor 정책 코드(원본·TEMPLATE override 포함) |
| `argo-apps/` | ArgoCD Application 정의 |

ArgoCD 운영 가정: **App-of-Apps 패턴**, sync wave로 정책 → libvirt → nova-compute 순서 보장, prune·selfHeal 활성화. 자세한 GitOps 설계는 7편에서 다룬다.

### 2.4 컴퓨트 노드 런타임 — DaemonSet 두 개

- `libvirt` DaemonSet:
  - 이미지: `docker.io/openstackhelm/libvirt:2025.1-ubuntu_noble`
  - 컨테이너 내부에 `libvirtd`, `virtlogd`, `virtlockd` 동작
  - `hostNetwork`, `hostPID`, `hostIPC` 활성. `/dev`, `/sys/fs/cgroup`, `/run/libvirt`, `/var/lib/libvirt`, `/var/lib/nova` 등을 hostPath 마운트.
- `nova-compute` DaemonSet:
  - openstack-helm `nova` chart의 nova-compute Pod
  - libvirt와 IPC(Unix 소켓) 통신
  - `/var/lib/nova/instances/<uuid>/...` 경로 패턴

**AppArmor 관점 핵심**: libvirt 컨테이너 안의 `apparmor_parser`가 *호스트 커널의 securityfs* 에 정책을 적재해야 한다. 이를 위해 다음이 필요하다.

- hostPath: `/sys/kernel/security` (rw) — securityfs 인터페이스
- hostPath: `/etc/apparmor.d` (rw) — 정책 텍스트, `libvirt/libvirt-<UUID>` 동적 프로파일이 쓰여지는 경로
- 적절한 capabilities (`SYS_ADMIN`, `MAC_ADMIN`) 또는 `privileged: true`
- AppArmor 측 컨테이너 자체 프로파일 — 보통 `unconfined` 또는 사이트 작성 프로파일

이 통합 설계의 상세는 6편에서 다룬다.

### 2.5 스토리지 — Rook-Ceph (RBD)

- Ceph 클러스터는 **Rook-Ceph 오퍼레이터**가 Kubernetes 위에서 운영.
- OpenStack 인스턴스 디스크와 Cinder 볼륨은 모두 **Ceph RBD** 를 사용.
- AppArmor 정책 관점에서의 함의:
  - 게스트별 동적 프로파일이 보호해야 할 대상은 **로컬 파일 경로가 아니라 RBD 디바이스 / RBD 클라이언트 라이브러리 호출 경로**.
  - 이미지·볼륨이 호스트에 마운트되는 형태가 아니라 QEMU가 librbd를 직접 호출(direct librbd) 하는 경로가 일반적.
  - 따라서 정책에는 다음 자원이 명시되어야 한다.
    - `/etc/ceph/*.conf`, `/etc/ceph/*.keyring` (또는 컨테이너 안 동등 경로) — read
    - `/var/lib/nova/instances/_base/...` — read (이미지 캐시는 RBD 직접 사용 시 미사용일 수 있음)
    - `/dev/rbd*` — direct kernel RBD 사용 시
    - `/run/ceph/...` — admin socket
  - `virt-aa-helper`가 RBD 디스크 항목을 인식하여 동적 프로파일에 반영하도록, libvirt 도메인 XML이 `<disk type='network' protocol='rbd'>` 형태인지 확인하는 절차를 4편·6편에서 다룬다.

### 2.6 가상화 — KVM + QEMU + sVirt-AppArmor

- nova-compute `[libvirt] virt_type = kvm`.
- libvirt `qemu.conf`에 `security_driver = "apparmor"` (시리즈에서 활성화).
- 게스트별 동적 프로파일은 호스트의 `/etc/apparmor.d/libvirt/libvirt-<UUID>` 에 생성되며, 베이스가 되는 TEMPLATE은 `/etc/apparmor.d/libvirt/TEMPLATE.qemu`.

## 3. 시리즈 작성 시 적용 규칙

이 환경 컨텍스트가 각 편의 작성 방식에 끼치는 구체적인 규칙을 정리한다.

### 3.1 모든 편 공통

- 명령어 예제의 호스트 OS·패키지 버전은 Ubuntu 24.04 / AppArmor 4.0.x 기준으로 작성.
- 호스트 측 작업과 컨테이너 측 작업을 명시적으로 구분(예: "**호스트 셸에서 실행**", "**libvirt 컨테이너 내부에서 실행**").
- "Helm install" 같은 표현 대신 **"Kustomize 오버레이에서 …", "ArgoCD Application의 values에서 …"** 표현 사용.
- 스토리지 예시는 Rook-Ceph RBD를 우선 사용. 로컬 디스크(qcow2 파일) 예시가 필요하면 학습용 부록으로만 처리.

### 3.2 편별 영향

| 편 | 본 환경에 따라 보강·조정될 핵심 |
|---|---|
| 1편 | 변경 없음 (개념 위주). 환경 인용은 9장 sVirt 섹션에서만 발생. |
| 2편 | 실습 환경은 단일 VM. 환경 컨텍스트 의존도 낮음. 단, 향후 통합편의 사전 학습 의도임을 명시. |
| 3편 | 위협 모델은 일반론이지만 "RBD 환경에서의 자산 식별"을 케이스로 포함. |
| 4편 | **단일 호스트 KVM이지만 Rook-Ceph RBD 가정**으로 정책 보강. 사이트 TEMPLATE 예시는 RBD 디스크 항목 포함. |
| 5편 | OpenStack-Helm + Kustomize + ArgoCD 컴포넌트 맵을 정확하게 반영. libvirt 이미지 태그 그대로 인용. |
| 6편 | hostPath 마운트, securityContext, capabilities 설계는 OpenStack-Helm `libvirt` chart 기준 values + Kustomize patch로 표현. |
| 7편 | "Helm install" 흐름이 아니라 **Kustomize 오버레이 + ArgoCD Application** 의 GitOps 흐름으로 통일. 정책 sync wave 설계 포함. |
| 8편 | 양성 시나리오에 **Rook-Ceph RBD 볼륨 attach·detach·snapshot**, **DaemonSet rollout·restart 시 정책 유지** 케이스 포함. |
| 9편 | 모니터링 스택은 Kubernetes 친화(Prometheus Operator + Loki/Promtail) 가정. ArgoCD drift 감지와 정책 drift 감지를 분리해 다룸. |
| 10편 | FAQ에 "ArgoCD가 정책 파일을 prune해버렸다", "Helm 차트 업그레이드로 libvirt DaemonSet이 재생성되며 동적 프로파일이 사라졌다" 같은 사이트 특화 케이스 포함. |

### 3.3 명명·표기 컨벤션

- 컨테이너 이미지: 처음 인용 시 풀 태그 사용(`docker.io/openstackhelm/libvirt:2025.1-ubuntu_noble`). 이후 약칭 시 "libvirt 컨테이너" 로 통일.
- DaemonSet 명: 본 시리즈는 OpenStack-Helm 기본값을 따른다고 가정 — `libvirt`, `nova-compute`. 사이트별로 prefix가 붙는 경우(`prod-libvirt` 등) 그대로 인용하지 않는다.
- 정책 파일명: 사이트 변경분은 `/etc/apparmor.d/local/<original-filename>` 에 두는 규약을 시리즈 전반에서 통일.

## 4. 환경 가정에서 *벗어날* 경우의 처리

본 컨텍스트와 다른 환경에서 자료를 사용·재현하려는 독자를 위해, 각 편 끝에 "다른 환경에서의 차이" 짧은 박스를 둔다. 다음을 다룬다.

- OpenStack-Helm이 아니라 OpenStack-Ansible / Kolla-Ansible을 쓰는 경우의 차이.
- Ceph가 아니라 LVM/iSCSI/NFS 백엔드인 경우의 정책 항목 변화.
- libvirt 컨테이너가 아니라 호스트 OS에 직접 패키지로 설치된 경우.
- ArgoCD가 아니라 Flux를 쓰는 경우.

이 차이 분석은 시리즈의 본문이 *현 환경에 집중하면서도* 다른 환경 운영자에게 학습 자료로 활용 가능하도록 한다.
