### 2편. AppArmor 프로파일 작성·운영 실습

**학습 목표**: 임의의 사용자 데몬 한 개에 대해 0에서 시작해 enforce까지 가져갈 수 있게 된다.

1. 학습 목표와 사전 준비
   - 1.1 실습용 단일 VM 환경 구성 (Multipass / KVM)
   - 1.2 실습 대상 데몬 선정 (예: `mini-fileserver`, 직접 작성한 작은 Python 데몬)
   - 1.3 실습 패키지 설치 (`apparmor-utils`, `auditd`)
2. 라이브 데모: 기본 분석 도구 손에 익히기
   - 2.1 `aa-status`, `/sys/kernel/security/apparmor/profiles` 비교
   - 2.2 `aa-unconfined`로 보호되지 않은 프로세스 식별
3. 첫 프로파일 - 빈 골격에서 시작하기
   - 3.1 `aa-autodep`으로 골격 생성
   - 3.2 골격을 손으로 읽어 보기 (헤더, abstractions/base, 빈 본문)
   - 3.3 complain 모드 적재
4. 학습 사이클 (complain → logprof → enforce)
   - 4.1 시나리오 정의 ("정상 사용 흐름" = X·Y·Z)
   - 4.2 시나리오 1회 실행 → 로그 수집
   - 4.3 `aa-logprof`로 한 줄씩 정책에 반영하기 (Allow/Glob/Inherit/Deny 선택 기준)
   - 4.4 반복 후 enforce 전환
5. 정책 리팩토링 패턴
   - 5.1 abstractions로 공통 묶기
   - 5.2 owner / glob / 변수(`@{HOME}`) 사용법
   - 5.3 `local/` 분리 — 사이트 패치 보존 패턴
6. 정책의 단위 테스트 (CI에서 검증)
   - 6.1 `apparmor_parser -QT` 문법 검증
   - 6.2 `--warn=rules-not-enforced` `--warn=rule-downgraded` 활용
   - 6.3 GitHub Actions / GitLab CI 샘플 워크플로
7. 시스템 통합 테스트
   - 7.1 정상 시나리오 자동화 (bats / pytest)
   - 7.2 거부 케이스 자동화 (의도적 위반이 정말 막히는지)
   - 7.3 로그 검증 (`ausearch` 자동화)
8. 자주 발생하는 시행착오와 해결
   - 8.1 mmap 거부 → `abstractions/base` 누락
   - 8.2 경로 와일드카드 빗나감 → bind mount/심링크 함정
   - 8.3 Px 전이 실패 → 자식 프로파일 부재
   - 8.4 silently ignored rules → ABI 또는 커널 features 미스매치
9. 점검 질문 + 다음 편 예고

## Phase 2 — 가상화 보안 맥락

### 3편. KVM/QEMU 위협 모델과 sVirt 개념

**학습 목표**: "왜 가상화 호스트에 MAC가 필요한가"를 위협 모델로 이해하고, sVirt가 정책으로 어떻게 응답하는지 설명할 수 있다.

1. 가상화 보안의 출발점
   - 1.1 KVM·QEMU 아키텍처 빠르게 (vCPU, vfio, virtio)
   - 1.2 게스트 ↔ 호스트 ↔ 다른 게스트 간 신뢰 경계
2. 위협 모델 (STRIDE 적용)
   - 2.1 자산: 게스트 디스크, vTPM, 호스트 커널, 다른 테넌트 데이터
   - 2.2 위협 행위자: 악의적 게스트, 제3자 공격자, 내부자
   - 2.3 공격 표면: QEMU 디바이스 모델, virtio 백엔드, vhost-user 소켓, 공유 메모리
3. 실제 사례 분석 (간단 정리, 학습 목적)
   - 3.1 Venom (CVE-2015-3456) 스타일 디바이스 에뮬레이션 취약점
   - 3.2 vhost-net / virtiofsd 사례
   - 3.3 사이드 채널 (Spectre 계열) — MAC가 닿는 영역과 닿지 않는 영역의 구분
4. 방어 깊이(Defense in Depth) 전략
   - 4.1 minimal QEMU build, seccomp, namespace, MAC
   - 4.2 각 계층의 한계와 MAC가 메우는 부분
5. sVirt — 가상화에 특화된 MAC 활용 패턴
   - 5.1 "정적 프로파일 + 게스트별 동적 프로파일"의 발상
   - 5.2 sVirt-AppArmor vs sVirt-SELinux 비교
   - 5.3 라벨 모델 vs 경로 모델 — Ubuntu가 AppArmor를 택한 이유
6. libvirt 보안 모델 일반론
   - 6.1 신뢰 경계 (libvirtd 신뢰, QEMU 불신)
   - 6.2 capability separation, drop privileges
   - 6.3 secret 관리(QEMU 암호화 디스크 등)와 MAC의 관계
7. 정책 설계 사고법
   - 7.1 "기본 거부, 명시 허용" 원칙
   - 7.2 자원 단위 식별: 어떤 경로·소켓·시그널이 게스트마다 다른가
   - 7.3 동적 vs 정적의 경계 결정
8. 점검 질문 + 다음 편 예고

### 4편. libvirt sVirt-AppArmor 단일 호스트 적용

**학습 목표**: 베어메탈 KVM 호스트 한 대에서 sVirt-AppArmor를 활성화·검증하고, 디바이스가 추가될 때 정책을 어떻게 보강하는지 손에 익힌다. **본 시리즈의 최종 환경(컨테이너화 + Rook-Ceph RBD) 적용 전 단계로, 같은 정책 항목을 단순한 환경에서 미리 익히는 것이 목적**.

1. 사전 점검과 호스트 준비
   - 1.1 가상화 확장 / IOMMU 확인
   - 1.2 `qemu-kvm`, `libvirt-daemon-system`, `libvirt-clients` 설치 (Ubuntu 24.04 noble)
   - 1.3 `apparmor-profiles-extra`(libvirt 프로파일 동봉)
   - 1.4 학습용 Ceph 클러스터 (단일 노드 ceph-mon/osd 또는 cephadm 미니 배포) 또는 외부 Ceph 접근
2. libvirt의 AppArmor 통합 활성화
   - 2.1 `/etc/libvirt/qemu.conf` — `security_driver = "apparmor"`
   - 2.2 `/etc/apparmor.d/usr.sbin.libvirtd` 정적 프로파일 분석
   - 2.3 `/etc/apparmor.d/usr.lib.libvirt.virt-aa-helper` 분석
   - 2.4 `/etc/apparmor.d/libvirt/TEMPLATE.qemu` 분석
3. 첫 게스트 띄우고 관찰
   - 3.1 cirros / Ubuntu cloud 이미지 게스트 정의
   - 3.2 `libvirt-<UUID>` 동적 프로파일 생성 확인
   - 3.3 `aa-status`로 부착 상태 확인
4. 정책이 실제로 동작하는지 검증
   - 4.1 게스트 종료 시 동적 프로파일 소멸 확인
   - 4.2 다른 게스트 디스크 경로에 대한 접근 거부 확인 (의도적 시도)
   - 4.3 `dmesg`에서 DENIED 이벤트 추적
5. **Rook-Ceph(RBD) 디스크 적용** — 본 시리즈 환경의 핵심 케이스
   - 5.1 `<disk type='network' protocol='rbd'>` 도메인 XML 패턴
   - 5.2 `virt-aa-helper`가 RBD 항목을 인식하는지 검증
   - 5.3 `/etc/ceph/*.conf`, `*.keyring`, `/run/ceph/...`, `/dev/rbd*` 정책 항목
   - 5.4 cephx 인증·secret 처리와 정책의 관계
6. 추가 디바이스 정책 보강 패턴
   - 6.1 vhost-user (DPDK)
   - 6.2 SR-IOV VF passthrough
   - 6.3 vTPM, swtpm 소켓
   - 6.4 hugepage / file-backed memory
7. TEMPLATE 커스터마이징
   - 7.1 사이트 공통 자원 추가 (예: Ceph 키링·conf 경로)
   - 7.2 packaging 충돌 회피 (`/etc/apparmor.d/local/` 패턴)
8. 트러블슈팅 케이스북
   - 8.1 "Failed to start domain — could not open disk" — `virt-aa-helper` 로그
   - 8.2 OVMF 펌웨어 경로 누락
   - 8.3 RBD secret 마운트/접근 실패와 AppArmor 거부의 구분
9. 검증 시나리오 5종 (마이그레이션·스냅샷·관리망 분리·RBD 볼륨 attach 포함)
10. 점검 질문 + 다음 편 예고

### 5편. OpenStack-Helm 2025.1(Epoxy) 보안 아키텍처 개요

**학습 목표**: nova-compute / libvirt가 DaemonSet으로 동작하고 ArgoCD가 Kustomize 오버레이로 배포하는 본 환경에서, 누가 무엇을 책임지는지(호스트 / 컨테이너 / Kubernetes / ArgoCD / AppArmor)를 그림으로 설명할 수 있다.

1. 환경 개관
   - 1.1 OpenStack 2025.1 (Epoxy) 컴포넌트 맵
   - 1.2 OpenStack-Helm chart 트리 + namespace 구조
   - 1.3 Kustomize + ArgoCD GitOps 파이프라인 (사전 렌더링 모델)
   - 1.4 Rook-Ceph 클러스터와 OpenStack의 결합점 (RBD)
2. 컴퓨트 노드 DaemonSet 구조
   - 2.1 `libvirt` DaemonSet (`docker.io/openstackhelm/libvirt:2025.1-ubuntu_noble`) — `libvirtd`, `virtlogd`, `virtlockd`
   - 2.2 `nova-compute` DaemonSet — initContainers 포함
   - 2.3 OVN/OVS 또는 OVS+Neutron 에이전트
   - 2.4 `nova-compute` ↔ `libvirt` IPC 경로 (host Unix 소켓)
3. 컨테이너화된 libvirt의 의미와 한계
   - 3.1 hostPID/hostIPC/hostNetwork 공유의 함의
   - 3.2 hostPath 마운트 패턴 (`/var/lib/libvirt`, `/var/lib/nova`, `/run/libvirt`, `/sys/fs/cgroup`, `/dev`, **`/etc/apparmor.d`**, **`/sys/kernel/security`**)
   - 3.3 컨테이너 baseOS가 noble이라는 사실의 의미와 무의미 (LSM은 호스트 커널)
4. AppArmor가 보는 풍경
   - 4.1 호스트 커널의 LSM은 namespace 구분 없이 작동
   - 4.2 컨테이너 안 `apparmor_parser`가 호스트 securityfs에 정책을 적재하는 메커니즘
   - 4.3 컨테이너 자체에 부착되는 AppArmor 프로파일과 그 안에서 도는 QEMU에 부착되는 동적 프로파일의 관계
   - 4.4 containerd/CRI-O의 기본 AppArmor 프로파일과의 상호작용
5. 보안 책임 분담 모델
   - 5.1 호스트 커널 + LSM
   - 5.2 컨테이너 런타임 + Pod securityContext
   - 5.3 OpenStack-Helm chart values + Kustomize patches
   - 5.4 ArgoCD Application·sync wave·prune 정책
   - 5.5 정책 코드 저장소 (GitOps)
6. 위협 시나리오로 본 책임 매트릭스
   - 6.1 게스트 → QEMU 탈취 → 호스트 접근 시도

### 6편. OpenStack-Helm + AppArmor 통합 설계 (Kustomize/ArgoCD)

**학습 목표**: 정책을 *어디에* *누가* *어떻게* 두고 적재할지 설계 결정을 내리고 그 근거를 설명할 수 있다. 본 환경(Kustomize + ArgoCD + OpenStack-Helm + DaemonSet)을 기준으로 한다.

1. 설계 의사결정 트리
   - 1.1 호스트에서 적재할까 컨테이너에서 적재할까? (정적 vs 동적의 분리)
   - 1.2 정적 프로파일은 노드 부팅 시 / 동적 프로파일은 libvirt 컨테이너 내부 / 양쪽 모두 호스트 securityfs로 수렴
   - 1.3 Multi-tenant 환경에서 정책 갱신 권한 분리
2. 호스트 측 사전 조건
   - 2.1 노드 부팅 시 정책 배포 옵션 (cloud-init / Ansible / 이미지 빌드 시 포함)
   - 2.2 `/etc/apparmor.d/` 디렉터리 구조 표준화
   - 2.3 `/etc/apparmor.d/local/` 사이트 정책 디렉터리 마련
   - 2.4 `kernel.apparmor_restrict_unprivileged_userns` 정책 결정
3. 컨테이너 측 결정 사항 — `docker.io/openstackhelm/libvirt:2025.1-ubuntu_noble`
   - 3.1 Pod securityContext: privileged vs `capabilities: [SYS_ADMIN, MAC_ADMIN, ...]` + AppArmor annotation
   - 3.2 필수 hostPath 마운트:
     - `/etc/apparmor.d` (rw) — 동적 프로파일이 호스트에 기록되도록

## 우선 순위 제안

전체 10편을 모두 만들기 전, 다음 우선순위로 작성하면 가장 빠르게 *현장 적용 가능한 최소 학습 셋*을 확보할 수 있다.

다만 학습 자료로서의 *흐름*을 중시한다면 1 → 2 → 3 → 4 → … 순서대로 작성하는 편이 독자 친화적이다.

## 결론

- **총 10편, 4개 단계**가 본 프로젝트의 목표("기초부터 최종 적용·검증까지")를 균형 있게 담는 단위다.
- 1편(작성 완료)이 토대를 잡았으므로, **다음 작성 권장은 2편(프로파일 작성·운영 실습)** 이다. 1편의 개념을 손으로 체득시키지 않으면 4편 이후 통합 단계에서 벽에 부딪힌다.
- 만약 분량을 줄여야 한다면, 위 "High Priority 5편" 셋이 최소 충분 집합이다.
