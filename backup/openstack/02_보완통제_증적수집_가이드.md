## 1. AppArmor sVirt 활성 증적

### 1.1 AppArmor 전체 상태 확인

```bash
sudo aa-status
```

**기대 출력 예시**:
```
apparmor module is loaded.
NN profiles are loaded.
NN profiles are in enforce mode.
   /usr/sbin/libvirtd
   /usr/lib/libvirt/virt-aa-helper
   libvirt-<UUID>
   ...
0 profiles are in complain mode.
NN processes have profiles defined.
NN processes are in enforce mode.
   /usr/sbin/libvirtd (PID)
   /usr/bin/qemu-system-x86_64 (PID) libvirt-<UUID>
   ...
```

**판정 기준**:
- ✅ `apparmor module is loaded.` — 커널 모듈 로드됨
- ✅ `/usr/sbin/libvirtd`가 enforce mode
- ✅ 각 실행 중 qemu-system-* 프로세스마다 `libvirt-<UUID>` 프로파일 부여됨

> **공식 근거**: Ubuntu Server Documentation
> > "By default, libvirt will spawn QEMU guests using AppArmor isolation for enhanced security."
> > 출처: https://ubuntu.com/server/docs/how-to/virtualisation/libvirt/

---

### 1.2 libvirt sVirt 보안 모델 확인

```bash
sudo virsh capabilities | grep -A3 secmodel
```

**기대 출력 예시**:
```xml
<secmodel>
  <model>apparmor</model>
  <doi>0</doi>
</secmodel>
<secmodel>
  <model>dac</model>
  <doi>0</doi>
  <baselabel type='kvm'>+64055:+108</baselabel>
  <baselabel type='qemu'>+64055:+108</baselabel>
</secmodel>
```

**판정 기준**:
- ✅ `<model>apparmor</model>` 존재 → libvirt가 AppArmor를 보안 드라이버로 사용 중

> **공식 근거**: libvirt.org
> > "If the sVirt security model is active, then the node capabilities XML will include its details."
> > 출처: https://libvirt.org/drvqemu.html

---

### 1.3 개별 VM의 AppArmor 프로파일 부여 확인

```bash
# 실행 중인 VM 목록
sudo virsh list

# 특정 VM의 보안 레이블 확인
sudo virsh dominfo <VM이름> | grep -i security
```

**기대 출력 예시**:
```
Security model:  apparmor
Security DOI:    0
Security label:  libvirt-XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX (enforcing)
```

**판정 기준**:
- ✅ Security model: `apparmor`
- ✅ Security label: `libvirt-<UUID> (enforcing)` — VM별 고유 프로파일 적용

> **공식 근거**: libvirt.org
> > "each qemu:///system QEMU virtual machine will have a profile created for it when the virtual machine is started ... contains rules allowing access to only the files it needs to run, such as its disks, pid file and log files."
> > 출처: https://libvirt.org/drvqemu.html

---

### 1.4 AppArmor 프로파일 파일 존재 확인

```bash
ls -la /etc/apparmor.d/abstractions/libvirt-qemu
ls -la /etc/apparmor.d/libvirt/ | head -20
```

**기대 출력 예시**:
```
-r--r--r-- 1 root root 1234 ... /etc/apparmor.d/abstractions/libvirt-qemu

/etc/apparmor.d/libvirt/:
-rw------- 1 root root 1234 ... libvirt-XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX
-rw------- 1 root root 5678 ... libvirt-XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX.files
TEMPLATE.qemu
...
```

**판정 기준**:
- ✅ `abstractions/libvirt-qemu` 존재 (모든 VM 공통 정적 부분)
- ✅ `libvirt/libvirt-<UUID>` 및 `.files` 파일 존재 (VM별 동적 부분)

> **공식 근거**: Ubuntu Server Documentation
> > "A static part that all guests share => /etc/apparmor.d/abstractions/libvirt-qemu. A dynamic part created at guest start time and modified on hotplug/unplug => /etc/apparmor.d/libvirt/libvirt-<UUID>.files"
> > 출처: https://ubuntu.com/server/docs/how-to/virtualisation/libvirt/

---

### 1.5 AppArmor 차단 이벤트 모니터링

```bash
# 최근 AppArmor 차단 이벤트 확인 (sVirt 작동 증명)
sudo journalctl -k --since "7 days ago" | grep -i apparmor | grep -i denied | head -20
```

**판정 기준**:
- 차단 이벤트가 존재한다면 → sVirt가 실제로 작동하여 위협을 차단했다는 강력한 증거
- 차단 이벤트가 없다면 → 정상 운영 상태 (그래도 §1.1~1.4로 동작 증명 가능)

## 2. 파일 무결성 모니터링 (FIM) 증적

> **공식 근거**: OpenStack Security Guide
> > "All such sensitive files should be given strict file level permissions, and monitored for changes through file integrity monitoring (FIM) tools such as iNotify or Samhain."
> > 출처: https://docs.openstack.org/security-guide/compute/hardening-deployments.html

### 2.1 AIDE 설치 및 운영 확인

```bash
# 설치 확인
dpkg -l | grep -i aide

# DB 초기화 확인
ls -la /var/lib/aide/aide.db.gz

# 정기 무결성 검사 cron 확인
sudo cat /etc/cron.daily/aide 2>/dev/null || sudo systemctl list-timers | grep -i aide
```

### 2.2 AIDE 모니터링 대상 확인

```bash
sudo grep -E "^/" /etc/aide/aide.conf | head -30
```

**필수 모니터링 대상 (예시)**:
- `/etc` (설정 파일)
- `/usr/sbin`, `/usr/bin` (실행 파일)
- `/boot` (커널 및 부트로더)
- `/usr/lib/modules` (커널 모듈)
- `/etc/nova/`, `/etc/libvirt/` (OpenStack/libvirt 설정)

> **공식 근거**: OpenStack Security Guide — `nova.conf` 등 민감 설정파일은 FIM 대상으로 명시
> > "One of the main security concerns with any OpenStack deployment is the security and controls around sensitive files, such as the nova.conf file."
> > 출처: 동일

### 2.3 AIDE 검사 결과 보존
```bash
# 최근 검사 보고서 확인
ls -la /var/log/aide/
sudo cat /var/log/aide/aide.log | tail -50
```

## 3. 호스트 OS 하드닝 증적

### 3.1 설치 패키지 인벤토리 (최소 설치 입증)

```bash
# 설치된 패키지 수 (최소화 입증)
dpkg -l | wc -l

# 불필요 서비스 비활성화 확인
systemctl list-unit-files --state=enabled --type=service
```

**판정 기준**:
- 설치 패키지: SCPv2 표준 베이스 이미지 기준값 ± 10% 이내
- 활성화된 서비스: nova-*, libvirt*, ovs-*, sshd, systemd-* 등 필수 서비스만

### 3.2 SSH 접근 제어 확인

```bash
# SSH 설정 확인 (root 직접 로그인 금지, MFA 등)
sudo grep -E "^(PermitRootLogin|PasswordAuthentication|PubkeyAuthentication|AllowUsers)" /etc/ssh/sshd_config

# 최근 접근 로그
sudo journalctl -u ssh --since "7 days ago" | grep -i "accepted" | tail -20
```

**판정 기준**:
- ✅ `PermitRootLogin no`
- ✅ `PasswordAuthentication no` (키 기반만)
- ✅ Bastion/점프호스트를 통해서만 접근 가능

### 3.3 커널 보안 모듈 확인

```bash
# 로드된 보안 관련 모듈
lsmod | grep -iE "apparmor|kvm|vhost"

# 커널 보안 파라미터 (sysctl)
sudo sysctl -a 2>/dev/null | grep -E "kernel.kptr_restrict|kernel.dmesg_restrict|kernel.yama.ptrace_scope|kernel.randomize_va_space"
```

**기대 출력**:
```
kernel.kptr_restrict = 2
kernel.dmesg_restrict = 1
kernel.yama.ptrace_scope = 1
kernel.randomize_va_space = 2
```

### 3.4 패치 관리 증적

```bash
# 마지막 패치 적용 일자
ls -la /var/log/apt/history.log* | head -5
sudo grep -i "upgrade" /var/log/apt/history.log | tail -20

# unattended-upgrades 설정
sudo cat /etc/apt/apt.conf.d/50unattended-upgrades | grep -v '^//' | grep -v '^$'

# 미적용 보안 업데이트 확인
sudo apt list --upgradable 2>/dev/null | grep -i security
```

> **CSAP 근거**: 9.1.4 본문 — "하이퍼바이저에 대한 소프트웨어 업데이트 및 보안패치를 최신으로 유지하여야 한다"

## 4. 네트워크 분리 증적

### 4.1 인터페이스 구성 확인

```bash
# 인터페이스 목록 및 IP
ip -br addr show

# 라우팅 테이블
ip route show
```

**판정 기준**:
- 관리망(eth0/bond0): 운영자 접근 전용, VM에서 도달 불가
- 데이터망(eth1/bond1): VM 트래픽 전용
- 스토리지망(eth2/bond2): Ceph/iSCSI 전용
- 각 망은 VLAN 또는 별도 NIC로 물리 분리

### 4.2 방화벽 규칙

```bash
# iptables 또는 nftables 규칙
sudo iptables -L INPUT -n -v --line-numbers | head -30
sudo nft list ruleset 2>/dev/null | head -50

# 외부 접근 가능 포트
sudo ss -tlnp | grep LISTEN
```

**판정 기준**:
- 호스트 외부 노출 포트: SSH(22), libvirtd 마이그레이션 포트만
- 마이그레이션 포트는 컴퓨트 노드 간 사설망에서만 접근 가능

## 5. OpenStack 컴퓨트 보안 설정 증적

### 5.1 nova.conf 보안 설정

```bash
sudo grep -E "^\[|^(force_config_drive|use_rootwrap|disable_libvirt_livesnapshot|cpu_mode|virt_type)" /etc/nova/nova.conf
```

**기대 설정 예시**:
```ini
[libvirt]
virt_type = kvm
cpu_mode = host-model
images_type = qcow2

[DEFAULT]
use_rootwrap = True
```

### 5.2 nova.conf 권한

```bash
ls -la /etc/nova/nova.conf
```

**기대 출력**:
```
-rw-r----- 1 root nova ...
```

> **공식 근거**: OpenStack Security Guide
> > "All such sensitive files should be given strict file level permissions"
> > 출처: docs.openstack.org/security-guide/compute/hardening-deployments.html

### 5.3 libvirt 보안 설정

```bash
sudo grep -E "^(listen_tls|listen_tcp|auth_tcp|key_file|cert_file|security_driver|security_default_confined)" /etc/libvirt/libvirtd.conf /etc/libvirt/qemu.conf 2>/dev/null
```

**기대 설정**:
- `security_driver = "apparmor"` (또는 자동 감지로 미설정)
- `security_default_confined = 1`
- TLS 사용 시: `listen_tls = 1`, 인증서 경로 지정

## 6. 감사 로깅 (auditd) 증적

```bash
# auditd 가동 확인
sudo systemctl status auditd

# 가상화 관련 audit 규칙
sudo auditctl -l | grep -iE "libvirt|qemu|nova"

# 최근 감사 이벤트
sudo ausearch -ts today -ui root 2>/dev/null | head -20
```

**권고 audit 규칙 예시**:
```
-w /etc/libvirt/ -p wa -k libvirt_config
-w /etc/nova/ -p wa -k nova_config
-w /var/lib/nova/instances/ -p wa -k nova_instances
```
