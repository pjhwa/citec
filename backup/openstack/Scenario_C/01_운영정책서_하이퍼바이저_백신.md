# 01. SCPv2 하이퍼바이저 악성코드 보호 운영정책서 v2.0

> **문서 목적**: CSAP 9.1.4-③ 대응을 위한 하이퍼바이저 악성코드 보호 공식 운영 정책
> **대상 환경**: SCPv2 Sovereign(PG) 상암/춘천 컴퓨트 노드 전체 (Ubuntu + OpenStack KVM)
> **관리 부서**: SCPv2 운영부서 (주관) + 정보보호팀 (감독) + CI-TEC (기술 지원)
> **시행일**: 2026-05-01 (본심사 통과 시 즉시 효력 발생)

---

## 1. 정책 선언

### 1-1. 기본 방침

Samsung Cloud Platform v2(이하 "SCPv2")는 하이퍼바이저 노드의 악성코드 보호를 위하여 다음의 통제 체계를 운영한다.

1. **하이퍼바이저 노드에는 전통적 백신 소프트웨어를 설치하지 않는다.**
2. **Linux 커널의 강제접근통제(MAC) 기능인 sVirt(AppArmor)를 주된 보호 수단으로 사용한다.**
3. **MAC을 보완하는 4종 보완통제 패키지 — FIM + 호스트 OS 최소화 + 관리망 분리 + 침해사고 대응 — 를 병행 운영한다.**
4. **본 정책은 3개월(2026-04-20~2026-07-20) 내 전 운영 환경 적용을 완료한다.**

### 1-2. 본 방침의 사유

하이퍼바이저 환경의 특수성 때문에 전통적 백신은 통제 목적을 달성하지 못하고 오히려 운영 안정성을 훼손한다 (`§2`). 반면 MAC은 본 통제 항목이 추구하는 목적 — 하이퍼바이저·VM의 무결성 보호 — 을 **구조적으로 더 효과적으로** 달성한다 (`§3`).

---

## 2. 백신 미설치 사유 (근거)

### 2-1. 기술적 부적합성 (4개 항목)

| # | 항목 | 구체 내용 |
| --- | --- | --- |
| 1 | qcow2/raw 디스크 특수성 | `/var/lib/nova/instances/` sparse 파일은 초 단위 변동, AV 스캔 불가능 |
| 2 | qemu-kvm 프로세스 종료 리스크 | 오탐 시 호스트의 모든 VM 즉시 다운 (대규모 장애) |
| 3 | 라이브 마이그레이션 간섭 | AV 실시간 훅이 dirty-page 전송 timeout 유발 |
| 4 | Linux 백신 제품의 KVM 미대응 | 국내 백신 제품 중 하이퍼바이저 전용 프로파일 존재 제품 전무 |

### 2-2. 공식 문서 근거

| 문서 | 핵심 내용 |
| --- | --- |
| OpenStack Security Guide | 하이퍼바이저 강화 3대 권고 = ①코드베이스 최소화 ②컴파일러 하드닝 ③MAC |
| Ubuntu Server 공식 문서 | libvirt는 기본적으로 AppArmor 격리로 QEMU guest spawn |
| NIST SP 800-125 | 가상화 호스트 보안 원칙 = attack surface 최소화 + 격리 + 무결성 |
| libvirt 공식 문서 | sVirt가 VM별 고유 AppArmor 프로파일을 동적 생성하여 커널 격리 |

본 4종 문서 어디에도 "하이퍼바이저에 전통적 백신을 설치하라"는 권고는 없다. 반대로 **MAC이 권고 항목의 필수 요소로 명시**되어 있다.

### 2-3. 업계 관행

- **AWS Nitro**: 호스트 OS에 shell·filesystem·user-space utility 자체가 없어 AV 설치 불가능 구조
- **Microsoft Azure**: Hyper-V 호스트에 VBS/HVCI 기반 보호 운용, 전통 AV 미설치
- **Google Cloud**: KVM/Borg 기반 gVisor + 자체 무결성 모니터링, 전통 AV 미설치
- **국내 CSP (NHN/KT/Naver)**: 모두 CSAP IaaS 인증 보유 (NHN Cloud는 OpenStack 기반 공식 홍보)

### 2-4. 삼성SDS 자체 선례

SCPv1 VMware 환경에서 "벤더 미지원"을 사유로 백신 미설치 + 보완통제로 CSAP 통과한 선례 존재. v2는 이 선례를 MAC 기반으로 고도화.

## 3. 보완통제 설계 (4종 패키지)

### 3-1. Primary Control: MAC (sVirt + AppArmor)

| 항목 | 내용 |
| --- | --- |
| 구현 기술 | libvirt sVirt + Ubuntu AppArmor |
| 작동 계층 | Linux 커널 공간 (root도 우회 불가) |
| 격리 단위 | VM별 QEMU 프로세스마다 동적 생성된 고유 AppArmor 프로파일 |
| 운영 모드 | **Enforce** (전 운영 환경 적용 목표, 2026-07-20까지) |
| 프로파일 템플릿 | `/etc/apparmor.d/libvirt/TEMPLATE.qemu` (OpenStack Helm 특화 커스터마이징) |
| 모니터링 | `aa-status` 주간 자동 수집, DENIED 로그 실시간 SIEM 전송 |

### 3-2. Secondary Control: FIM (File Integrity Monitoring)

| 항목 | 내용 |
| --- | --- |
| 구현 도구 | AIDE (Advanced Intrusion Detection Environment) |
| 감시 대상 | `/etc/`, `/bin/`, `/sbin/`, `/usr/bin/`, `/usr/sbin/`, `/boot/`, `/lib/modules/` |
| 제외 경로 | `/var/lib/nova/instances/` (VM 디스크 동적 변동) |
| 실행 주기 | 일 1회 (03:00 KST, 노드별 10분 시차) |
| 보고 | 무결성 이상 탐지 시 즉시 Alert → SCPv2 운영 NOC |
| 베이스라인 갱신 | 커널/패치 적용 시 승인 후 갱신 |

### 3-3. Tertiary Control: 호스트 OS 최소화

| 항목 | 내용 |
| --- | --- |
| 방침 | 컴퓨트 노드 OS에 **운영 필수 패키지만** 설치 |
| 금지 항목 | `gcc`, `make`, `python-dev` 등 빌드 도구, `telnet`, `ftp` 등 취약 프로토콜 |
| 관리 방법 | 골드 이미지 기반 배포, `dpkg --get-selections` 일 1회 diff 수집 |
| 패치 관리 | Canonical Ubuntu Pro + Landscape, Kernel Live Patching 사용 |

### 3-4. Quaternary Control: 관리망 완전 분리

| 항목 | 내용 |
| --- | --- |
| 네트워크 구성 | 컴퓨트 노드 관리 인터페이스 = 별도 VLAN (외부 인터넷 접근 불가) |
| 접근 통제 | Bastion Host 경유 + MFA + 세션 녹화 |
| 원격 접근 | Zero Trust 정책 (하루 24시간 활성 세션 최대 10분) |
| 모니터링 | 관리 인터페이스 로그인 성공·실패 전수 기록, 이상 징후 Alert |

## 4. 운영 절차

### 4-1. 평상시 운영

| 주기 | 작업 | 담당 | 증적 |
| --- | --- | --- | --- |
| 실시간 | AppArmor DENIED 로그 SIEM 전송 | SCPv2 운영 | SIEM 로그 |
| 일 1회 | AIDE FIM 스캔 | 자동화 | AIDE 보고서 |
| 일 1회 | 컴퓨트 노드 패키지 인벤토리 diff | 자동화 | dpkg 변경 로그 |
| 주 1회 | `aa-status` 전 노드 수집 → 대시보드 갱신 | CI-TEC | aa-status JSON |
| 월 1회 | MAC 프로파일 버전 관리 이력 검토 | 정보보호팀 | Git log |
| 분기 1회 | MAC 정책 유효성 검토 (신규 워크로드 반영) | 정보보호팀 + CI-TEC | 검토 보고서 |
| 연 1회 | 침투 시뮬레이션 (sVirt 회피 시도) | 외부 전문업체 | 모의해킹 보고서 |

### 4-2. MAC 프로파일 변경 절차

1. 변경 요청 제출 → 정보보호팀 검토 (SLA: 영업일 5일)
2. Test 환경에서 complain 모드로 배포 → 2주 모니터링
3. Dev 환경 배포 → 1주 모니터링
4. 운영 환경 canary 배포 → 1주 모니터링 (1개 노드)
5. 운영 환경 전사 배포
6. Git commit + 변경 증적 보관

### 4-3. 예외 운영 (부득이 MAC 비활성화 시)

1. 정보보호팀 사전 승인 필수 (CISO 결재)
2. 비활성화 기간 최대 24시간 (연장 시 재승인)
3. 해당 노드 VM 마이그레이션 후 격리 운영
4. 조치 완료 후 즉시 MAC 재활성화 및 검증

## 5. 침해사고 대응 (9.1.4-③ 두 번째 ∎ 충족)

본 정책의 악성코드 탐지·대응 절차는 별도 문서 `03_악성코드_대응절차서.md` 를 따른다. 요약:

1. **탐지**: FIM 이상 + AppArmor DENIED 급증 + 네트워크 이상 트래픽 중 2개 이상 일치
2. **격리**: 해당 노드 위 VM 라이브 마이그레이션 (Dry-run 후 실행)
3. **분리**: 노드를 운영 네트워크에서 분리 (관리 VLAN만 유지)
4. **포렌식**: 메모리 덤프 + 디스크 이미지 확보, 외부 포렌식 전문가 투입
5. **복구**: 골드 이미지 재배포 → FIM 베이스라인 재생성 → MAC 프로파일 재배포
