## 1. 정책 개요

### 1-1. 탐지 체계

SCPv2는 하이퍼바이저 악성코드를 다음 3개 독립 채널로 탐지한다:

| 채널 | 탐지 대상 | 기술 | 운영 주체 |
| --- | --- | --- | --- |
| **C1: MAC DENIED** | 프로세스 권한 이탈 시도 | AppArmor → SIEM | SCPv2 운영 NOC |
| **C2: FIM 변조** | 호스트 중요 파일 변경 | AIDE → 일일 보고서 | SCPv2 운영 NOC |
| **C3: 네트워크 이상** | 비정상 outbound 트래픽 | NDR + IDS | 정보보호팀 SOC |

**탐지 기준**: 위 3개 중 **2개 이상이 동시에 트리거**되면 침해사고 의심 상태로 격상.

### 1-2. 대응 체계 (3단계)

```
Stage 1 (Detect & Assess)  →  Stage 2 (Contain)  →  Stage 3 (Eradicate & Recover)
    최대 15분 내              최대 1시간 내            최대 24시간 내
```

## 2. Stage 1: 탐지 및 초기 평가 (D+15분)

### 2-1. 탐지 시점 자동 대응

| 트리거 | 자동 동작 |
| --- | --- |
| SIEM Alert (C1+C2 동시) | PagerDuty → On-call 호출 |
| FIM 일일 보고서 이상 | 이메일 + Slack 운영채널 통보 |
| NDR 임계치 초과 | SOC 티켓 자동 생성 |

### 2-2. 초기 평가 체크리스트 (NOC)

- [ ] 영향 노드 식별: 호스트명, 물리 위치, 해당 노드의 VM 목록
- [ ] 탐지 상세 확인:
  - AppArmor DENIED 로그 원문 추출 (`grep denied /var/log/syslog | tail -50`)
  - AIDE 변경 파일 목록 확인 (`/var/log/aide/aide-check-*.log`)
  - NDR 이상 트래픽 소스·목적지 포트 확인
- [ ] **정상 운영 행위 여부 판단** (False Positive 1차 스크리닝):
  - 패치 배포 작업 중인가?
  - 신규 VM 스폰 중인가?
  - Ceph RBD 장애 복구 중인가?

### 2-3. 판단 기준

| 판단 | 조치 |
| --- | --- |
| 정상 작업 (F/P) | 로그에 조치 사유 기록 후 종료 |
| 의심 사례 | Stage 2로 격상 + 정보보호팀 CERT 호출 |
| 명백한 침해 | Stage 2로 즉시 이행 + CISO 통보 |

### 2-4. 에스컬레이션 연락처

| 역할 | 담당 | 연락 수단 |
| --- | --- | --- |
| NOC 1차 대응 | SCPv2 운영 On-call | 24/7 PagerDuty |
| CERT 2차 대응 | 정보보호팀 침해사고대응팀 | 24/7 PagerDuty + 전화 |
| 기술 자문 | CI-TEC 인프라파트 | 업무시간 + 긴급 연락망 |
| 최종 의사결정 | CISO | 긴급 Hotline |

## 3. Stage 2: 격리 및 차단 (D+1시간)

### 3-1. 워크로드 격리 (VM 라이브 마이그레이션)

감염 의심 노드의 **고객 VM을 정상 노드로 라이브 마이그레이션**하여 서비스 지속성 확보:

```bash
# 1. 대상 노드의 VM 목록
nova list --host compute-node-X --all-tenants

# 2. Dry-run으로 마이그레이션 가능성 확인
openstack server migrate <VM_UUID> --live --dry-run

# 3. 실제 라이브 마이그레이션 (VM별 순차 실행)
for vm in $(nova list --host compute-node-X --all-tenants -c ID -f value); do
  echo "Migrating $vm..."
  openstack server migrate $vm --live
  sleep 30
done

# 4. 마이그레이션 완료 확인
nova list --host compute-node-X --all-tenants
# → 결과가 비어있어야 함
```

**주의**: 감염 의심 상태의 VM 디스크 이미지가 다른 노드로 이동하면 감염 확산 가능. **마이그레이션은 호스트 감염이 "호스트 자체"로 한정될 때만 수행**. VM 자체가 감염원일 가능성이 있으면 마이그레이션 대신 snapshot 후 정지.

### 3-2. 노드 네트워크 분리

| 통제 | 명령 |
| --- | --- |
| 서비스망 인터페이스 차단 | `sudo ip link set <service-NIC> down` |
| 관리망만 유지 (포렌식용) | 관리 VLAN만 활성 유지 |
| iptables 전면 차단 | `sudo iptables -P OUTPUT DROP` + Bastion IP만 ACCEPT |
| Kubernetes cordon (Helm 환경) | `kubectl cordon <node>` |
| DaemonSet 제거 | `kubectl drain <node> --ignore-daemonsets --delete-emptydir-data` |

### 3-3. 휘발성 증거 확보 (포렌식 전 필수)

```bash
# 1. 메모리 덤프 (가능 시)
sudo apt install lime-forensics-dkms  # 설치 필요
sudo insmod /lib/modules/$(uname -r)/updates/dkms/lime.ko \
  "path=/mnt/forensic/memdump.lime format=lime"

# 2. 실행 중 프로세스 전체 스냅샷
ps -eZ auxf > /mnt/forensic/ps-$(date +%s).txt
ps -eo pid,ppid,cmd,args,nlwp,thcount,stime,time > /mnt/forensic/ps-detail.txt

# 3. 네트워크 연결 상태
ss -tnpa > /mnt/forensic/ss-tcp.txt
ss -unpa > /mnt/forensic/ss-udp.txt

# 4. 로그인 세션
who > /mnt/forensic/who.txt
last > /mnt/forensic/last.txt

# 5. AppArmor 전체 프로파일 상태
sudo aa-status --json > /mnt/forensic/aa-status.json

# 6. journalctl 전체 부팅 이후 로그
sudo journalctl -b > /mnt/forensic/journal-this-boot.txt

# 7. /tmp, /var/tmp 전체 아카이브
tar czf /mnt/forensic/tmp-dirs.tar.gz /tmp /var/tmp 2>/dev/null
```

**주의**: `/mnt/forensic` 은 별도 마운트된 수집용 디스크 사용. 감염 노드 로컬 디스크에 증거 저장 금지.

### 3-4. 고객 통지 (해당 노드 VM이 있었을 경우)

- 통지 책임: SCPv2 운영부서 → 영업팀 → 고객
- 통지 내용: 마이그레이션 완료, 서비스 영향 없음, 재발 방지 대책
- SLA: **격리 완료 후 1시간 이내 1차 통지**

### 3-5. Stage 2 완료 기준

- [ ] 감염 의심 노드에 VM 0개
- [ ] 네트워크 격리 확인 (외부 통신 불가)
- [ ] 휘발성 증거 확보 완료
- [ ] CERT 투입 완료
- [ ] 고객 통지 완료 (해당 시)

## 4. Stage 3: 근절 및 복구 (D+24시간)

### 4-1. 포렌식 분석

| 항목 | 담당 | 산출물 |
| --- | --- | --- |
| 메모리 분석 (Volatility) | 외부 포렌식 전문업체 | 메모리 분석 보고서 |
| 디스크 이미징 및 분석 | 외부 포렌식 전문업체 | 타임라인 분석 |
| AppArmor 로그 분석 | 정보보호팀 CERT | 공격 경로 추정 보고서 |
| 네트워크 로그 분석 | 정보보호팀 SOC | C2 통신 증거 |
| 루트캐즈 확인 | 정보보호팀 + 외부 | RCA 보고서 |

### 4-2. 취약점 조치

포렌식으로 확인된 취약점에 대해:
1. CVE 번호 확인 → 패치 가용성 확인
2. 테스트 환경 검증 후 **전 컴퓨트 노드 일괄 패치**
3. 패치 후 AppArmor 프로파일 영향도 재확인
4. FIM 베이스라인 재생성

### 4-3. 노드 복구 절차

```bash
# Option A: 골드 이미지 재배포 (권장)
# 1. Ubuntu 골드 이미지로 OS 재설치
# 2. OpenStack Helm DaemonSet 재합류
# 3. AppArmor 프로파일 Git에서 pull
# 4. AIDE 베이스라인 재생성
# 5. FIM·AppArmor 작동 확인 후 uncordon

# Option B: 부분 복구 (조기 서비스 복귀 필요 시)
# 1. 감염 파일 제거
# 2. AppArmor 프로파일 재적용 + 테스트
# 3. AIDE 베이스라인 재생성
# 4. 72시간 집중 모니터링 후 운영 복귀
# ※ Option B는 명확한 rootcause가 확인된 경우에만 선택
```

### 4-4. 복구 검증 체크리스트

- [ ] `aa-status`: enforce 모드, 예상 프로파일 수 일치
- [ ] `virsh capabilities`: apparmor 활성
- [ ] AIDE: 새 baseline 생성, 스캔 정상 완료
- [ ] OpenStack: `nova service-list` UP 확인
- [ ] Ceph: RBD 연결 정상
- [ ] 테스트 VM 스폰 → 부팅 → 삭제 성공
- [ ] 72시간 관찰 기간 내 이상 징후 없음

### 4-5. 서비스 복귀

- SCPv2 운영팀 VIP 승인 필수
- `kubectl uncordon <node>` → 스케줄링 재개
- 고객 복구 완료 통지

### 6-1. 사이버 훈련

| 훈련 | 주기 | 참여 |

| 표상 훈련 (TTX) | 분기 1회 | NOC + CERT + CI-TEC |

### 6-2. 시나리오 예시

1. **시나리오 A**: AppArmor DENIED 급증 + AIDE 탐지 → 대응 시간 측정

3. **시나리오 C**: C2 트래픽 탐지 → 전 노드 격리·검색

## 7. Runbook 색인 (Quick Reference)

### 7-1. 즉시 조치 카드 (NOC 대시보드 상단 배치)

```
┌─────────────────────────────────────────────────────────┐
│  하이퍼바이저 악성코드 대응 Quick Action                  │
├─────────────────────────────────────────────────────────┤
│  [1] AppArmor DENIED Alert                              │
│      → grep DENIED /var/log/syslog | tail -20           │
│      → 오탐 시: 프로파일 업데이트 티켓 생성              │
│      → 진성 시: CERT 호출 (on-call)                      │
│                                                          │
│  [2] AIDE 이상 탐지                                      │
│      → less /var/log/aide/aide-check-YYYYMMDD.log       │
│      → 정상 패치 시: 로그 기록 후 baseline 갱신          │
│      → 비정상 시: CERT 호출                              │
│                                                          │
│  [3] 네트워크 이상 (C2 의심)                             │
│      → 즉시 서비스 NIC 다운 + VM 마이그레이션            │
│      → CERT + CISO 동시 호출                             │
└─────────────────────────────────────────────────────────┘
```

### 7-2. 주요 명령 Cheatsheet

| 작업 | 명령 |
| --- | --- |
| VM 목록 확인 | `nova list --host <node> --all-tenants` |
| 라이브 마이그레이션 | `openstack server migrate <uuid> --live` |
| 노드 cordon | `kubectl cordon <node>` |
| 네트워크 차단 | `sudo ip link set <NIC> down` |
| AppArmor 상태 | `sudo aa-status` |
| FIM 검사 | `sudo aide --check` |
| 프로세스 레이블 | `ps -eZ | grep qemu-system` |
| 메모리 덤프 | `lime.ko` module load |
