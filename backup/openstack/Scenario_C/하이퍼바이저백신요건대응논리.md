## 1. 대응 논리의 기본 구조

### 1-1. 통제 목적의 재해석

CSAP 9.1.4-③의 본질적 통제 목적은 **"하이퍼바이저가 악성코드로 인해 무결성·기밀성·가용성을 상실하지 않도록 보호"** 하는 것이다. 백신 설치는 이 목적을 달성하는 **수단 중 하나**일 뿐 목적 그 자체가 아니다.

### 1-2. 4가지 대응 주장 (Line of Defense)

| # | 주장 | 핵심 내용 |
| --- | --- | --- |
| 1 | **기술적 부적합성** | KVM 하이퍼바이저에 전통적 AV 설치는 가상화 워크로드에 구조적 위험을 초래 |
| 2 | **보완통제의 동등성** | MAC(sVirt/AppArmor)은 AV가 목표하는 보호 수준을 **구조적으로 더 강력하게** 달성 |
| 3 | **국제 표준·공식 문서 부합** | OpenStack Security Guide, NIST SP 800-125, libvirt 공식 문서가 MAC을 우선 권고 |
| 4 | **업계 관행** | 글로벌 하이퍼스케일러 전체가 전통적 AV 미사용 (AWS/Azure/GCP 공식 문서로 입증) |

이 4가지를 **동시에** 제시해야 하며, 단일 논리로는 심사관 수용이 어려움.

## 2. 주장 1: 기술적 부적합성 (왜 설치하면 안 되는가)

### 2-1. qcow2/raw 디스크 이미지의 특수성

- `/var/lib/nova/instances/` 디렉터리는 VM 메모리·디스크 변경에 따라 **초 단위로 내용이 변동**하는 sparse 파일의 집합
- 전통 AV의 시그니처 스캔은 VM 내부 OS의 파일 시스템 구조를 이해하지 못함 → **의미 없는 부하만 유발**
- OpenStack 공식 가이드도 "백업 시 본 디렉터리 제외" 권고 (후술 §4)

### 2-2. qemu-kvm 프로세스 격리 실패 시 파급효과

- AV가 qemu-kvm 프로세스를 오탐/격리/종료하는 순간 → **해당 노드의 모든 VM 즉시 다운**
- 일반 서버 대상 AV 정책을 그대로 적용하면 **대규모 서비스 장애** 가능성 높음
- 격리 기능 OFF · 예외 경로 관리만으로는 완전한 안전 보장 불가 (AV 벤더 업데이트로 정책이 의도치 않게 변경되는 사례 존재)

### 2-3. 라이브 마이그레이션 호환성 이슈

- AV 실시간 파일시스템 훅(hook)은 page-cache·inode lookup에 latency 추가 → 마이그레이션 dirty-page 전송 중 타임아웃 발생 가능
- KVM 커뮤니티에는 "AV로 인한 live migration 실패" 사례가 지속 보고됨

### 2-4. Linux 백신 제품의 성숙도 한계

- 대다수 국내 Linux 백신 제품은 Windows Server 대응에 최적화, **KVM 호스트 전용 프로파일 부재**
- "예외 처리 + 예약 스캔만" 운영 시, 실질적으로 AV가 하는 일이 없어짐 → **"설치 자체가 형식적 대응"이 되어 통제 목적 달성 실패**

> **결론**: AV 설치는 통제 목적 달성보다 운영 리스크를 더 키운다. 형식 요건을 맞추기 위해 실질적 안정성을 훼손하는 것은 정보보호의 본질에 반함.

## 3. 주장 2: 보완통제(MAC)의 동등성·우월성

### 3-1. MAC의 작동 원리 — 프로세스 단위 격리

`sVirt`는 libvirt가 각 VM의 QEMU 프로세스마다 **고유한 AppArmor 프로파일**을 동적 생성하여 커널 레벨에서 접근을 강제 제어함.

| 차원 | 전통 AV | **MAC (sVirt/AppArmor)** |
| --- | --- | --- |
| 탐지 방식 | 시그니처 기반 (사후) | **정책 기반 (사전)** |
| 작동 계층 | 사용자 공간 | **커널 공간** |
| Zero-day 대응 | 패턴 업데이트 후 탐지 | **정책 위반은 무조건 차단** |
| VM-VM 격리 | 불가능 (호스트 파일만 스캔) | **VM 프로세스별 고유 프로파일** |
| 성능 오버헤드 | 실시간 스캔 시 높음 | **1-3% 미만** |
| 오탐 시 영향 | qemu 프로세스 종료 → VM 다운 | **로그 기록 (complain) / 차단 (enforce)** |

### 3-2. MAC이 막는 공격 시나리오 (AV로는 대응 불가한 영역)

1. **VM Escape → 다른 VM 디스크 접근**: svirt_image_t 레이블로 커널이 원천 차단
2. **악성 VM이 호스트 파일시스템 변조**: AppArmor 프로파일이 허용 경로 외 모든 접근 거부
3. **하이퍼바이저 프로세스의 권한 상승**: cgroups + capabilities 제한으로 원천 봉쇄
4. **공유 라이브러리 hijacking**: 프로파일이 mmap 가능 파일을 명시 → 차단

### 3-3. FIM · 최소 설치 · 네트워크 분리 3종 보완

MAC 단독이 아니라 다음 4종 통제의 패키지로 운영:

| 통제 | 역할 | 증적 |
| --- | --- | --- |
| **MAC (sVirt/AppArmor)** | 프로세스 단위 강제 격리 | `aa-status`, `virsh capabilities` |
| **FIM (AIDE/Samhain)** | 호스트 주요 파일 변경 탐지 | 일일 보고서, 탐지 로그 |
| **호스트 OS 최소 설치** | Attack surface 축소 | 패키지 인벤토리, dpkg log |
| **관리망 완전 분리** | 외부 유입 경로 차단 | VLAN 구성도, ACL 룰 |

## 4. 주장 3: 공식 문서 근거 5종

### 4-1. OpenStack Security Guide — MAC을 3대 권고사항에 포함 (백신은 미언급)

> "We recommend three specific steps: Minimizing the code base. Using compiler hardening. **Using mandatory access controls such as sVirt, SELinux, or AppArmor.**"

출처: <https://docs.openstack.org/security-guide/compute/hardening-the-virtualization-layers.html>

### 4-2. Ubuntu Server 공식 문서 — libvirt 기본 AppArmor 격리

> "By default, libvirt will spawn QEMU guests using AppArmor isolation for enhanced security."

출처: <https://ubuntu.com/server/docs/how-to/virtualisation/libvirt/>

### 4-3. libvirt 공식 문서 — sVirt의 VM별 프로파일 자동 생성

> "The AppArmor sVirt protection for QEMU virtual machines builds on this basic level of protection, to also allow individual guests to be protected from each other."

출처: <https://libvirt.org/drvqemu.html>

### 4-4. NIST SP 800-125 "Guide to Security for Full Virtualization Technologies"

하이퍼바이저 보안 3대 원칙으로 ① **attack surface 최소화** ② **격리(isolation)** ③ **무결성 검증**을 제시. 추가 에이전트(AV) 설치는 attack surface 확대 방향 → 원칙 1과 상충.

출처: <https://csrc.nist.gov/pubs/sp/800/125/final>

### 4-5. OpenStack Security Guide — FIM 권고 (AV가 아님)

> "All such sensitive files should be given strict file level permissions, and monitored for changes through file integrity monitoring (FIM) tools such as iNotify or Samhain."

출처: <https://docs.openstack.org/security-guide/compute/hardening-deployments.html>

> **핵심**: 관련 국제 표준·공식 문서는 **전통 AV를 권고 항목에 포함하지 않음**. 본 대응 방향이 베스트 프랙티스와 정합.

## 5. 주장 4: 글로벌·국내 CSP 사례

### 5-1. 글로벌 CSP (공개 자료로 입증 가능)

| CSP | 하이퍼바이저 | 호스트 AV 운용 여부 | 공식 근거 |
| --- | --- | --- | --- |
| **AWS** | Nitro | **미설치** (호스트 OS 자체에 shell/FS 미존재) | AWS Nitro Security Whitepaper |
| **Microsoft Azure** | Hyper-V | **미설치** (VBS + HVCI 기반) | Azure Hypervisor Security docs |
| **Google Cloud** | KVM/Borg | **미설치** (gVisor + 자체 무결성 모니터링) | GCP Security Whitepaper |

→ **글로벌 하이퍼스케일러 전체가 전통적 AV 대신 "MAC + 무결성 + 최소화" 접근을 채택**. 본 통제 항목의 본질적 해결책이 MAC임을 전 세계 표준이 입증.

### 5-2. 국내 CSP (공개 자료 범위 내)

| CSP | 인증 취득 | 기반 기술 | AV 운용 여부 |
| --- | --- | --- | --- |
| **NHN Cloud** | IaaS/DaaS/SaaS 모두 CSAP 보유 | **OpenStack 기반** (공식 홍보) | **미공개** (추정: MAC 기반) |
| **KT Cloud** | IaaS CSAP 보유 | 자체 가상화 + OpenStack | **미공개** |
| **네이버클라우드** | IaaS CSAP 보유 | 자체 가상화 | **미공개** |
| **가비아** | IaaS CSAP 보유 | 자체 가상화 | **미공개** |

⚠️ **솔직한 제약**:
- 국내 CSP의 **하이퍼바이저 AV 설치 여부는 공개 자료에 나타나지 않음** (보안 아키텍처 상세는 비공개)
- 다만 NHN Cloud가 OpenStack KVM 기반으로 **CSAP IaaS 정식 취득**이라는 사실은, 해당 환경에서 본 통제 항목을 통과한 사례가 분명히 존재함을 시사
- **추가 검증 수단**: NIA 공공 SaaS 사업 담당자 또는 피어 네트워크를 통해 **NHN/KT 엔지니어의 비공식 정보** 확보 시도 권장

### 5-3. 삼성SDS 자체 선례 (v1)

- SCPv1 VMware 환경에서 **"벤더 미지원"을 사유로 백신 미설치 + 보완통제**로 CSAP 통과한 선례 존재
- v2에서는 "벤더 미지원" 논리는 약화 (Linux용 백신은 존재) → 대신 **"가상화 워크로드 부적합성 + MAC 동등성"** 논리로 전환

## 6. 3개월 검증·적용 로드맵

> MAC은 이미 Ubuntu KVM에 **기본 탑재·동작 중**이지만, SCPv2 전 구간에 대한 체계적 검증·프로파일 튜닝·운영화가 필요. 이 로드맵 자체가 "본 통제 항목을 계속 강화하고 있다"는 심사관 설득의 핵심 근거.

### 6-1. 전체 타임라인

| Phase | 기간 | 목표 산출물 | 담당 |
| --- | --- | --- | --- |
| **Phase 1 (M+0~M+1)** | 1개월 | ① 현황 조사 ② 프로파일 템플릿 확정 ③ 테스트 환경 complain 모드 적용 | CI-TEC |
| **Phase 2 (M+1~M+2)** | 1개월 | ① 테스트 환경 enforce 전환 ② 개발 환경 complain 모드 적용 ③ PoC 결과 보고서 | CI-TEC + SCPv2 운영 |
| **Phase 3 (M+2~M+3)** | 1개월 | ① 운영 환경 Canary(1노드) ② 전 노드 complain ③ 운영 정책 문서화 | SCPv2 운영 |

### 6-2. 심사 시점 대응 상태

- **사전점검(~4/29)**: Phase 1 완료 시점 — "테스트 환경 검증 완료 + 운영 환경 적용 계획 수립" 증적 제시
- **본심사(5/18~22)**: Phase 2 완료 시점 — "개발 환경 운영 중 + 운영 환경 Canary 개시" 증적 제시
- **사후평가(연 1회)**: Phase 3 완료 후 — "전 운영 환경 enforce 모드" 증적 제시

### 6-3. 각 Phase의 검증 기준

| Phase | 합격 기준 |
| --- | --- |
| Phase 1 | Test 환경에서 VM 부팅/마이그레이션/볼륨 attach/콘솔 100% 성공 (complain 모드) |
| Phase 2 | Dev 환경에서 성능 저하 < 3%, AppArmor DENIED 로그 0건 (enforce 모드) |
| Phase 3 | 운영 환경 Canary 2주 무장애, 전 노드 complain 모드 14일간 DENIED 0건 |

### 6-4. 롤백 기준

- VM 부팅 실패율 > 1% → 즉시 complain 모드 회귀
- 라이브 마이그레이션 실패 발생 → 프로파일 수정·재배포
- 고객 장애 신고 발생 → 전체 롤백 + 원인 분석

## 8. 대응 시 주의사항

### 8-1. 반드시 피해야 할 논리

1. ❌ **"벤더 미지원"만으로 방어** → v2에서는 Linux용 AV가 존재하므로 이 논리 자체 성립 불가
2. ❌ **"성능 저하 때문에"만 강조** → 운영 편의 사유로 읽혀 심사관 수용 어려움
3. ❌ **"우리는 안전하니까"식 주관적 주장** → 반드시 공식 문서 + 측정 데이터로 뒷받침

### 8-2. 반드시 포함해야 할 논리

1. ✅ **통제 목적 관점의 재정의** (§1-1)
2. ✅ **4가지 주장의 패키지화** (§1-2)
3. ✅ **MAC 작동 원리의 구체 설명** (§3-1)
4. ✅ **3개월 로드맵의 commitment** (§6) — "검증 중"과 "아무것도 안 함"은 전혀 다름
5. ✅ **롤백·사고 대응 절차** (§03 문서)

### 8-3. 외부 변수 관리

- ⚠️ **KISA 심사관 재량 영역**: 본 대응은 "문언 일탈"이므로 심사관 설득 난이도가 높음. **사전점검 노브레이크와의 사전 조율이 인증 성공의 결정적 변수**
- ⚠️ **피어 CSP 정보**: NHN/KT 엔지니어의 비공식 confirm이 있으면 설득력 급상승. 피어 네트워크 활용 권장
- ⚠️ **운영부서 책임 공유**: 본 방침은 CI-TEC 단독이 아닌 **SCPv2 운영·보안팀 공동 결정** 문서화 필수 (추후 감사 시 책임 분산)
