운영정책서:

### 3-5. Fifth Control: 이미지·스크립트 사전 검사 계층 (신설)

MAC이 권한 상승·비정상 접속 차단에 강력하지만, 악성코드가 **nova 이미지·스크립트 형태로 미리 업로드**되는 경우를 방지하기 위해 다음 계층을 추가로 운영한다.

| 통제 | 내용 | 담당 | 증적 |
|------|------|------|------|
| **파일 무결성 검사** | nova 이미지 업로드 시 서명 검증 (GPG/코사인) 필수 | CI-TEC | 서명 검증 로그 |
| **업로드 경로 최소화** | `/var/lib/nova/instances/` 외 경로 업로드 금지 + Ansible Tower 경유만 허용 | SCPv2 운영 | 경로 ACL + 감사 로그 |
| **패치·라이브러리 저장소 스캔** | Canonical Ubuntu Pro 저장소 + 내부 미러 주 1회 ClamAV 스캔 | CI-TEC | 스캔 보고서 |
| **Ansible 스크립트 코드 스캔** | 배포 전 Trivy / Semgrep으로 악성 패턴·취약점 스캔 | CI-TEC | 스캔 결과 + 승인 기록 |

**FIM(File Integrity Monitoring)이란?**

### 1. FIM의 정의

**FIM(File Integrity Monitoring)**은 **파일 무결성 모니터링**을 의미합니다.

시스템 내 **중요 파일**의 내용, 속성(권한, 소유자, 타임스탬프 등)이 **무단으로 변경**되었는지 실시간 또는 주기적으로 감시하는 보안 기술입니다.

간단히 말해:
> "우리 시스템의 중요한 파일이 누군가 몰래 바꿨는지 자동으로 감시하는 감시 카메라"입니다.

---

### 2. 왜 FIM이 필요한가?

악성코드나 공격자는 보통 다음과 같은 행동을 합니다:

- 시스템 파일을 변조 (예: `/etc/passwd`, `/bin/login` 등)
- 백도어 설치
- 설정 파일 변경
- 악성 스크립트 삽입

이때 **MAC(sVirt/AppArmor)**은 "프로세스가 할 수 있는 행동을 제한"하는 **예방 통제**라면,  
**FIM**은 "이미 발생한 변경을 빠르게 탐지"하는 **탐지 통제**입니다.

즉, **MAC이 막지 못한 변경을 FIM이 잡아내는 2차 방어선** 역할을 합니다.

---

### 3. FIM의 동작 원리

FIM은 보통 다음과 같은 방식으로 동작합니다:

1. **베이스라인(기준선) 생성**  
   처음에 시스템의 중요 파일 상태를 스냅샷으로 저장 (해시값, 권한, 소유자 등)

2. **주기적 검사**  
   일정 주기(예: 매일 새벽 3시)마다 현재 파일 상태를 베이스라인과 비교

3. **변경 탐지**  
   차이가 발생하면 즉시 **Alert** 발생 (이메일, SIEM, Slack 등)

4. **대응**  
   변경 내역을 분석 → 정상 변경인지, 악성 변경인지 판단

---

### 4. 이 프로젝트에서 사용하는 FIM (AIDE)

문서들에서 FIM 도구로 **AIDE(Advanced Intrusion Detection Environment)**를 사용합니다.

#### 주요 특징

| 항목 | 내용 |
|------|------|
| 도구 이름 | **AIDE** |
| 설치 패키지 | `aide`, `aide-common` |
| 감시 대상 | `/etc/`, `/bin/`, `/sbin/`, `/usr/bin/`, `/boot/`, `/lib/` 등 |
| **제외 대상** | `/var/lib/nova/instances/` (VM 디스크는 계속 변하므로 제외) |
| 실행 주기 | **매일 새벽 3시** (cron으로 자동 실행) |
| 보고 방식 | 변경 발생 시 SIEM(Splunk)으로 자동 전송 + 운영팀 Alert |

#### 실제 설정 예시 (aide.conf)

```bash
# 감시할 디렉터리
/boot           NORMAL
/etc            NORMAL
/bin            NORMAL
/usr/bin        NORMAL
/usr/sbin       NORMAL

# 제외할 디렉터리 (가상화 환경 특성상)
/var/lib/nova/instances   !
/var/lib/libvirt/qemu     !
/var/log                  !
```

### 5. MAC과 FIM의 관계 (중요!)

| 구분 | MAC (sVirt/AppArmor) | FIM (AIDE) |
|------|----------------------|------------|
| 목적 | **예방** (권한 상승·비정상 접근 차단) | **탐지** (파일 변경 감지) |
| 동작 시점 | 실시간 (커널 레벨) | 주기적 (매일) |
| 강점 | Zero-day 공격에도 강력 | 이미 발생한 변조를 빠르게 발견 |
| 약점 | 설정이 복잡, 로그가 많이 쌓임 | 변경 후에야 알 수 있음 |
| 상호 보완 | MAC이 막지 못한 변경을 FIM이 잡음 | FIM이 탐지한 변경을 MAC이 재발 방지 |

**결론**:  
MAC과 FIM은 **서로 보완 관계**입니다.  
문서에서는 이 둘을 함께 운영하는 **4종 보완통제 패키지**의 핵심 요소로 설명하고 있습니다.

---

### 6. CSAP 관점에서의 중요성

CSAP 9.1.4-③에서 FIM은 다음과 같은 이유로 매우 중요합니다:

- MAC만으로는 **"이미 업로드된 악성 이미지"**나 **"설정 파일 변조"**를 완전히 막기 어렵다.
- FIM이 변경을 탐지하면 → 즉시 대응 가능 (03_악성코드_대응절차서.md 참조)
- 심사관에게 **"예방 + 탐지"**라는 이중 방어를 갖추었다는 증거가 된다.

---

### 7. 증적 수집 방법 (실제 문서에서 나오는 내용)

문서 `02_보완통제_증적수집_가이드.md`에 따르면:

- 매일 AIDE 검사 결과 로그 자동 생성
- 변경 탐지 시 SIEM으로 자동 전송
- 최근 30일간 탐지·조치 이력 문서화 필요
- 베이스라인 갱신은 커널 패치나 승인된 변경 시에만 수행

> **FIM은 파일이 몰래 바뀌었는지 감시하는 보안 감시 시스템**이며, MAC이 "막는" 역할을 한다면 FIM은 "발견하는" 역할을 합니다. 이 프로젝트에서는 **AIDE**를 사용하여 매일 중요 파일을 검사하고, 변경 발생 시 즉시 Alert를 띄우는 방식으로 운영하고 있습니다.

**SIEM이란?**

### 1. SIEM의 정의

**SIEM**은 **Security Information and Event Management**의 약자입니다.

한국어로는 **보안 정보 및 이벤트 관리**라고 합니다.

간단히 말해:
> **기업의 모든 보안 로그와 이벤트를 한 곳에서 수집·분석·모니터링하는 통합 보안 관제 시스템**입니다.

---

### 2. SIEM의 주요 기능

SIEM은 보통 다음과 같은 기능을 수행합니다:

| 기능 | 설명 |
|------|------|
| **로그 수집** | 방화벽, 서버, 애플리케이션, MAC(AppArmor), FIM(AIDE), 백신 등 다양한 장비의 로그를 실시간으로 수집 |
| **로그 분석** | 수많은 로그 중에서 **의심스러운 패턴**을 자동으로 찾아냄 |
| **이벤트 상관 분석 (Correlation)** | 여러 로그를 종합해서 "이건 공격이다"라고 판단 (예: MAC DENIED + FIM 변경 + 비정상 네트워크 접속 동시 발생) |
| **실시간 모니터링** | 대시보드를 통해 현재 보안 상태를 한눈에 확인 |
| **Alert (알림)** | 이상 징후가 발견되면 즉시 보안팀에게 알림 (이메일, Slack, SMS 등) |
| **로그 보존** | 법적 요구사항에 따라 로그를 장기 보관 (보통 1~5년) |

---

### 3. 왜 SIEM이 중요한가?

현대 보안 환경에서는 **로그가 너무 많아서** 사람이 일일이 다 볼 수 없습니다.

예를 들어:
- 한 대의 서버에서 하루에 수십만~수백만 건의 로그가 발생
- MAC(AppArmor)만 해도 DENIED 로그가 계속 쌓임

이때 **SIEM**이 없으면:
- 중요한 보안 이벤트(침해 시도)를 놓칠 수 있음
- 사고 발생 후 원인 분석이 매우 어려움

**SIEM은 "보안의 두뇌"** 역할을 합니다.

---

### 4. 이 프로젝트에서 SIEM의 역할

문서들(`02_보완통제_증적수집_가이드.md`, `01_운영정책서.md`, `03_악성코드_대응절차서.md`)에서 SIEM은 다음과 같이 사용됩니다:

| 용도 | 내용 |
|------|------|
| **AppArmor DENIED 로그** | 모든 DENIED 로그를 **실시간으로 Splunk(SIEM)**에 전송 |
| **AIDE(FIM) 변경 탐지** | 파일 변경이 감지되면 SIEM으로 자동 Alert 전송 |
| **통합 모니터링** | MAC + FIM + 네트워크 로그를 한 곳에서 함께 분석 |
| **임계치 Alert** | 예: "1시간 내 DENIED 로그 100건 이상" 발생 시 SOC에 즉시 알림 |
| **감사 증적** | CSAP 심사 시 로그 보존 및 분석 증거로 제출 |

**특히 중요한 점**:
- MAC은 로그가 많이 발생할 수 있는데, **SIEM이 없으면 로그 폭주로 인해 중요한 신호를 놓칠 수 있음**
- 따라서 문서에서는 **"MAC 로그 중앙관리 및 모니터링이 필수"**라고 명시하고 있습니다.

---

### 5. MAC + FIM + SIEM의 관계 (한눈에 보기)

| 도구 | 역할 | 언제 작동? | 누가 관리? |
|------|------|------------|------------|
| **MAC (AppArmor)** | 예방 (권한 상승 차단) | 실시간 | 커널 |
| **FIM (AIDE)** | 탐지 (파일 변경 감지) | 매일 주기적 | AIDE + SIEM |
| **SIEM (Splunk)** | 분석 + 모니터링 + Alert | 실시간 | 보안팀 (SOC) |

**요약**:
- **MAC** = 문을 잠그는 역할
- **FIM** = 누가 문을 열었는지 감시하는 역할
- **SIEM** = 모든 감시 카메라 영상을 한 곳에서 보고, 이상한 일이 있으면 바로 알려주는 **통합 관제 센터**

---

### 6. 실제로 많이 쓰이는 SIEM 제품

- **상용**: Splunk, IBM QRadar, Microsoft Sentinel, LogRhythm
- **오픈소스**: ELK Stack (Elasticsearch + Logstash + Kibana), Graylog

이 프로젝트에서는 **Splunk**를 사용하고 있는 것으로 문서에 나와 있습니다.

---

**한 줄 요약**:

> **SIEM은 수많은 보안 로그를 한곳에 모아서 분석하고, 이상 징후가 있으면 즉시 알려주는 보안 관제 시스템**입니다.  
> MAC과 FIM이 탐지한 정보를 SIEM이 종합적으로 분석·관리하기 때문에, 현대 보안 환경에서 **필수적인 3대장** 중 하나입니다.
