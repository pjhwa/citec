**✅ 의견 분석 및 대응 방안 보고서**

### 1. 제공된 의견 요약

> MAC은 **권한 상승·비정상 접속 차단**이 주 목적이며, **EPP(백신)처럼 악성코드 탐지**를 주 목적으로 할 수 없다.  
> 따라서 **nova에 업로드되는 파일(이미지, 스크립트 등)에 대한 사전 검사**가 반드시 필요하며, MAC 통제 시 발생하는 **수많은 로그에 대한 중앙관리·모니터링**도 필수다.

구체적 요구사항:
- 파일 무결성 검사 (서명 등)
- 서버 내 파일 업로드 경로 최소화
- 라이브러리·패치 저장소 주기적 악성코드 스캔
- Ansible 배포 스크립트 코드 스캔
- MAC 로그 중앙관리 및 모니터링

### 3. 부족한 부분 (추가 대응이 필요한 영역)

| 부족 영역 | 현재 문서 상태 | 위험도 |
|-----------|----------------|--------|
| **nova 업로드 파일(이미지, 스크립트) 사전 검사** | 전혀 명시 없음 | **높음** |
| **패치·라이브러리 저장소 악성코드 스캔** | 간접적 언급만 | 중간 |
| **Ansible 배포 스크립트 코드 스캔** | 전혀 없음 | 중간 |
| **MAC 로그 폭주 대응 (중앙관리·모니터링 강화)** | SIEM 연동만 언급, 구체적 대응 부족 | 중간 |

**가장 큰 Gap**:  
MAC이 "악성코드 탐지"가 아닌 "권한 상승·비정상 접속 차단"이라는 점을 인정하면서도, **nova에 업로드되는 파일 자체의 악성코드 유입 경로를 차단**하는 계층이 명확히 드러나지 않는다.

### 4. 추가 대응 방안 (추천 내용)

#### 4-1. 보완통제 5종 패키지로 확장 제안 (가장 강력 추천)

기존 4종 보완통제에 **5번째 통제**를 추가:

| # | 통제 | 역할 | 증적 |
|---|------|------|------|
| 1 | MAC (sVirt/AppArmor) | 프로세스 단위 강제 격리 | aa-status, virsh capabilities |
| 2 | FIM (AIDE) | 호스트 파일 변경 탐지 | 일일 보고서 |
| 3 | 호스트 OS 최소화 | Attack surface 축소 | dpkg 인벤토리 |
| 4 | 관리망 분리 | 외부 유입 경로 차단 | VLAN, ACL, Bastion |
| **5** | **이미지·스크립트 사전 검사 계층** | **nova 업로드 파일 악성코드 유입 차단** | 서명 검증 + 코드 스캔 + 업로드 경로 제한 |

#### 4-2. 구체적 대응 내용 (문서에 추가할 텍스트)

**추가할 내용 (01_운영정책서.md §3-5 신설 추천)**:

```markdown
### 3-5. Fifth Control: 이미지·스크립트 사전 검사 계층 (신설)

MAC이 권한 상승·비정상 접속 차단에 강력하지만, 악성코드가 **nova 이미지·스크립트 형태로 미리 업로드**되는 경우를 방지하기 위해 다음 계층을 추가로 운영한다.

| 통제 | 내용 | 담당 | 증적 |
|------|------|------|------|
| **파일 무결성 검사** | nova 이미지 업로드 시 서명 검증 (GPG/코사인) 필수 | CI-TEC | 서명 검증 로그 |
| **업로드 경로 최소화** | `/var/lib/nova/instances/` 외 경로 업로드 금지 + Ansible Tower 경유만 허용 | SCPv2 운영 | 경로 ACL + 감사 로그 |
| **패치·라이브러리 저장소 스캔** | Canonical Ubuntu Pro 저장소 + 내부 미러 주 1회 ClamAV 스캔 | CI-TEC | 스캔 보고서 |
| **Ansible 스크립트 코드 스캔** | 배포 전 Trivy / Semgrep으로 악성 패턴·취약점 스캔 | CI-TEC | 스캔 결과 + 승인 기록 |
```

**03_악성코드_대응절차서.md**에 추가할 내용:

```markdown
### 1-3. 탐지 채널 확장 (신설)

기존 3채널(C1: MAC DENIED, C2: FIM, C3: 네트워크 이상)에 **C4: 업로드 파일 사전 검사 실패**를 추가한다.

탐지 기준: 위 4개 중 **2개 이상 동시 트리거** 시 침해사고 의심으로 격상.
```

**로그 중앙관리 강화 (02_보완통제_증적수집_가이드.md E-05 보강)**:

```markdown
### E-05 보강: MAC 로그 중앙관리·모니터링

- 모든 AppArmor DENIED 로그는 **실시간 Splunk**로 전송
- **주간 DENIED 통계 대시보드** 운영 (주 1회 정보보호팀 검토)
- **임계치 Alert**: 단일 호스트 1시간 내 DENIED 50건 이상 → SOC 즉시 에스컬레이션
- **로그 보존 기간**: 5년 (CSAP 요구사항 충족)
```
