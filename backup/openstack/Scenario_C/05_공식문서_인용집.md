## 2. C-01: OpenStack Security Guide — Hardening the virtualization layers

**원문 URL**: <https://docs.openstack.org/security-guide/compute/hardening-the-virtualization-layers.html>

### 2-1. 원문 (영어)

> "We recommend three specific steps:
> - Minimizing the code base.
> - Using compiler hardening.
> - **Using mandatory access controls such as sVirt, SELinux, or AppArmor.**"

### 2-2. 한국어 요약

OpenStack 공식 보안 가이드는 하이퍼바이저(QEMU/KVM) 보안 강화를 위해 다음 3가지를 권고한다:
1. 코드베이스 최소화
2. 컴파일러 하드닝
3. **강제접근통제(MAC) 사용 — sVirt, SELinux, 또는 AppArmor**

### 2-3. 본 대응과의 연계

- **AV는 3대 권고 항목에 포함되지 않는다.** 본 가이드는 OpenStack 커뮤니티가 수년에 걸쳐 정제한 공식 권고이며, AV가 필수라고 판단했다면 반드시 포함됐을 것.
- 반대로 **MAC을 명시적으로 권고**한다. 본 방침이 권고하는 sVirt(AppArmor)는 가이드 권고와 정확히 일치.

## 3. C-02: OpenStack Security Guide — Hardening Compute deployments

**원문 URL**: <https://docs.openstack.org/security-guide/compute/hardening-deployments.html>

### 3-1. 원문 (영어)

> "If your deployment does not require full virtual machine backups, we recommend excluding the /var/lib/nova/instances directory as it will be as large as the combined space of each vm running on that node."

> "All such sensitive files should be given strict file level permissions, and monitored for changes through **file integrity monitoring (FIM) tools such as iNotify or Samhain**."

### 3-2. 한국어 요약

- `/var/lib/nova/instances` 디렉터리는 해당 노드에서 실행 중인 모든 VM의 합산 크기와 같으므로 **백업에서 제외할 것을 권고**.
- 호스트의 민감 파일은 엄격한 권한 부여와 함께 **파일 무결성 모니터링(FIM) 도구(iNotify, Samhain 등)로 변경 감시**할 것을 권고.

### 3-3. 본 대응과의 연계

- OpenStack 공식 가이드가 호스트 무결성 보호 수단으로 **AV가 아닌 FIM을 명시** — 본 방침의 AIDE 도입 논리와 직접 일치.
- `/var/lib/nova/instances`의 특수성(대용량 sparse 파일)을 공식이 인정 — AV 실시간 스캔의 부적합성 근거.

## 4. C-03: OpenStack Security Guide — Hypervisor selection

**원문 URL**: <https://docs.openstack.org/security-guide/compute/hypervisor-selection.html>

### 4-1. 원문 (영어)

> "NIST provides additional guidance in Special Publication 800-125, 'Guide to Security for Full Virtualization Technologies'."

### 4-2. 한국어 요약

OpenStack Security Guide는 하이퍼바이저 선택 시 **NIST SP 800-125 『완전 가상화 기술을 위한 보안 가이드』** 를 추가 참조 문서로 권고한다.

### 4-3. 본 대응과의 연계

- NIST SP 800-125의 핵심 권고는 **attack surface 최소화 + 격리 + 무결성** (C-10 참조).
- 추가 에이전트(AV) 설치는 attack surface **확대 방향** → 본 NIST 원칙과 상충.
- 본 방침이 채택한 "호스트 최소화 + MAC 격리 + FIM 무결성" 3종 설계는 NIST 원칙과 정확히 일치.

## 5. C-04: Ubuntu Server Documentation — Libvirt

**원문 URL**: <https://ubuntu.com/server/docs/how-to/virtualisation/libvirt/>

### 5-1. 원문 (영어)

> "By default, libvirt will spawn QEMU guests using AppArmor isolation for enhanced security."

### 5-2. 한국어 요약

**Ubuntu의 libvirt는 보안 강화를 위해 기본적으로 AppArmor 격리 환경에서 QEMU 게스트를 스폰(spawn)한다.**

### 5-3. 본 대응과의 연계

- SCPv2 컴퓨트 노드(Ubuntu)는 **libvirt 설치 시점부터 AppArmor sVirt가 이미 기본 활성 상태**.
- 즉, 본 대응은 "새로운 보호 수단을 추가하는 것"이 아니라 "**이미 존재하는 보호 수단을 체계화·운영화·검증**"하는 것.
- 사전점검 시점에도 "이미 작동 중"이라는 증적(`aa-status` 출력) 제시 가능.

## 6. C-05: libvirt.org — QEMU/KVM hypervisor driver

**원문 URL**: <https://libvirt.org/drvqemu.html>

### 6-1. 원문 (영어)

> "The AppArmor sVirt protection for QEMU virtual machines builds on this basic level of protection, to also allow individual guests to be protected from each other. In the sVirt model, if a profile is loaded for the libvirtd daemon, then each qemu:///system QEMU virtual machine will have a profile created for it when the virtual machine is started if one does not already exist. This generated profile uses a profile name based on the UUID of the QEMU virtual machine and contains rules allowing access to only the files it needs to run, such as its disks, pid file and log files."

### 6-2. 한국어 요약

- sVirt의 **AppArmor 보호는 VM 간 상호 격리**를 가능하게 함.
- libvirtd 데몬에 프로파일이 로드되면, **각 QEMU 가상머신이 시작될 때 VM의 UUID 기반 고유 프로파일이 자동 생성**됨.
- 생성된 프로파일은 해당 VM이 실행에 필요한 파일(디스크, pid 파일, 로그 파일 등)에만 접근 허용.

### 6-3. 본 대응과의 연계

- **AV가 할 수 없는 "VM 간 격리"를 sVirt가 커널 레벨에서 구현**.
- VM별로 고유한 접근 경로만 허용 → 한 VM이 탈출해도 다른 VM의 디스크·메모리 접근 원천 차단.
- 이는 전통 AV(호스트 파일시스템 스캔)이 제공하지 못하는 보호 수단.

## 7. C-06: Red Hat RHEL — Virtualization Security Guide (sVirt 원리)

**원문 URL**: <https://docs.redhat.com/en/documentation/red_hat_enterprise_linux/7/html/virtualization_security_guide/chap-virtualization_security_guide-svirt>

### 7-1. 원문 (영어)

> "Each KVM-based virtual machine is a process which is labeled by SELinux, effectively establishing a security boundary around each virtual machine. This security boundary is monitored and enforced by the Linux kernel, restricting the virtual machine's access to resources outside of its boundary, such as host machine data files or other VMs."

### 7-2. 한국어 요약

- 각 KVM 기반 VM은 **SELinux에 의해 레이블된 프로세스**로서, VM 주변에 보안 경계를 효과적으로 수립.
- 이 보안 경계는 **Linux 커널에 의해 모니터링·강제**되며, VM이 자신의 경계 외부 리소스(호스트 데이터 파일, 다른 VM 등)에 접근하는 것을 제한.

### 7-3. 본 대응과의 연계

- 본 인용은 **Ubuntu AppArmor에도 동일하게 적용**되는 개념 (libvirt는 둘 다 지원).
- 즉, 본 방침의 sVirt(AppArmor)는 Red Hat 엔터프라이즈 환경에서 수년간 검증된 보호 수단.
- "커널이 강제하는 경계"는 사용자 공간 AV보다 구조적으로 강력함 (root 권한을 가진 공격자도 우회 불가).

## 8. C-07: Mirantis MCP — Compute and hypervisor security

**원문 URL**: <https://docs.mirantis.com/mcp/q4-18/mcp-security-best-practices/openstack/compute-and-hypervisor-security.html>

### 8-1. 원문 (영어)

> "Enable mandatory access control (MAC) with SELinux, AppArmor, or grsecurity."

### 8-2. 한국어 요약

Mirantis(상용 OpenStack 벤더) 보안 베스트 프랙티스는 하이퍼바이저 OS 보호의 핵심으로 **"SELinux, AppArmor, 또는 grsecurity로 MAC을 활성화"** 할 것을 권고.

### 8-3. 본 대응과의 연계

- Mirantis는 상용 OpenStack의 대표 벤더로, **엔터프라이즈 운영 현장의 검증된 베스트 프랙티스** 제공.
- 여기에도 AV 권고는 없음 — MAC이 하이퍼바이저 OS 보호의 핵심이라는 업계 컨센서스 확인.

## 9. C-08: AWS Nitro System Security Whitepaper

**원문 URL**: <https://docs.aws.amazon.com/whitepapers/latest/security-design-of-aws-nitro-system/>

### 9-1. 주요 내용 (한국어 요약)

- AWS Nitro 호스트는 **"no shell access, no SSH, no interactive operator access"** 로 설계됨.
- 호스트 내 실행 프로세스는 **암호학적으로 서명된 펌웨어**로만 실행.
- 결과: AV를 설치할 운영체제 인터페이스 자체가 **존재하지 않음**.

### 9-2. 본 대응과의 연계

- AWS가 본 통제 항목의 본질을 "**AV 설치 여부**"가 아닌 "**하이퍼바이저 무결성 보호**"로 해석하고 있음을 공식 백서로 증명.
- 글로벌 최대 CSP가 공개적으로 채택한 접근이 본 방침과 동일 방향.

## 10. C-09: Microsoft Azure + KISA CSAP 선례

**관련 URL**:
- Azure Hypervisor Security: https://learn.microsoft.com/en-us/azure/security/fundamentals/hypervisor
- KISA CSAP 발표: Microsoft 공식 발표 (2024년 11월 20일)

### 10-1. 주요 내용 (한국어 요약)
- Microsoft Azure는 **2024년 11월 20일, 글로벌 CSP 최초로 CSAP "하" 등급(IaaS, Low tier)** 인증을 KISA로부터 취득.
- Azure 하이퍼바이저 보안 모델: **VBS (Virtualization-Based Security) + HVCI + Secured-Core** 기반 (전통 AV 미설치).
- KISA가 Azure의 "전통 백신 미설치 + 플랫폼 차원 강제 격리·무결성 보호" 방식을 **CSAP 심사에서 수용**한 공식 선례.

### 10-2. 본 대응과의 연계 (가장 강력한 근거)
- **KISA가 이미 2024년 11월 20일에 "AV 미설치 + MAC/HVCI 등 대체 통제" 방식을 CSAP "하" 등급으로 인증**.
- 이는 본 방침의 핵심 주장(문언 준수 대신 **통제 목적 동등 달성**)을 KISA 스스로가 수용한 **국내 공식 전례**.
- Azure 하이퍼바이저 보안 모델은 본 방침의 sVirt(AppArmor) + FIM + 최소화와 본질적으로 동일한 방향성.

**심사 현장 활용 포인트**:
> "KISA가 2024년 11월 20일 Azure를 CSAP '하' 등급으로 인증할 때, Azure도 전통 백신을 하이퍼바이저에 설치하지 않았습니다. KISA가 이미 동일한 논리를 수용한 전례."

## 11. C-10: NIST SP 800-125 — Guide to Security for Full Virtualization Technologies

**원문 URL**: <https://csrc.nist.gov/pubs/sp/800/125/final>

### 11-1. 주요 내용 (한국어 요약)

가상화 호스트(하이퍼바이저) 보안의 3대 원칙:

1. **Attack Surface 최소화**
   - 호스트 OS에 불필요한 서비스·패키지·에이전트 최소화
   - 관리 인터페이스 접근 엄격 제한

2. **격리(Isolation)**
   - VM 간 격리 및 호스트와 VM 간 격리 보장
   - MAC, 네트워크 분리, cgroups 등 다층 격리

3. **무결성(Integrity)**
   - 호스트 OS 및 하이퍼바이저 무결성 모니터링
   - Secure Boot, TPM, FIM 등 활용

### 11-2. 본 대응과의 연계

| NIST 원칙 | 본 방침의 구현 |
| --- | --- |
| 1. Attack Surface 최소화 | 호스트 OS 최소화 + 관리망 분리 + AV 미설치 |
| 2. 격리 | sVirt(AppArmor) + 네트워크 VLAN 분리 |
| 3. 무결성 | AIDE FIM + AppArmor DENIED 로그 |

- NIST 공식 가이드의 3대 원칙이 본 방침의 4종 보완통제와 **정확히 대응**.
- **AV 설치는 원칙 1(최소화)과 정면 충돌**하는 방향 — 공식 표준이 본 대응의 정당성을 뒷받침.
