본 문서는 SCPv2의 CSAP 9.1.4-③ 대응 논리에 인용되는 **모든 공식 문서의 원문과 한국어 요약**을 한 곳에 모은 것이다. 본심사 시 심사관이 "이 주장의 근거가 무엇인가"를 질의할 경우, 본 문서의 해당 항목을 직접 제시할 수 있다.

각 항목은 ① 인용 위치(우리 문서 어디에 사용되었는가) ② 영어 원문 ③ 한국어 요약 ④ 출처 URL ⑤ 우리 대응에서의 의미 순으로 정리.

## 1. OpenStack 공식 보안 가이드

### 1-A. 하이퍼바이저 강화의 3대 권고사항 — MAC 우선

**인용 위치**: 운영정책서 §3.2.2, README §2-2

**영어 원문**:
> "It is important to take proactive steps to harden QEMU. We recommend three specific steps:
> - Minimizing the code base.
> - Using compiler hardening.
> - Using mandatory access controls such as sVirt, SELinux, or AppArmor."

**한국어 요약**: OpenStack 보안 가이드는 QEMU/KVM 하이퍼바이저를 강화하기 위한 3대 권고사항으로 ① 코드베이스 최소화, ② 컴파일러 하드닝, ③ **강제접근통제(MAC; sVirt·SELinux·AppArmor)** 사용을 명시한다. **백신(AV)은 권고 항목에 포함되지 않으며, MAC이 본질적 보호 수단으로 제시됨.**

**출처**: https://docs.openstack.org/security-guide/compute/hardening-the-virtualization-layers.html
**문서 발행 주체**: OpenStack Security Group (OSSG) — OpenStack Foundation 공식 보안 워킹그룹

**우리 대응에서의 의미**: 호스트에 백신을 추가하는 것보다 sVirt(AppArmor)로 VM 격리를 강화하는 것이 OpenStack 공식 권고에 더 부합한다. SCPv2는 Ubuntu KVM의 기본 sVirt가 자동 작동 중이며, 백신은 보완수단으로 운영한다.

---

### 1-B. /var/lib/nova/instances 디렉터리 특수성 인정

**인용 위치**: 운영정책서 §3.2.1, §5.1

**영어 원문**:
> "If your deployment does not require full virtual machine backups, we recommend excluding the /var/lib/nova/instances directory as it will be as large as the combined space of each vm running on that node."

**한국어 요약**: 백업 시 `/var/lib/nova/instances` 디렉터리는 해당 노드의 모든 VM 디스크 합산만큼 거대하므로, 전체 VM 백업이 필요하지 않다면 **제외할 것을 권고**한다.

**출처**: https://docs.openstack.org/security-guide/compute/hardening-deployments.html

**우리 대응에서의 의미**: 동일 디렉터리는 백신 실시간 스캔에서도 제외 정당성을 갖는다. qcow2/raw 이미지는 VM 메모리·디스크 변경에 따라 끊임없이 변동되는 거대 sparse 파일이며, 호스트 OS 관점에서 "사용자 파일"이 아닌 "가상화 백엔드 데이터"이다.

---

### 1-C. 파일 무결성 모니터링(FIM) 권고

**인용 위치**: 운영정책서 §9, 보완통제 가이드 §2

**영어 원문**:
> "One of the main security concerns with any OpenStack deployment is the security and controls around sensitive files, such as the nova.conf file. Normally contained in the /etc directory, this configuration file contains many sensitive options including configuration details and service passwords. **All such sensitive files should be given strict file level permissions, and monitored for changes through file integrity monitoring (FIM) tools such as iNotify or Samhain.**"

**한국어 요약**: OpenStack의 민감 설정 파일(`nova.conf` 등)은 엄격한 파일 권한 부여 및 **iNotify·Samhain 같은 FIM 도구로 변경을 모니터링**해야 한다.

**출처**: https://docs.openstack.org/security-guide/compute/hardening-deployments.html

**우리 대응에서의 의미**: 호스트 무결성 보호의 공식 권고 도구는 백신이 아닌 **FIM**이다. SCPv2는 AIDE를 운영하여 본 권고를 충족한다.

---

### 1-D. NIST SP 800-125 가상화 보안 가이드 인용

**인용 위치**: README §2-2, 비교표 §2.3

**영어 원문**:
> "NIST provides additional guidance in Special Publication 800-125, 'Guide to Security for Full Virtualization Technologies'. As part of your hypervisor selection process, you must consider a number of important factors to help increase your security posture."

**한국어 요약**: 하이퍼바이저 선택 시 NIST SP 800-125 "Guide to Security for Full Virtualization Technologies"를 참고하라는 OpenStack 공식 권고. NIST 가이드는 가상화 호스트 보안의 핵심 원칙으로 ① attack surface 최소화, ② 격리, ③ 무결성 검증을 제시한다.

**출처**: https://docs.openstack.org/security-guide/compute/hypervisor-selection.html

**우리 대응에서의 의미**: NIST의 권고 방향은 호스트 OS에 추가 에이전트(AV)를 늘리는 것이 아니라 minimal OS + MAC + 무결성 검증이다. 이는 SCPv2의 보완통제 패키지와 일치한다.

## 2. Ubuntu Server 공식 문서

### 2-A. libvirt의 AppArmor sVirt 기본 활성

**인용 위치**: 운영정책서 §9, 보완통제 가이드 §1.1, README §2-2

**영어 원문**:
> "**By default, libvirt will spawn QEMU guests using AppArmor isolation for enhanced security.** The AppArmor rules for a guest will consist of multiple elements:
> - A static part that all guests share => /etc/apparmor.d/abstractions/libvirt-qemu
> - A dynamic part created at guest start time and modified on hotplug/unplug => /etc/apparmor.d/libvirt/libvirt-f9533e35-6b63-45f5-96be-7cccc9696d5e.files"

**한국어 요약**: Ubuntu Server에서 libvirt는 **기본적으로 AppArmor 격리 환경에서 QEMU 게스트를 spawn**한다. 각 VM은 ① 모든 게스트가 공유하는 정적 규칙(`/etc/apparmor.d/abstractions/libvirt-qemu`)과 ② VM 시작 시 동적으로 생성되는 VM별 규칙(`/etc/apparmor.d/libvirt/libvirt-<UUID>.files`)으로 보호된다.

**출처**: https://ubuntu.com/server/docs/how-to/virtualisation/libvirt/
**문서 발행 주체**: Canonical Ltd. (Ubuntu 공식 문서)

**우리 대응에서의 의미**: SCPv2는 Ubuntu LTS를 호스트로 사용하므로, **추가 솔루션 설치 없이도 sVirt(AppArmor) 보호가 자동 작동 중**이다. 본 문서를 근거로 컴퓨트 노드의 AppArmor 활성 상태(`aa-status`)를 보여주면 보완통제 작동의 강력한 입증이 된다.

## 3. libvirt.org 공식 문서

### 3-A. sVirt 보안 모델 동작 원리

**인용 위치**: 보완통제 가이드 §1.2, §1.3

**영어 원문**:
> "The AppArmor sVirt protection for QEMU virtual machines builds on this basic level of protection, to also allow individual guests to be protected from each other. In the sVirt model, if a profile is loaded for the libvirtd daemon, then **each qemu:///system QEMU virtual machine will have a profile created for it when the virtual machine is started** if one does not already exist. This generated profile uses a profile name based on the UUID of the QEMU virtual machine and **contains rules allowing access to only the files it needs to run, such as its disks, pid file and log files.**"

**한국어 요약**: libvirt의 sVirt 보안 모델은 각 QEMU VM에 대해 시작 시 자동으로 AppArmor 프로파일을 생성하며, 해당 프로파일은 VM의 UUID 기반 이름을 가지고 **자신이 실행에 필요한 파일(자신의 디스크, PID 파일, 로그 파일)에만 접근**할 수 있도록 규칙이 정의된다.

**출처**: https://libvirt.org/drvqemu.html
**문서 발행 주체**: libvirt 프로젝트 (Red Hat 주도, OpenStack의 핵심 의존성)

**우리 대응에서의 의미**: 이는 전통적 백신이 보호하려는 "프로세스가 임의 파일 변조" 시나리오를 **커널 차원에서 원천 봉쇄**한다. 백신의 시그니처 기반 보호보다 zero-day에 대한 구조적 보호력이 더 큼.

---

### 3-B. SELinux sVirt (참조용)

**인용 위치**: 보완통제 가이드 §1 일반

**영어 원문**:
> "In the sVirt model, each QEMU virtual machine runs under its own confined domain, which is based on system_u:system_r:svirt_t:s0 with a unique category appended, eg, system_u:system_r:svirt_t:s0:c34,c44. The rules are setup such that a domain can only access files which are labelled with the matching category level, eg system_u:object_r:svirt_image_t:s0:c34,c44. **This prevents one QEMU process accessing any file resources that are prevent to another QEMU process.**"

**한국어 요약**: SELinux sVirt는 각 QEMU VM에 고유 카테고리 레이블(`svirt_t:s0:c34,c44`)을 부여하고, VM 디스크 이미지에도 동일 카테고리(`svirt_image_t`)를 부여하여 **VM 간 파일 접근을 원천 차단**한다. (Ubuntu는 AppArmor를 사용하지만 동일 사상)

**출처**: https://libvirt.org/drvqemu.html

## 4. Mirantis MCP 보안 베스트 프랙티스

### 4-A. 하이퍼바이저 보호의 권고 통제

**인용 위치**: README §2-2

**영어 원문**:
> "Limit an access by providing strict access permissions to the nova.conf file and /var/lib/nova folder. Use file integrity monitoring (FIM) tools such as iNotify or Samhain to trace unexpected files modifications. Use rootwrap to execute Compute commands as the root user. Use trusted compute pools. Implement centralized logging to mitigate a risk of a repudiation attack. **Enable mandatory access control (MAC) with SELinux, AppArmor, or grsecurity.**"

**한국어 요약**: Mirantis(상용 OpenStack 벤더) 보안 베스트 프랙티스는 OpenStack 컴퓨트 노드 보호 통제로 ① `nova.conf`/`/var/lib/nova` 권한 제한, ② FIM 운영, ③ rootwrap 사용, ④ trusted compute pools, ⑤ 중앙 집중 로깅, ⑥ **MAC(SELinux/AppArmor/grsecurity) 활성화** 를 제시한다. **백신은 권고 항목에 포함되지 않음.**

**출처**: https://docs.mirantis.com/mcp/q4-18/mcp-security-best-practices/openstack/compute-and-hypervisor-security.html
**문서 발행 주체**: Mirantis (OpenStack 상용 배포판 벤더, 글로벌 OpenStack 시장 점유율 상위)

**우리 대응에서의 의미**: 상용 OpenStack 벤더의 공식 보안 가이드도 백신이 아닌 MAC + FIM을 권고하고 있어, SCPv2의 보완통제 접근이 업계 표준에 부합함을 입증.

## 5. AWS Nitro System 보안 백서

### 5-A. Nitro Hypervisor의 의도적 최소화

**인용 위치**: 비교표 §2

**영어 원문**:
> "**Within the Nitro Hypervisor, there is, by design, no networking stack, no general-purpose file system implementations, and no peripheral device driver support.** The Nitro Hypervisor has been designed to include only those services and features which are strictly necessary for its task; it is not a general-purpose system and includes neither a shell nor any type of i[nteractive access]."

**한국어 요약**: AWS Nitro Hypervisor는 설계상 네트워킹 스택, 일반 파일시스템, 주변장치 드라이버를 일절 포함하지 않으며, **shell이나 대화형 접근 수단도 없다.** 이는 의도적 최소화로 attack surface를 극단적으로 축소한 결과이다.

**출처**: https://docs.aws.amazon.com/whitepapers/latest/security-design-of-aws-nitro-system/the-components-of-the-nitro-system.html
**문서 발행 주체**: Amazon Web Services (AWS 공식 백서)

**우리 대응에서의 의미**: AWS는 호스트에 백신을 설치할 수 없는 구조를 의도적으로 설계함. 이는 글로벌 No.1 CSP가 **"백신은 보호의 본질이 아니다"** 라는 철학을 보안 아키텍처로 구현한 사례.

### 5-B. Nitro System은 KVM 기반

**인용 위치**: 비교표 §2

**영어 원문**:
> "Starting with the C5 instance type introduced in 2017, the Nitro System has entirely eliminated the need for Dom0 on an EC2 instance. Instead, **a custom-developed, minimized hypervisor based on KVM** provides a lightweight VMM, while offloading other functions such as those previously performed by the device-models in Dom0 into a set of discrete Nitro Cards."

**한국어 요약**: AWS는 2017년 C5 인스턴스부터 Xen Dom0를 제거하고 **KVM 기반의 자체 개발 경량화 하이퍼바이저(Nitro)** 로 전환했다.

**우리 대응에서의 의미**: SCPv2와 AWS Nitro는 **동일한 KVM 가상화 코어 기술**을 사용한다. 따라서 KVM의 표준 보안 메커니즘(sVirt, AppArmor)을 활용하는 것이 글로벌 베스트 프랙티스.

## 9. Red Hat Enterprise Linux 가상화 보안 가이드

### 9-A. sVirt와 KVM 보안 모델

**인용 위치**: 보완통제 가이드 §1 일반 배경

**영어 원문**:
> "Since virtual machines under KVM are implemented as Linux processes, KVM uses the standard Linux security model to provide isolation and resource controls. The Linux kernel includes Security-Enhanced Linux (SELinux) to add mandatory access control (MAC), multi-level security (MLS) and multi-category security (MCS) through a flexible and customizable security policy. SELinux provides strict resource isolation and confinement for processes running on top of the Linux kernel, including virtual machine processes. The sVirt project builds upon SELinux to further enable virtual machine isolation and controlled sharing."

**한국어 요약**: KVM 하의 VM은 Linux 프로세스로 구현되므로 표준 Linux 보안 모델(SELinux MAC/MLS/MCS)이 그대로 적용된다. sVirt는 SELinux 위에 가상화 특화 격리를 추가한다.

**출처**: https://docs.redhat.com/en/documentation/red_hat_enterprise_linux/7/html/virtualization_security_guide/chap-virtualization_security_guide-svirt
**문서 발행 주체**: Red Hat, Inc.

## 10. ERNW Hardening KVM 가이드 (참조)

### 10-A. sVirt는 KVM 보안의 필수

**인용 위치**: 보완통제 가이드 일반

**영어 원문**:
> "**The sVirt framework must be active and used by KVM** to isolate the processes belonging to the different virtual machines from each other and make the exploitation of virtual (QEMU) devices (significantly) harder. sVirt requires either SELinux or AppArmor, and its active use can be verified by inspecting the Linux process list."

**한국어 요약**: sVirt 프레임워크는 KVM에서 **반드시(must) 활성화·사용되어야** 하며, VM 간 프로세스 격리 및 QEMU 디바이스 취약점 악용 차단의 핵심이다. sVirt는 SELinux 또는 AppArmor를 필요로 한다.
