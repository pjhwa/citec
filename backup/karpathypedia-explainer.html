구현 코드

import OpenAI from 'openai'; const client = new OpenAI({ baseURL: process.env.OPENAI_BASE_URL, apiKey: process.env.OPENAI_API_KEY, }); async function extractIdeaBlocks( chunk: string, source: string ): Promise<IdeaBlock[]> { const res = await client.chat.completions.create({ model: process.env.OPENAI_MODEL, messages: [ { role: 'system', content: EXTRACT_SYSTEM }, { role: 'user', content: chunk } ], response_format: { type: 'json_object' }, temperature: 0.1, max_tokens: 4096, }); const data = JSON.parse(res.choices[0].message.content!); return data.ideablocks.map(b => ({ ...b, id: `ib_${sha256(b.question)}`, source, })); }

128K 컨텍스트 → 50+ 블록을 한 번에 병합 호출 가능 (원본 15개 제한 대비 3배+)

순수 JS Jaccard 구현

function jaccard( a: Set<string>, b: Set<string> ): number { const inter = new Set( [...a].filter(x => b.has(x)) ); const union = new Set([...a, ...b]); return union.size === 0 ? 0 : inter.size / union.size; } function blockSim( a: IdeaBlock, b: IdeaBlock ): number { const kw = jaccard( new Set(a.keywords.map(k => k.toLowerCase())), new Set(b.keywords.map(k => k.toLowerCase())) ); const ent = jaccard( new Set(a.entities.map(e => e.toLowerCase())), new Set(b.entities.map(e => e.toLowerCase())) ); const tag = jaccard( new Set(a.tags), new Set(b.tags) ); return 0.5*kw + 0.3*ent + 0.2*tag; }

LLM 병합 (클러스터 발견 후)

async function mergeCluster( cluster: IdeaBlock[] ): Promise<IdeaBlock> { // 128K 컨텍스트: 50+ 블록도 한 번에 처리 const res = await client.chat.completions.create({ model: process.env.OPENAI_MODEL, messages: [ { role: 'system', content: MERGE_SYSTEM }, { role: 'user', content: JSON.stringify(cluster, null, 2) } ], response_format: { type: 'json_object' }, temperature: 0.1, }); const merged = JSON.parse(res.choices[0].message.content!); return { ...merged, id: `ib_${sha256(merged.question)}`, source_ids: cluster.map(b => b.id), }; }

// Jaccard보다 의미론적 유사도가 더 정확할 때 const emb = await client.embeddings.create({ model: 'text-embedding-ada-002', // 또는 로컬 임베딩 input: blocks.map(b => b.question + ' ' + b.answer), }); function cosineSim(a: number[], b: number[]): number { const dot = a.reduce((s, ai, i) => s + ai * b[i], 0); const magA = Math.sqrt(a.reduce((s,ai) => s+ai*ai, 0)); const magB = Math.sqrt(b.reduce((s,bi) => s+bi*bi, 0)); return dot / (magA * magB); }

프롬프트 설계 가이드 신규

GPT-OSS-128K 자체 구현의 핵심은 프롬프트 품질이다. 각 단계별 검증된 프롬프트를 제공한다.

Stage 1 System Prompt — IdeaBlock 추출
You are a knowledge extraction expert. Given a text passage, extract structured Q&A pairs called IdeaBlocks. Each IdeaBlock captures ONE atomic fact or concept from the text. Return ONLY valid JSON in this EXACT format: { "ideablocks": [ { "question": "Specific, answerable question based on the text", "answer": "Concise factual answer (1-4 sentences max)", "entities": ["ProperNoun1", "SystemName2"], "keywords": ["technical_term1", "domain_term2"], "tags": ["topic_category1", "topic_category2"] } ] } RULES: - Each IdeaBlock = ONE atomic concept (not summaries of multiple ideas) - Questions must be specific enough to have a single clear answer - Entities: only proper nouns (systems, tools, names, organizations) - Keywords: technical/domain terms (lowercase, normalized) - Tags: 2-4 high-level categories per block - Generate 3-8 IdeaBlocks per 2,000 characters of input - DO NOT hallucinate — only extract what is explicitly in the text - If text is in Korean, questions and answers should also be in Korean
핵심 포인트
• response_format: {type: 'json_object'} 필수
• temperature 0.1 (결정론적 추출)
• "ONE atomic concept" 강조 → 과세분화 방지
• Korean 지원 명시 (한국어 문서 처리)
품질 검증 방법
• 질문이 답에서 답을 찾을 수 있는가?
• 엔티티가 실제 고유명사인가?
• 키워드가 도메인 특화인가?
• 블록이 원본에 없는 정보를 생성했나?
Stage 2 System Prompt — 블록 병합 (Distill)
You are a knowledge consolidation expert. You will receive a cluster of similar IdeaBlocks that discuss overlapping concepts. Merge them into ONE comprehensive IdeaBlock that captures all unique information. Return ONLY valid JSON: { "question": "A comprehensive question covering the main concept across all source blocks", "answer": "A thorough answer combining all unique information from source blocks (2-6 sentences)", "entities": ["all unique entities from all source blocks"], "keywords": ["all unique keywords from all source blocks"], "tags": ["unified tags representing the merged concept"], "source_ids": ["source block IDs that were merged"] } RULES: - The merged question must be more general than any single source question - The merged answer must contain ALL unique facts from ALL source blocks - Do NOT lose information — every unique fact must be preserved - Remove redundant information but keep all distinct details - Entities and keywords should be the UNION of all source blocks' fields - If source blocks are in Korean, keep all output in Korean
128K 컨텍스트 활용: 클러스터 크기 제한 없음. 50+ 블록도 한 번에 전송 가능. 원본 Blockify distill API의 15블록 제한 초과
Stage 4 System Prompt — 위키 아티클 컴파일
You are a technical encyclopedia writer. You will receive a collection of IdeaBlocks (Q&A pairs) on a related topic cluster, and a list of related article topics for cross-referencing. Write a comprehensive Wikipedia-style article using ONLY the information provided in the IdeaBlocks. Output format — Markdown with YAML frontmatter: --- title: "Article Title" summary: "One-sentence summary of the article's main topic" category: "Category name" tags: [tag1, tag2, tag3] related: [related-article-slug-1, related-article-slug-2] ideablock_ids: [ib_xxx, ib_yyy] --- # Article Title ## Overview [2-3 paragraph introduction] ## [Section derived from content] [Content organized from IdeaBlocks] ## [Additional sections as needed] ## Related Topics - [[Related Article 1]] - [[Related Article 2]] RULES: - Encyclopedic, neutral, third-person tone - ONLY use information from the provided IdeaBlocks — no hallucination - Every major claim should be traceable to a specific IdeaBlock - Use [[Double Bracket]] syntax for internal wiki links - Organize content logically with appropriate section headers - The summary field must be a single sentence - If IdeaBlocks are in Korean, write the article in Korean - ideablock_ids must list ALL source block IDs used
128K 컴파일 전략
• 25블록 제한 제거 → 100+ 블록 처리
• 인접 커뮤니티 정보도 컨텍스트에 포함
• 여러 커뮤니티 동시 처리로 일관성 향상
품질 튜닝
• temperature 0.3 (창의적이지만 사실 기반)
• max_tokens 8192 (충분한 아티클 길이)
• 검증: ideablock_ids가 전부 포함됐나?

1. 전체 문서 컨텍스트 활용 (128K 핵심 이점)

문서가 50K 토큰 미만이면 청킹 없이 전체를 한 번에 처리하라. 청크 경계에서 잘리는 개념이 사라지지 않는다.

const DOC_TOKEN_ESTIMATE = text.length / 4; if (DOC_TOKEN_ESTIMATE < 50000) { // 청킹 없이 전체 처리 return await extractIdeaBlocks(text, source); } else { // 섹션 단위 분할 (헤더 기준) const chunks = splitByHeaders(text, maxChunkSize: 40000); return (await Promise.all(chunks.map(c => extractIdeaBlocks(c, source)))).flat(); }

2. JSON mode 필수 — 구조화 출력 강제

IdeaBlock 추출 품질의 90%는 JSON mode 사용 여부에 달려있다. 미사용 시 파싱 실패 위험이 크다.

// 필수: response_format 지정 response_format: { type: 'json_object' } // 또는 structured output (지원 모델) response_format: { type: 'json_schema', json_schema: { name: 'ideablock_extraction', schema: IdeaBlockSchema // Zod/JSON Schema } }

3. Distill: Jaccard 먼저, LLM 나중

Jaccard 계산은 API 비용 없음. 0.55 임계값으로 후보 클러스터만 찾은 뒤 LLM 병합 호출. 전체 비용을 40~60% 절감한다.

// 비용 효율적 Distill 전략 1. blockSim() 계산 → O(n²) but API 비용 없음 2. BFS → 클러스터 그룹만 추출 3. 클러스터만 → LLM merge 호출 (비용 발생) 4. 단독 블록 → 그대로 사용 (LLM 호출 없음)
