● 다각도 분석: Fabrix 출력 토큰 제한으로 인한 응답 잘림

  ---
  현황 파악

  현재 코드(_company_stream)는 Fabrix에 단일 요청을 보내고 SSE 스트림을 그대로 받습니다. Fabrix가 max output 한도에 도달하면
  스트림이 그냥 끊기고, 클라이언트는 done 이벤트를 받아 완료로 처리합니다. 잘렸는지 정상 종료인지 구분 불가인 상태입니다.

  ---
  방안 A — 출력 토큰 절약 (프롬프트 최적화)

  아이디어: 입력을 줄여 출력 예산을 늘린다.

  ┌────────────────────────────────────────────────────┬──────────────┬──────────────────────────────┐
  │                     세부 방법                      │     효과     │             한계             │
  ├────────────────────────────────────────────────────┼──────────────┼──────────────────────────────┤
  │ 시스템 프롬프트 압축                               │ 출력 예산 ↑  │ 규칙이 많아 대폭 축소 어려움 │
  ├────────────────────────────────────────────────────┼──────────────┼──────────────────────────────┤
  │ 컨텍스트 청크 수 축소 (현재 5개 → 3개)             │ 입력 토큰 ↓↓ │ 관련 문서 누락 가능          │
  ├────────────────────────────────────────────────────┼──────────────┼──────────────────────────────┤
  │ 응답 형식을 간결하게 지시 ("불필요한 도입부 생략") │ 출력 밀도 ↑  │ LLM이 항상 따르지 않음       │
  └────────────────────────────────────────────────────┴──────────────┴──────────────────────────────┘

  평가: 가장 쉽게 적용 가능하나 근본적 해결책이 아님. 보조 수단으로 적합.

  ---
  방안 B — 완료 마커 + 잘림 감지

  아이디어: LLM에게 응답이 끝날 때 특정 마커(<!-- CITEC_END -->)를 출력하도록 시스템 프롬프트에 지시. 스트림 종료 시 마커
  유무로 잘림 여부를 판단.

  흐름:
  LLM 응답 → <!-- CITEC_END --> 있음 → 정상 완료
  LLM 응답 → 마커 없이 스트림 종료 → 잘림 감지 → 사용자에게 경고 표시

  장점: 구현 간단, 잘림 여부를 명확히 알 수 있음
  한계: 잘림을 감지할 뿐 해결하지는 않음. 이후 방안과 조합 필요.

  ---
  방안 C — 자동 이어쓰기 (Multi-turn Continuation)

  아이디어: 잘림이 감지되면 "이전 응답에 이어서 계속 작성해줘" 요청을 자동으로 발송하고 응답을 이어붙임.

  1회 요청 → 응답 A (잘림 감지)
  → 2회 요청: "아래 내용에 이어서 계속 작성하시오:\n[응답 A의 마지막 500토큰]"
  → 응답 B (완료 또는 재잘림)
  → 필요 시 3회 반복

  장점: 완전한 응답 자동 완성, 사용자 개입 불필요
  고려사항:
  - 이어붙임 시 중복·어색한 연결 발생 가능
  - Fabrix 입력 토큰도 소모 (이전 응답 일부를 컨텍스트로 재전송)
  - 무한 루프 방지를 위한 최대 회차 제한 필요 (예: 최대 3회)
  - API 비용·속도 증가

  ---
  방안 D — 사용자 주도 이어쓰기 (수동 트리거)

  아이디어: 잘림이 감지되면 프론트엔드에 "이어서 생성" 버튼을 노출. 사용자가 클릭 시 C 방식의 continuation 요청 발송.

  결과 페이지 하단: ⚠ 응답이 잘렸습니다. [↪ 이어서 생성]

  장점: C보다 구현 단순, 사용자가 필요 시에만 추가 생성 (불필요한 API 호출 방지)
  한계: 사용자 개입 필요, UX 불편

  ---
  방안 E — 섹션별 분할 생성

  아이디어: 단일 거대 응답 대신, 먼저 목차를 생성하고 각 섹션을 별도 요청으로 생성·병합.

  1단계: "목차(outline)만 생성" → [1.개요, 2.원인, 3.점검절차, 4.예방]
  2단계: 각 섹션별 별도 요청 → 병렬 또는 순차 실행
  3단계: 프론트에서 병합하여 표시

  장점: 섹션별로 토큰 예산을 확실히 통제, 잘림 근본 해결
  한계: 구현 복잡도 가장 높음, 응답 시간 증가, 섹션 간 문체 불일치 가능

  ---
  방안 F — 응답 길이 스스로 조절하도록 지시

  아이디어: 시스템 프롬프트에 "응답이 너무 길어질 것 같으면 각 섹션을 요약해서라도 완결성을 유지하라"고 지시.

  규칙 추가: "전체 응답이 길어질 경우, 개별 섹션의 상세도를 줄이더라도
  반드시 모든 섹션을 포함한 완전한 답변을 제공하십시오.
  표·코드블록은 핵심만 남기고, 설명은 간결하게 작성하십시오."

  장점: API 호출 1회로 해결, 구현 없음
  한계: LLM이 항상 따르지 않음, 내용 품질 저하 가능

  ---
  종합 권장안

  ┌──────────┬──────────────────────────────┬───────────────────────────────────────────────────┐
  │ 우선순위 │             방안             │                       이유                        │
  ├──────────┼──────────────────────────────┼───────────────────────────────────────────────────┤
  │ 1순위    │ B (마커) + D (수동 이어쓰기) │ 구현 간단, 즉시 효과, 사용자가 필요시만 추가 호출 │
  ├──────────┼──────────────────────────────┼───────────────────────────────────────────────────┤
  │ 2순위    │ F (자기 조절 지시)           │ 프롬프트만 수정, 비용 없음, 1순위와 병행          │
  ├──────────┼──────────────────────────────┼───────────────────────────────────────────────────┤
  │ 3순위    │ C (자동 이어쓰기)            │ B+D 검증 후 자동화로 업그레이드                   │
  ├──────────┼──────────────────────────────┼───────────────────────────────────────────────────┤
  │ 보류     │ E (섹션 분할)                │ 효과는 확실하지만 구조 변경 크고 복잡             │
  └──────────┴──────────────────────────────┴───────────────────────────────────────────────────┘

  핵심 판단: 현실적으로 가장 빠르고 안정적인 방향은 B+D 조합입니다. 잘림을 감지해 사용자에게 알리고, 요청 시 이어쓰기를
  실행하는 방식은 Fabrix의 실제 한도가 환경마다 다르다는 불확실성에도 유연하게 대응할 수 있습니다.

● B+C 조합 방안 심층 분석

  ---
  핵심 아이디어

  B(마커 감지)가 "언제 이어써야 하는지" 를 결정하고, C(자동 이어쓰기)가 "어떻게 이어쓰는지" 를 실행합니다. B 없는 C는 매번
  불필요하게 continuation을 보내고, C 없는 B는 감지만 할 뿐 해결하지 못합니다.

  ---
  전체 흐름

  사용자 질문
      │
      ▼
  [1차 요청] system_prompt + query → Fabrix SSE 스트림
      │
      ├─ 스트림 중 토큰 누적 (answer_text)
      │
      ├─ <!-- CITEC_END --> 발견됨
      │       └─ 정상 완료 → done 이벤트 전송
      │
      └─ 마커 없이 스트림 종료 (잘림 감지)
              │
              ▼
          [2차 요청] continuation_prompt → Fabrix SSE
              │
              ├─ 마커 발견 → 정상 완료
              └─ 또 잘림 → [3차 요청] (최대 N회)
                              │
                              └─ N회 초과 → 경고 표시 후 done

  ---
  구성 요소별 상세 분석

  1. 완료 마커 설계

  시스템 프롬프트 끝에 아래 지시를 추가합니다.

  응답이 완전히 완료되었을 때, 마지막 줄에 반드시 아래 마커를 단독으로 출력하십시오:
  <!-- CITEC_END -->
  이 마커가 없으면 응답이 불완전한 것으로 간주됩니다.

  마커 설계 시 고려사항:

  ┌───────────┬────────────────────────────────────────────────────────────────────────┐
  │   항목    │                                  내용                                  │
  ├───────────┼────────────────────────────────────────────────────────────────────────┤
  │ 형식      │ HTML 주석 <!-- CITEC_END --> — 렌더링 시 화면에 표시되지 않음          │
  ├───────────┼────────────────────────────────────────────────────────────────────────┤
  │ 오탐 방지 │ 마커가 단독 줄에 있을 때만 유효로 인정 (본문 중간 발생 무시)           │
  ├───────────┼────────────────────────────────────────────────────────────────────────┤
  │ LLM 망각  │ LLM이 마커를 빠뜨릴 수 있으므로, "마커 없음 = 잘림"을 기본 전제로 설계 │
  └───────────┴────────────────────────────────────────────────────────────────────────┘

  ---
  2. 잘림 감지 로직 (백엔드)

  스트림이 종료된 시점에 마커 존재 여부로 판단합니다.

  # 개념 코드
  COMPLETION_MARKER = "<!-- CITEC_END -->"

  marker_found = COMPLETION_MARKER in answer_text
  if marker_found:
      answer_text = answer_text.replace(COMPLETION_MARKER, "").rstrip()
      # 정상 완료 처리
  else:
      # 잘림 → continuation 시작

  ---
  3. Continuation 프롬프트 설계 (핵심 난관)

  이어쓰기 요청에는 얼마나 많은 이전 응답을 컨텍스트로 넘기느냐가 핵심입니다.

  옵션 비교:

  ┌─────────────────────┬───────────┬─────────────────┬────────────────────────────┐
  │        방식         │ 입력 토큰 │ 연결 자연스러움 │            위험            │
  ├─────────────────────┼───────────┼─────────────────┼────────────────────────────┤
  │ 이전 응답 전체 전달 │ 매우 많음 │ 매우 좋음       │ Fabrix 입력 한도 초과 가능 │
  ├─────────────────────┼───────────┼─────────────────┼────────────────────────────┤
  │ 마지막 N토큰만 전달 │ 적음      │ 보통            │ 맥락 부족으로 어색한 이음  │
  ├─────────────────────┼───────────┼─────────────────┼────────────────────────────┤
  │ 섹션 헤더 목록 전달 │ 적음      │ 보통            │ LLM이 자체 판단해야 함     │
  └─────────────────────┴───────────┴─────────────────┴────────────────────────────┘

  권장 방식: 마지막 1,500~2,000 토큰(약 1,000~1,300 한국어 단어)만 전달. Fabrix 입력 한도를 초과하지 않으면서 충분한 맥락
  제공.

  CONTEXT_WINDOW = 2000  # 토큰 단위 근사값 (문자 기준 약 3,000자)

  tail_context = answer_text[-CONTEXT_WINDOW:]

  continuation_user_msg = f"""이전 응답이 중간에 끊겼습니다.
  아래 내용에 이어서 자연스럽게 계속 작성하십시오.
  이전 내용을 반복하지 말고, 끊긴 지점부터 바로 이어서 작성하십시오.

  [이전 응답 끝부분]
  {tail_context}"""

  ---
  4. 반복 구조와 종료 조건

  최대 반복 횟수: 3회 (1차 요청 + 최대 2회 continuation)

  종료 조건:
    1. CITEC_END 마커 발견 → 정상 종료
    2. 최대 반복 횟수 초과 → 경고 플래그와 함께 종료
    3. 응답이 빈 문자열 → 비정상, 즉시 종료

  3회를 넘어가는 경우는 응답이 128k 출력 한도를 3번 이상 채울 만큼 긴 것인데, 현실적으로 드물고 이 경우는 사용자에게 "응답이
  매우 길어 일부만 표시됩니다" 경고를 표시하는 것이 적절합니다.

  ---
  5. SSE 스트림 연속성 (프론트엔드 관점)

  사용자 입장에서는 하나의 긴 스트림처럼 보여야 합니다.

  [현재 구조]
  1차 스트림 token... → done

  [B+C 구조]
  1차 스트림 token...
    → status: "이어서 생성 중... (2/3)"
  2차 스트림 token... (answerBuffer에 append)
    → status: "이어서 생성 중... (3/3)"  ← 필요 시
  3차 스트림 token...
    → done (최종 완료)

  주의: done 이벤트를 첫 번째 스트림 종료 시 보내지 않아야 합니다. 모든 continuation이 완료된 후에만 done을 보냅니다. 이를
  위해 백엔드 제너레이터(generator) 함수의 구조를 수정해야 합니다.

  ---
  6. 마커의 프론트엔드 처리

  마커가 스트림 토큰 중간에 끊겨서 올 수 있습니다 (예: <!- / - CITEC / _END -->처럼 분할 도착). 따라서:

  - 백엔드에서 누적 후 마커를 제거하고 최종 텍스트만 프론트로 전송하는 것이 가장 안전
  - 또는 프론트에서도 answerBuffer에서 마커 패턴 제거 (정규식 사용)

  합성 문서 저장 처리

  continuation이 끝난 후 저장되는 합성 문서(synthesis/items/*.md)에는:
  - 마커 <!-- CITEC_END --> 제거
  - 잘림이 발생했던 경우 frontmatter에 truncated: true 또는 continuation_count: 2 기록 (운영 추적용)
  - 정상 완료 시 기존과 동일하게 저장

  ---
  결론

  B+C 조합은 구현 복잡도 대비 효과가 높습니다. 핵심은 백엔드 제너레이터를 루프 구조로 전환하는 것이고, 프론트엔드는 기존 SSE
  처리 구조를 거의 그대로 유지할 수 있습니다. 가장 까다로운 부분은 continuation 프롬프트의 tail 길이 튜닝과 이음새 품질인데,
  이는 실제 Fabrix 환경에서 테스트하며 조정이 필요합니다.

● 문장 중간 잘림 처리 가능성 분석

  ---
  결론부터

  가능합니다. 오히려 문장 중간 잘림은 LLM이 가장 잘 처리하는 케이스입니다. LLM의 근본 동작이 "다음 토큰 예측"이기 때문에,
  메모리 증이라는 불완전한 끝에서 설이 필요하다를 자연스럽게 이어쓰는 것은 LLM에게 매우 쉬운 작업입니다.

  단순 문자열 연결만으로 완성됩니다.

  ---
  진짜 위험 — 오버랩(중복) 문제

  LLM이 continuation을 시작할 때 부분 단어를 처음부터 다시 출력하는 경우가 있습니다.

  [예상] "설이 필요하다."          → 병합: "메모리 증설이 필요하다." ✓
  [오버랩] "메모리 증설이 필요하다." → 병합: "메모리 증메모리 증설이 필요하다." ✗
  [부분오버랩] "증설이 필요하다."   → 병합: "메모리 증증설이 필요하다." ✗

  이것이 실질적인 핵심 문제입니다. 세 가지 해결 방법이 있습니다.

  ---
  오버랩 해결 방법

  방법 1 — 후위/전위 매칭으로 중복 제거 (코드 레벨)

  백엔드에서 병합 전에 중복을 감지하고 제거합니다.

  def merge_continuation(original: str, continuation: str, check_len: int = 80) -> str:
      """
      original의 끝부분과 continuation의 시작부분이 겹치는 구간을 탐지해 제거.
      예: original="메모리 증", continuation="증설이 필요하다"
          → overlap="증" 감지 → "메모리 증" + "설이 필요하다"
      """
      tail = original[-check_len:]

      # tail의 접미사(suffix)가 continuation의 접두사(prefix)와 일치하는 최장 구간 탐색
      for length in range(min(len(tail), len(continuation)), 0, -1):
          if tail.endswith(continuation[:length]):
              return original + continuation[length:]

      return original + continuation  # 겹치는 부분 없음

  원본 끝:  "...메모리 증"
  연속 시작: "설이 필요하다"  → 겹침 없음 → 그냥 연결
  원본 끝:  "...메모리 증"
  연속 시작: "증설이 필요하다" → "증" 겹침 감지 → "설이 필요하다" 부터 연결
  원본 끝:  "...메모리 증"
  연속 시작: "메모리 증설..."  → "메모리 증" 겹침 감지 → "설..." 부터 연결

  방법 2 — 프롬프트로 LLM에게 지시

  이전 응답이 중간에 잘렸습니다.
  아래는 이전 응답의 마지막 부분입니다:
  ---
  ...서버의 부하가 급격히 증가하면 메모리 증
  ---
  "메모리 증" 이후 부터만 이어서 출력하십시오.
  이전 내용을 절대 반복하지 마십시오.

  LLM이 지시를 따르면 설이 필요하다.부터 시작합니다. 하지만 LLM이 항상 따른다는 보장이 없으므로 방법 1과 병행하는 것이
  안전합니다.

  방법 3 — 병합 후 렌더링 레벨 스무딩

  위 두 방법으로도 어색한 이음새가 남을 경우, 최종 저장 전에 "이음새 정리" LLM 호출을 한 번 더 하는 방법입니다. 단, API 비용과
   시간이 추가되므로 최후의 수단으로 고려합니다.

  실용적 결론

  단어·문장 중간 잘림은 B+C 조합으로 충분히 처리 가능하고, 오버랩 제거 로직(방법 1)을 추가하면 사용자가 잘림을 인지하지 못할
  수준으로 매끄럽게 이어집니다.

  코드블록·테이블 내부 잘림은 추가적인 상태 감지 로직이 필요하지만, 구현 가능한 범위입니다. 가장 간단한 접근은 continuation
  프롬프트에 현재 열려 있는 마크다운 구조(펜스, 테이블)를 명시하는 것입니다.
