상황을 정리하면 팀장이 던진 질문들이 **이미 확정된 사실**과 **아직 가정인 것**이 섞여 있습니다. 좋은 진단은 이 둘을 먼저 분리한 뒤, 가정을 하나씩 검증하는 순서로 가야 합니다. 특히 팀장 발언 중 "NW가 다 했어야지", "NW가 알았을 건데"는 결론이 아니라 **진단으로 확인해야 할 가설**입니다. 이걸 사실처럼 깔고 시작하면 진단이 아니라 책임전가 문서가 됩니다.

진단 항목을 쟁점별로 나누면 다음과 같습니다.

**1. 인수인계(지식이전) 적정성**
핵심은 "reconcile 메커니즘"과 "K8s로 생성한 LB 인증서는 K8s에서 갱신해야 한다(SCP 콘솔 단독 변경은 reconcile 시 덮어쓰기됨)"는 두 가지 운영 지식이 고객/운영자에게 전달됐는가입니다.
- 확인할 것: 인수인계 문서·운영 매뉴얼·교육 이력에 해당 내용이 있는가, "권장 SSL 갱신 Flow (1)~(2)"가 고객에게 실제 공유됐는가(문서엔 권장 flow가 있다고 쓰여 있음 → 그렇다면 공유 증빙이 있어야 함).
- "구두로 설명했다"는 입증 불가하면 적정 인수인계로 인정받기 어렵고, "운영자는 다 안다"는 가정입니다. K8s 프로비저닝 LB의 이중 인증서 구조와 reconcile 덮어쓰기는 일반 운영 상식이 아니라 이 CSP/구성 특유의 함정이라, "다 안다" 주장은 진단으로 깨질 가능성이 큽니다.

**2. SSL 작업 R&R 경계** (팀장의 "NW가 다 했어야지"에 대한 진단)
- 확인할 것: 1/30 SR의 **작업 범위(scope)**가 "인증서 갱신 전체"였는지 "SCP 콘솔 갱신"이었는지. 계약/운영 R&R 상 K8s LB 인증서 갱신 주체가 SDS인지 고객인지(문서엔 K8s 운영=고객 책임으로 기술됨).
- 즉 NW가 K8s까지 손댔어야 했다는 전제 자체가 R&R상 맞는지부터 검증해야 합니다. 만약 K8s가 고객 영역이라면 NW가 "안 한 게" 아니라 "할 영역이 아니었던" 것일 수 있고, 그렇다면 책임은 경계를 명확히 안 한 쪽(설계/인수인계)으로 이동합니다.

**3. 표준 절차(runbook) 존재·이행 여부**
- 확인할 것: 인증서 갱신 표준 절차에 K8s 갱신 단계가 포함돼 있었는가. Double-Check 프로세스가 왜 없었는가.
- 개선사항에 "Double-Check 신규 적용", "툴팁 추가"가 들어간 건 곧 **기존엔 절차가 없었다**는 방증입니다. 절차 부재면 개인(NW) 과실보다 프로세스 결함 쪽 진단이 맞습니다.

**4. 만료 모니터링·장애 인지 체계** (팀장의 "알았을 건데"에 대한 진단)
- 확인할 것: 인증서 만료 임박/만료에 대한 모니터링·알람이 있었는가. K8s에 만료 인증서가 남아있던 상태(1/30~4/30)를 누가 인지 가능했는가. NW가 실제로 인지했는지 로그·정황으로 검증.
- "알았을 것"은 추정이므로, 인지 가능성 자체가 없었다면(모니터링 부재) 신고 누락이 아니라 탐지 체계 부재가 됩니다.

**5. 신고·에스컬레이션·고객통보 프로세스**
- 확인할 것: 장애 인지 시점→내부 신고→고객 통보까지 타임라인. 신고/통보 기준이 정의돼 있었는가. 누락이 "몰라서"인지 "알고도 안 해서"인지.

# 모니모 LB 인증서 만료 장애 — 진단 계획서

| 구분 | 내용 |
|---|---|
| 문서명 | 모니모 시스템 LB 인증서 만료 장애 진단 계획서 |
| 대상 장애 | 2026.4.30 ~ 5.1 모니모 원앱 LB SSL 인증서 만료 장애 |
| 작성 목적 | 사실관계 및 R&R 경계 규명, 재발방지 근거 확보 |
| 작성일 | 2026. __. __ |

---

## 1. 배경 및 목적

2026년 4월 30일 및 5월 1일, 모니모 원앱 시스템의 LB(Load Balancer)에 만료된 SSL 인증서가 동기화되어 관계사 간 서비스 호출 오류가 발생하였다. 직접 원인은 K8s에 만료 인증서가 잔존한 상태에서 디스크 Fault에 의한 K8s Reconcile이 수행되며 만료 인증서가 LB로 덮어쓰기된 것으로 확인되었다.

본 진단의 목적은 **책임 주체를 단정하는 것이 아니라**, 장애에 이르게 된 사실관계와 운영 책임 경계(R&R)를 증빙 기반으로 규명하고, 이를 토대로 실효성 있는 재발방지 대책을 도출하는 데 있다.

---

## 2. 진단 원칙

진단의 객관성과 신뢰성 확보를 위해 다음 원칙을 적용한다.

- **확정 사실과 가설을 분리한다.** 타임라인·로그로 입증된 사실과, 추정에 기반한 가설을 구분하여 기술한다.
- **제기된 지적사항도 검증 대상 가설로 취급한다.** "NW 부서가 인증서 교체를 모두 수행했어야 한다", "NW 부서가 장애를 인지했을 것이다" 등 현재 제기된 의견은 결론이 아닌, 진단을 통해 참·거짓을 확인할 가설로 다룬다.
- **모든 판단은 증빙에 근거한다.** SR 접수내역, 인수인계 문서, 운영 매뉴얼, 모니터링·알람 이력, 작업·동기화 로그 등 객관적 증빙으로 뒷받침한다.

---

## 3. 확정 사실 vs 검증 필요 가설

| 구분 | 내용 |
|---|---|
| **확정 사실** | • 1/30 SCP 콘솔에서 LB 인증서 갱신 완료 (2/10 만료 사전 교체) |
| | • K8s에 저장된 인증서는 갱신되지 않아 만료 인증서가 잔존 |
| | • 4/30 디스크 Fault → K8s Reconcile 수행 → 만료 인증서가 LB로 동기화되어 장애 발생 |
| | • 4/30 SCP 콘솔 재갱신으로 일시 정상화, 5/1 동일 현상 재발, K8s Service config 수정 후 정상화 |
| **검증 필요 가설** | ① reconcile 동작 및 K8s LB 인증서 갱신 방식이 고객/운영자에게 전달되었는가 |
| | ② K8s LB 인증서 갱신 주체가 R&R상 SDS인가, 고객인가 |
| | ③ 인증서 갱신에 K8s 단계를 포함한 표준 절차가 존재했는가 |
| | ④ 만료 인증서 잔존 상태를 인지할 수 있는 모니터링 체계가 있었는가 |
| | ⑤ 장애 인지 시 신고·고객통보 프로세스가 정의·이행되었는가 |

---

## 4. 진단 항목별 상세 계획

### 4-1. 인수인계(지식이전) 적정성

- **검증 가설**: reconcile 메커니즘, 그리고 "K8s로 생성된 LB의 인증서는 K8s에서 갱신해야 하며 SCP 콘솔 단독 변경은 reconcile 시 덮어쓰기된다"는 운영 지식이 고객/운영자에게 전달되었는가.
- **확보 증빙**: 핸드오버 문서, 운영 매뉴얼, 교육·전달 이력, 권장 SSL 갱신 Flow 공유 증빙.
- **유의점**: "구두로 설명했다"는 입증 불가 시 적정 인수인계로 인정받기 어렵다. "운영자는 기본적으로 다 안다"는 가정으로, K8s 프로비저닝 LB의 이중 인증서 구조 및 reconcile 덮어쓰기는 일반 운영 상식이 아니라 본 구성 특유의 함정에 해당하므로 별도 검증이 필요하다.

### 4-2. SSL 작업 R&R 경계

- **검증 가설**: 1/30 SR의 작업 범위가 "인증서 갱신 전체"였는가, "SCP 콘솔 갱신"에 한정되었는가. K8s LB 인증서 갱신 주체가 SDS인가, 고객인가.
- **확보 증빙**: 1/30 SR 접수내역(작업 범위 명시), 계약·운영 R&R 정의서, K8s 운영 책임 주체 규정.
- **유의점**: "NW 부서가 K8s까지 수행했어야 한다"는 전제의 성립 여부가 R&R에 종속된다. K8s가 고객 운영 영역이라면 "안 한 것"이 아니라 "수행 책임 범위가 아니었던 것"이며, 이 경우 쟁점은 경계 미명확(설계·인수인계)으로 이동한다.

### 4-3. 표준 절차(Runbook) 존재·이행 여부

- **검증 가설**: 인증서 갱신 표준 절차에 K8s 갱신 단계가 포함되어 있었는가. Double-Check 절차가 부재했던 사유는 무엇인가.
- **확보 증빙**: 기존 인증서 갱신 절차서, 변경관리 SR 처리 표준, 체크리스트.
- **유의점**: 개선사항으로 "Double-Check 신규 적용", "콘솔 툴팁 추가"가 제시된 점은 기존 절차 부재를 시사한다. 절차 부재가 확인되면 개인 과실보다 프로세스 결함 관점의 진단이 타당하다.

### 4-4. 만료 모니터링 및 장애 인지 체계

- **검증 가설**: 인증서 만료 임박·만료에 대한 모니터링·알람이 존재했는가. 1/30~4/30 기간 동안 K8s 만료 인증서 잔존을 누가 인지할 수 있었는가. NW 부서가 실제로 인지했는가.
- **확보 증빙**: 인증서 만료 모니터링 구성 여부, 알람 이력, 장애 인지 시점 로그·정황.
- **유의점**: "NW 부서가 알았을 것"은 추정이다. 인지 가능 체계 자체가 없었다면 이는 신고 누락이 아니라 탐지 체계 부재로 귀결된다.

### 4-5. 신고·에스컬레이션·고객통보 프로세스

- **검증 가설**: 장애 인지 → 내부 신고 → 고객 통보까지의 기준이 정의·이행되었는가. 통보 누락이 "인지 불가"에 기인하는가, "인지 후 미이행"에 기인하는가.
- **확보 증빙**: 장애 신고·에스컬레이션 기준, 고객 통보 SLA/프로세스, 실제 통보 타임라인.
