§ 01 — Executive Summary

핵심 결론

초기 가설(메모리 압박 → 회수 → CPU 포화)은 SAR로 확증됐고, "왜 여유 RAM에도 회수가 폭주했나"는 sosreport로 NUMA 노드 간 불균형임이 밝혀졌다.

  • 확증 1 — 회수 폭주. pgscank/s(kswapd 스캔) 0→195,000, pgsteal 동반 폭증, 캐시(kbcached) 149→121 GB 축출.
  • 확증 2 — 런큐 폭증. runq-sz 14→234, ldavg-1 8→131(48 CPU 대비 ~5×). pidstat %wait 폭증과 일치.
  • 확증 3 — 스왑 아님. pswpin/out ≈ 0, majflt ≈ 0, swappiness=1 → 순수 파일 캐시 회수.
  • 트리거 규명 — NUMA 불균형. Node 0의 free가 low 워터마크에 정확히 붙어 있고 Node 1은 16 GB+ 여유. 전체 free의 92%가 Node 1에 편중. pgscan_direct=0(부팅 이후 평생) = kswapd 전용 회수. 전역 고갈이 아닌 노드 로컬 부족이 원인.
  • 피드백. 느린 쿼리가 커넥션 점유 → plist-sz 1,177→1,283. 09:35 일시 회복 후 09:36 2차 피크, ~09:40 회수 종료.

§ 05 — Config Forensics (sosreport)

NUMA 구성 포렌식: 회수 폭주의 트리거

5/15 sosreport의 cmdline·sysctl·zoneinfo·vmstat를 분석했다. 설정·구조는 재부팅·변경이 없으면 5/26에도 동일하므로 트리거 판단의 토대가 된다. 후보였던 두 메커니즘은 설정상 꺼져 있었고, 대신 NUMA 노드 간 메모리 불균형이라는 더 날카로운 원인이 드러났다.

결정적 증거 — Node 0은 워터마크에 붙어 있고, Node 1은 텅 비어 있다 (proc/zoneinfo, Normal zone)

Node 0 · zone Normalfree 0.97 GB
low 0.97 = kswapd 가동선
min 0.88free = low → kswapd 회수 중high 1.06
Node 1 · zone Normalfree 16.27 GB
low 0.68
min 0.58free ≫ high (약 21배) → 여유high 0.77

단위 GB(4KB page 환산) · Node 0 막대는 그 노드의 워터마크 대역(min~high)을, Node 1 막대는 free가 high를 크게 초과함을 표시 · 시스템 전체 free의 약 92%가 Node 1에 편중.

이 그림이 5/26 SAR의 역설 — 전체 RAM의 60%가 가용인데 kswapd만(pgscand=0) 맹렬히 회수 — 을 설명한다. 회수가 전역 부족이 아니라 Node 0 로컬 부족 때문이었던 것. 아침 부하가 (CPU·백엔드가 몰린) Node 0에 익명 메모리를 쌓자 Node 0이 워터마크 아래로 떨어졌고 kswapd0이 그 노드의 캐시를 축출했다.

§ 07 — Ruled Out

배제된 원인 (증거 기반)

✗THP 컴팩션

cmdline transparent_hugepage=never, 모든 thp_* 카운터·AnonHugePages 0, compact_stall 0. THP 활동 전무.

✗NUMA zone-reclaim

vm.zone_reclaim_mode=0. (원인은 zone-reclaim 동작이 아니라 노드 간 메모리 불균형 자체)

✗전역 메모리 고갈 / overcommit

최악에도 ~113 GB 가용, Committed_AS 54 GB ≪ CommitLimit 162 GB. 절대량 부족 아님.

✗디스크 대역폭

스톨 피크 동안 tps·bwrtn 오히려 낮았고 대용량 쓰기는 09:34 회복기에 발생. 5/25엔 216 MB/s 쓰기도 무탈.

✗네트워크 / DRBD

ping >50ms는 09시에만 집중(catch-up=호스트 스톨). 스탠바이 001은 같은 시각 완전 평온.

✗하이퍼바이저 경합

게스트 %steal=0이 VMware Ready 0.78% / CSTP 0.25%(≈0)와 일치. 호스트는 vCPU 즉시 공급.

✗스왑 thrashing

pswpin/out≈0 · majflt≈0. 회수 대상은 스왑이 아닌 클린 파일 캐시.

§ 08 — Recommendations

권고 사항

P1 · NUMA

노드 간 메모리 불균형 해소 — 근본 해법VMware에서 이 48 vCPU 와이드 VM의 vNUMA가 물리 NUMA와 정렬됐는지, vCPU·메모리가 한 노드로 쏠리지 않는지 점검. OS/DB 차원에서 메모리를 두 노드에 분산(numactl --interleave 또는 Postgres NUMA 배치 검토). 현재 kernel.numa_balancing=1은 부하 중 페이지 마이그레이션 오버헤드도 유발하므로 동작 재검토.

OS

회수 증폭 요인 완화 (보조)Red Hat이 "여유 메모리에도 kswapd가 캐시를 공격적으로 회수" 증상에 문서화한 vm.watermark_boost_factor=0 검토. Node 0 여유 확보를 위해 vm.min_free_kbytes 소폭 상향도 완화책. 단 근본은 노드 균형이며 RAM 증설은 핵심 해법이 아님.

DB

메모리 폭증 쿼리 식별pg_stat_statements(또는 슬로우 쿼리 로그·Kibana)로 09:30 전후 대형 정렬/해시·계획 변화 추적. work_mem·max_connections 점검 및 커넥션 풀로 동시 작업메모리 상한 설정.

MON

에이전트 부하 조정metricbeat가 피크에 171% CPU(system 99%)로 장애 가중. process metricset 주기·범위 축소. SAR 1분 수집은 유효했으므로 유지.

CLUSTER

펜싱 안전여유 재검토약 5분 스톨이 corosync token timeout / SBD watchdog에 근접했는지 점검. 동일 스톨 재발 시 펜싱·스위치오버로 번지지 않도록 임계값 확보.
