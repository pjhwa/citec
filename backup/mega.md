먼저 이 기획안에서 심사에서 깨질 수 있는 약점부터 짚고, 그 위에 설득 논리를 채우겠습니다. 약점을 미리 막아둔 기획서가 더 설득력 높습니다.

## 심사 통과를 위해 먼저 고쳐야 할 4가지 (선보완)

**1. "원인 분석 3일 → 4시간(83%)" 지표는 그대로 쓰면 공격받습니다. (신뢰도: High)**
AI가 빠르게 하는 것은 *데이터 수집·요약·1차 이상징후 탐지*입니다. "원인 분석(RCA)"은 상관분석·재현·벤더 협의가 끼면 AI 단독으로 4시간에 끝나지 않습니다. 심사위원이 "AI가 환각으로 틀린 원인을 4시간 만에 자신있게 제시하면 더 위험한 것 아니냐"고 물으면 방어가 어렵습니다.
→ **지표를 분리하세요.** "초동 분석·요약 시간 3일→4시간(83% 단축)" + "RCA 초안 작성 자동화"로 표현하면 사실에 부합하고 더 안전합니다.

**2. DB 종수 불일치.** 본문에 나열된 건 Oracle, MySQL, MariaDB, PostgreSQL, EDB = **5종**인데 목표는 "6종"입니다. Db2 또는 Tibero/CUBRID 등을 추가하든지 5종으로 맞추든지 확정 필요. (신뢰도: High — 단순 카운트)

**3. "전체 데이터 분석"이라는 표현이 기술적으로 위험합니다. (신뢰도: High)**
sosreport는 수백 개 파일·수 GB이고, sar도 컨텍스트 윈도우에 원본 그대로 다 못 넣습니다. "전체 원본을 LLM에 투입"으로 읽히면 실현 불가능으로 평가됩니다.
→ "**전처리·정제 파이프라인으로 전체 데이터를 대상화**(선별·집계·RAG)한 뒤 AI 분석"으로 표현해야 합니다. 이미 3단계 pre-LLM 최적화 파이프라인 자산이 있으니 이걸 차별점으로 내세우는 게 유리합니다.

**4. 가장 큰 미기재 리스크 = 환각과 데이터 민감성.** 심사에서 반드시 나오는 질문입니다. 아래 본문에 "신뢰도 스코어 + Human-in-the-loop"와 "내부 FabriX 폐쇄망 처리"를 명시적으로 넣었습니다. CSAP 맥락상 데이터 주권 논리는 오히려 강한 가산점입니다.

# 장애 대응 Frontline Agent

## 선정 사유
- **장애 대응 시간의 80% 이상이 "데이터 수집·해석"에 소모**되고, 정작 판단은 마지막 단계다. AI가 가장 잘하는 영역(정형 데이터의 패턴 인식·요약)과 업무 병목이 정확히 일치한다.
- 분석 품질이 **소수 시니어 엔지니어 역량에 종속(속인화)**되어 있어, 인력 이탈·야간 대응 시 품질 편차가 크다.
- sar·sosreport·DB 진단 데이터는 **이미 표준화·정형화**되어 있어 AI 적용 적합도(데이터 가용성·구조화 수준)가 사내 어느 과제보다 높다. → *PoC 리스크가 낮고 즉시 효과 검증 가능*.
- 1차 기술지원(Frontline)은 반복·정형 분석이 다수 → **자동화 ROI가 즉시 회수**된다.

## As-is
**As-is 프로세스:**
장애 접수 → 엔지니어가 sar/sosreport 수동 수집 → grep·육안으로 메트릭/로그 개별 확인 → 경험 기반 가설 수립 → 필요 시 벤더 문의·재현 → 원인 확정·RCA 문서 수기 작성

**Pain Point:**
- 분석 품질이 개인 경험에 종속(속인화), 시니어 부재 시 지연·누락
- sosreport는 수백 개 파일·수 GB → 육안 분석의 한계, 핵심 단서 누락 위험
- 야간·주말 초동 분석 지연 (시니어 호출 대기)
- 동일 유형 장애의 반복 분석, 분석 지식이 개인에 머물고 조직에 비축적
- RCA 문서화 부담 → 사후 지식화가 후순위로 밀림

## To-be
**To-be 프로세스:**
장애 접수 → **데이터 자동 수집·정제 파이프라인** → **AI가 전체 데이터를 대상으로 메트릭·로그 교차분석** → **이상징후·상관관계·추정 원인을 신뢰도 점수와 함께 제시** → 엔지니어가 검증·확정(Human-in-the-loop) → **RCA 초안 자동 생성**

**핵심 변화:**
- 사람이 데이터를 *읽던* 작업 → AI가 1차 분석, 사람은 *검증·판단*에 집중
- **속인화 → 표준화**: 주니어도 시니어 수준의 1차 분석 결과를 즉시 확보
- **사후 분석 → 실시간 초동**: 24/7 1차 대응을 AI가 담당
- **휘발 → 자산화**: 모든 분석이 신뢰도 점수와 함께 지식베이스에 자동 축적

## 연내 달성할 정량적 목표
| 구분 | 목표 | 비고 |
|---|---|---|
| **속도/효율** | 초동 분석·요약 3일 → 4시간 (**83% 단축**), RCA 초안 자동 생성 | 신뢰도 High(요약·초동) / Moderate(원인확정은 검증 단계 유지) |
| **규모/처리량** | OS 1종(Linux) + DB 5종(Oracle·MySQL·MariaDB·PostgreSQL·EDB) 분석 엔진 확보 | "6종"으로 갈 경우 Db2/Tibero 등 1종 추가 확정 필요 |
| **매출/비용** | 장애 분석 공수 절감으로 **건당 인시(man-hour) ~80% 절감**, 동일 인력으로 처리 가능 장애 건수 증가 → MSP 기술지원 원가 절감 | 매출기여(+XX억)는 GDCV·MSP 확산 시점에 산정 권고 |

> 비용 효과를 "+XX억" 빈칸으로 두면 약해 보입니다. 현재 단계에서는 **"분석 인시 절감 → 동일 인력 처리량 N배"**로 정량화하고, 매출은 Phase 3 확산 KPI로 분리하는 편이 방어에 유리합니다. (신뢰도: Moderate — 실제 건당 평균 분석 공수 데이터로 보정 필요)

## 실행 로드맵 — Phase 별 Milestone
**PHASE 1: Quick Win [~7월]**
- Linux OS 성능(sar) / 로그(sosreport) AI 분석 엔진 확보
- 전처리 파이프라인 + 신뢰도 스코어링 + RCA 초안 자동화 MVP
- 내부 실장애 데이터 5~10건 대상 정확도 검증

**PHASE 2: Scale [~10월]**
- DB 5종 성능/로그 AI 원인분석 엔진 확보 (엔진별 수집기: AWR/ASH, slow log, pg_stat_statements 등)
- Grafana 연동 통합 대시보드, 운영 적용

**PHASE 3: Transform [2027년]**
- 국내 확산 (MSP기술지원그룹)
- 해외 확산 (GDCV)

> 로드맵 리스크(신뢰도: Moderate): DB 5종을 4명 TF가 3개월(7→10월)에 모두 구현하는 것은 공격적입니다. 엔진별 데이터 수집기 개발이 병목 → **Phase 2에서 우선순위 2~3종(Oracle, PostgreSQL/EDB) 선출시 후 나머지 순차**로 명시하면 일정 리스크 방어 가능.

## 필요자원 및 확보방안
- **인력:** TF 4명 (PM 박재화, 이창호, 홍성순, 김영도)
- **기술/플랫폼:** FabriX API(폐쇄망 LLM 추론 — 데이터 주권·CSAP 정합성 확보), Claude Code(개발 생산성), Grafana(결과 시각화)
- **데이터/파트너십:** System 성능/로그 데이터 (OS sar·sosreport, DB 성능/로그)

> **데이터 민감성 방어 논리(반드시 추가 권고, 신뢰도: High):** sar·sosreport·DB 로그에는 호스트명·IP·구성정보·자격증명이 포함될 수 있음. 외부 공개 API가 아닌 **사내 FabriX 폐쇄망 처리 + 마스킹**을 명시 → CSAP/보안 관점에서 감점이 아니라 가점 요소로 전환.
