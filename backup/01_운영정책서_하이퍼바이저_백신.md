## 1. 목적 및 배경

### 1.1 목적
본 정책서는 CSAP 9.1.4-③ 통제항목 충족을 위해 SCPv2 컴퓨트 노드에 설치되는 백신 프로그램의 운영 방식, 예외 처리 정책, 보완통제 연계 방안을 규정한다.

### 1.2 배경
SCPv2는 Ubuntu LTS 호스트 OS 위에서 OpenStack KVM 하이퍼바이저로 가상자원을 제공한다. 일반적인 Linux 서버용 백신을 컴퓨트 노드에 그대로 적용할 경우, 다음과 같은 운영 리스크가 존재한다:

1. qcow2/raw 가상디스크 이미지의 실시간 스캔으로 인한 VM I/O 지연
2. qemu-kvm·libvirtd 프로세스에 대한 행위기반 탐지 오탐 시 VM 다운
3. 라이브 마이그레이션 중 파일·포트 접근 차단으로 인한 마이그레이션 실패
4. 호스트 리소스(CPU·메모리) 경합으로 인한 VM 밀도 감소

본 정책은 위 리스크를 관리 가능 수준으로 통제하면서 CSAP 통제항목을 충족하기 위한 운영 표준을 제시한다.

## 2. 백신 설치 정책

### 2.1 설치 대상
- 모든 컴퓨트 노드 (nova-compute 가동 노드)
- 컨트롤러 노드, 네트워크 노드, 스토리지 노드 (별도 정책 적용 가능)

### 2.2 설치 솔루션
- **선정 기준** (4/19 최종 결정 예정)
  1. Linux 서버용 정식 라이선스 보유
  2. 패턴/엔진 자동 업데이트 지원
  3. 실시간 스캔 on/off 분리 가능
  4. 디렉터리·프로세스·해시 기반 예외 정책 지원
  5. 자동 격리/삭제 비활성화 가능
- **후보 솔루션**: AhnLab V3 Net for Linux Server / TrendMicro Server Protect for Linux / ViRobot Server / Trellix ENS for Linux

### 2.3 설치 방식
- 패키지 매니저(dpkg/apt) 또는 벤더 제공 설치 스크립트로 일괄 배포
- Ansible/Salt 등 구성관리 도구로 설정 일관성 보장
- 설치 검증: 모든 컴퓨트 노드에서 `systemctl status <vendor-daemon>` 정상 확인

## 3. 실시간 스캔 정책 (Real-time Protection)

### 3.1 기본 원칙
**실시간 파일 스캔은 비활성화한다.**

### 3.2 비활성화 사유 (공식 근거 인용)

#### 3.2.1 OpenStack 공식 가이드의 디렉터리 특수성 인정
OpenStack Security Guide는 `/var/lib/nova/instances` 디렉터리의 거대 사이즈와 특수성을 인정하여 백업 대상에서 제외할 것을 권고한다 (출처: docs.openstack.org/security-guide/compute/hardening-deployments.html).

> "If your deployment does not require full virtual machine backups, we recommend excluding the /var/lib/nova/instances directory as it will be as large as the combined space of each vm running on that node."

→ 동일 논리로, 해당 디렉터리에 대한 백신 실시간 스캔도 의미 없는 부하를 유발하므로 제외함이 타당하다. qcow2/raw 이미지는 VM 메모리·디스크 변경에 따라 끊임없이 변동되는 거대 sparse 파일이며, 이는 호스트 OS 관점에서 "사용자 파일"이 아닌 "가상화 백엔드 데이터"이다.

#### 3.2.2 OpenStack 공식 보안 권고는 MAC을 우선
OpenStack Security Guide는 하이퍼바이저(QEMU/KVM) 강화의 3대 권고사항을 다음과 같이 제시한다 (출처: docs.openstack.org/security-guide/compute/hardening-the-virtualization-layers.html):

> "We recommend three specific steps: Minimizing the code base. Using compiler hardening. Using mandatory access controls such as sVirt, SELinux, or AppArmor."

→ 백신은 권고 항목에 포함되지 않으며, MAC(sVirt/AppArmor)이 본질적 보호 수단으로 명시됨. 본 정책의 §5 보완통제로 충족.

### 3.3 실시간 스캔 적용 영역 (제한적 활성화)
다음 경로는 호스트 관리 영역으로 한정 실시간 스캔을 적용한다:
- `/etc` (설정 파일)
- `/root`, `/home/*` (관리자 작업 영역)
- `/usr`, `/opt` (패키지 설치 경로)
- `/tmp`, `/var/tmp` (임시 파일)
- 외부 업로드 가능 경로 (별도 식별 시)

## 4. 예약 스캔 정책 (Scheduled Scan)

### 4.1 스캔 주기
- **전체 스캔**: 주 1회 (매주 일요일 02:00 KST)
- **빠른 스캔**: 일 1회 (매일 03:00 KST, 시스템 디렉터리만)

### 4.2 노드 시차 실행
동일 시점에 모든 컴퓨트 노드 스캔 시 클러스터 전체 부하 집중. 노드 ID 해시값 기준 0~3600초 랜덤 지연 적용.

### 4.3 스캔 전 사전 조치
1. 가능 시 해당 노드의 VM을 다른 노드로 라이브 마이그레이션 (`openstack server migrate --live-migration`)
2. 마이그레이션 불가 시 점검 모드 진입 알림 (운영팀 슬랙)

### 4.4 스캔 결과 처리
- **자동 격리/삭제: 비활성화 (탐지 후 알림만)**
- 탐지 시 운영팀에 즉시 알림 (Slack + 이메일)
- 격리는 §3 악성코드 대응절차서에 따라 수동 수행

## 5. 예외 처리 정책 (Exclusion)

### 5.1 디렉터리 예외 (실시간·예약 스캔 모두 제외)

| 경로 | 사유 |
|------|------|
| `/var/lib/nova/instances/` | Nova ephemeral VM 디스크 (qcow2/raw, 거대 sparse 파일, 지속적 I/O) |
| `/var/lib/libvirt/qemu/` | libvirt per-domain 상태 파일 (master-key.aes, XDG 데이터) |
| `/var/lib/libvirt/images/` | libvirt 이미지 풀 |
| `/var/lib/libvirt/qemu/domain-*/` | 도메인별 통신 소켓 및 일시 파일 |
| `/var/log/libvirt/qemu/` | qemu 로그 (지속적 쓰기) |
| `/var/log/nova/` | Nova 로그 |
| `/var/lib/ceph/` | (Ceph 백엔드 사용 시) OSD 데이터 |
| `/var/lib/kolla/` | (Kolla 배포 시) 컨테이너 데이터 디렉터리 |
| `/dev/kvm`, `/dev/vhost-*`, `/dev/vfio/*` | KVM 디바이스 노드 |

> **근거**: 위 디렉터리는 OpenStack Nova/libvirt 공식 문서에 정의된 가상화 백엔드 데이터 저장 경로이며, qemu-kvm 프로세스가 지속적으로 read/write를 수행한다. 실시간 스캔 시 락다운 위험 (출처: docs.openstack.org/nova/latest/admin/configuration/hypervisor-kvm.html, libvirt.org/drvqemu.html).

### 5.2 프로세스 예외 (행위기반 탐지 제외)

| 프로세스 | 역할 |
|---------|------|
| `qemu-kvm`, `qemu-system-x86_64`, `qemu-system-aarch64` | KVM VM 실행 프로세스 |
| `libvirtd`, `virtqemud`, `virtlogd`, `virtsecretd` | libvirt 데몬 |
| `nova-compute`, `nova-api`, `nova-conductor` | OpenStack Nova 데몬 |
| `ovs-vswitchd`, `ovsdb-server` | Open vSwitch |
| `ovn-controller`, `ovn-northd` | OVN 네트워크 |
| `neutron-openvswitch-agent`, `neutron-l3-agent` | Neutron 에이전트 |

> **근거**: qemu-kvm은 root 권한으로 block/PCI device에 광범위한 syscall을 수행하며, 이는 일반적 행위기반 탐지에서 의심 패턴으로 분류될 수 있다 (출처: pve.proxmox.com/wiki/Qemu/KVM_Virtual_Machines).

### 5.3 네트워크 포트 예외 (방화벽형 EPP 사용 시)

| 포트 | 용도 |
|------|------|
| 16509/tcp | libvirtd (TCP, plain) |
| 16514/tcp | libvirtd (TLS) |
| 49152–49215/tcp | QEMU 라이브 마이그레이션 동적 포트 |
| 22/tcp | SSH 터널 마이그레이션 |
| 6800–7300/tcp | (Ceph 사용 시) OSD 통신 |

### 5.4 예외 변경 절차
- 예외 추가/변경은 본 정책서 개정으로 관리
- OpenStack 또는 배포 도구(Kolla 등) 버전 업그레이드 시 경로 변경 여부 재검토
- 변경 이력은 §10 변경관리에 기록

## 6. 패턴 및 엔진 업데이트 정책

### 6.1 자동 업데이트
- 백신 패턴은 **일 1회 자동 업데이트** (벤더 권장 기본값 기준)
- 엔진 업데이트는 월 1회 정기 점검 후 적용
- 폐쇄망 환경 시 내부 업데이트 서버를 통한 mirror 운영

### 6.2 업데이트 검증
- 모든 노드의 패턴 버전을 일 1회 자동 수집 (관리 시스템 대시보드)
- 7일 이상 미업데이트 노드 자동 알림

### 6.3 업데이트 후 검증
- 엔진 업데이트 후 무작위 1개 노드에서 서비스 정상 동작 확인 (qemu-kvm 프로세스 정상, libvirtd 정상, VM 부팅 정상)
- 이상 발견 시 전체 적용 보류

## 7. 리소스 격리 정책

### 7.1 cgroups 기반 자원 한도
백신 데몬을 systemd 슬라이스로 격리하여 컴퓨트 자원과 분리한다.

```ini
# /etc/systemd/system/<vendor-av>.service.d/limits.conf
[Service]
CPUQuota=50%
MemoryMax=2G
CPUAffinity=0-1
IOSchedulingClass=idle
```

### 7.2 NUMA 격리 권고
- 백신 데몬은 관리용 NUMA 노드(예: NUMA0)에 고정
- VM 워크로드는 OpenStack Nova의 `cpu_dedicated_set` 설정으로 별도 NUMA 노드 사용

### 7.3 우선순위
- I/O 스케줄링 클래스를 `idle`로 설정하여 VM I/O와 경합 시 양보
- `nice -n 19`로 CPU 우선순위 최저 설정

## 8. 모니터링 지표

운영 중 다음 지표를 5분 간격으로 수집한다:

| 지표 | 임계치 | 알림 |
|------|-------|------|
| 백신 데몬 가동 상태 | DOWN | 즉시 |
| 패턴 버전 (최신 대비 경과일) | > 7일 | 일 1회 |
| 예약 스캔 마지막 수행일 | > 7일 | 일 1회 |
| 예약 스캔 큐 적체 | > 1000 파일 | 1시간 |
| 백신 프로세스 CPU 사용률 | > 50% (5분 평균) | 즉시 |
| 백신 프로세스 메모리 사용량 | > 2GB | 즉시 |
| 호스트 iowait | > 20% (5분 평균) | 즉시 |
| 호스트 steal time | > 5% (5분 평균) | 즉시 |
| qemu-kvm 스레드 latency | > 1ms (p99) | 즉시 |
| 라이브 마이그레이션 성공률 | < 99% (일별) | 일 1회 |
| 악성코드 탐지 이벤트 | > 0 | 즉시 |

## 9. 보완통제 연계

본 정책의 §3.2~5 운영 방식으로 인한 잔여 위험은 다음 보완통제로 보강한다 (상세는 02_보완통제_증적수집_가이드.md 참조):

| 보완통제 | 역할 | 근거 |
|---------|------|------|
| **AppArmor sVirt** (libvirt 기본) | VM 프로세스별 강제접근통제, 호스트 파일 접근 차단 | Ubuntu Server 공식 문서 |
| **파일 무결성 모니터링 (FIM)** | `/etc`, `/usr/sbin` 등 핵심 시스템 파일 변조 탐지 (AIDE 또는 Samhain) | OpenStack Security Guide |
| **호스트 OS 최소 설치** | 불필요 패키지 제거로 attack surface 축소 | NIST SP 800-125 |
| **네트워크 분리** | 관리망/VM망 물리·VLAN 분리, 호스트 OS는 관리망에서만 접근 가능 | KVM 보안 베스트 프랙티스 |
| **세션 관리 및 감사 로그** | 호스트 SSH 접근 통제(MFA, bastion), auditd 풀 로깅 | CSAP 9.1.1, 9.1.2 |
| **취약점 패치 관리** | 호스트 OS 및 KVM 보안 패치 즉시 적용 (CSAP 9.1.4 본문) | CSAP 9.1.4 본문 |
