
기존 시나리오에 대해 말씀주신 결점에 대한 질문이 있습니다.



기존 시나리오의 절차 중 3번 단계(파란색)에서 NTP #1이 3분 당겨진 NTP #2의 시간을 쫒아가기 시작할 때

동기화 속도 500PPM(0.0005초)에 따라 NTP #1이 128ms 내로 서서히 시간이 조정되니, NTP #1의 시간을 참조하는 스토리지의 시간도 128ms 안쪽에서 쫒아가겠다는 생각을 했습니다.

즉, 스토리지에서 step이 안일어나도록 중간에 NTP #1이 범퍼 역할을 둔거거든요. (그런데 찾아보니 chronyc 환경에선 기본 속도가 훨씬 공격적인 값으로 보이네요..기본 값 8,333PPM)

NTP #2가 3분 당겨진 상태에서 NTP #1이 8,333ppm(0.008333초) 속도로 NTP #2의 시간을 쫒아갈때도 스토리지에서 step이 발생할 가능성이 있을까요?



1. 임시 NTP 2대(NTP #1, #2)를 구성 및 시간을 스토리지 시간과 일치 시킴 (현재 시간 -22분)

2. 다음과 같이 NTP 구성

   스토리지 → NTP #1 → NTP #2

3. NTP #2의 시간을 +3분 빠르게 변경하여 NTP #1이 slew 옵션으로 점진적으로 동기화하여 +3분 당겨지도록 함

4. 스토리지의 시간도 NTP #1을 참조하여 동기화 시작, 이때 slew 옵션 적용되어 동기화하면서 +3분 당겨짐

5. 3개 노드가 동일 시간으로 맞춰지면 다시 6~7차례 반복하며 +22분 차이를 줄임

 ** 단, 전제 조건은 스토리지와 NTP #1의 시간 차이가 128ms 내에서 움직여야 함


Chrony의 공식 문서 및 소스 코드에서 정의하는 기본 최대 Slew 속도(`maxslewrate`)에 대한 인용 정보입니다.

### 1. 공식 매뉴얼 페이지 (chrony.conf) 인용
Chrony의 공식 매뉴얼(`man chrony.conf`)에서는 `maxslewrate` 지시어에 대해 다음과 같이 명시하고 있습니다.

> **"The maxslewrate directive specifies the maximum rate at which chronyd will slew the system clock. The value is specified in parts per million (ppm). The default value is 83333.333, which corresponds to 1/12 of the actual frequency."**
>
> (해석: **maxslewrate** 지시어는 **chronyd**가 시스템 시계를 Slew(점진적 동기화)할 수 있는 최대 속도를 지정합니다. 값은 ppm 단위로 지정됩니다. **기본값은 83333.333이며, 이는 실제 주파수의 1/12에 해당합니다.**)

이 수치는 일반적인 `ntpd`나 Linux 커널의 한계치인 500 PPM보다 약 166배나 공격적인 수치입니다.

---

### 2. 소스 코드 (conf.c) 인용
Chrony 소스 코드 내에서 이 기본값이 정의된 부분은 `conf.c` 파일(또는 버전에 따라 관련 헤더 파일)에서 확인하실 수 있습니다.

```c
/* conf.c 파일 내 정의 부분 */

#define DEFAULT_MAX_SLEW_RATE 83333.333333

/* ... 중략 ... */

void
CNF_Initialise(int reading_from_line, int sparse)
{
  /* ... */
  max_slew_rate = DEFAULT_MAX_SLEW_RATE;
  /* ... */
}
```

---

### 3. 기술적 의미 해석
인용된 **83,333.333 PPM**은 수학적으로 **1/12**이라는 분수에서 기인합니다. 이를 실제 시간 보정 속도로 환산하면 다음과 같습니다.

* **PPM 계산:** $1,000,000 \mu s \div 12 \approx 83,333.333 \mu s$
* **초당 보정량:** 초당 약 **83.3ms** (0.0833초)
* **의미:** Chrony는 기본 설정에서 1초가 흐를 때 시스템 시간을 최대 0.0833초만큼 더 빨리 가게 하거나 느리게 가서 오차를 줄일 수 있습니다.

프로님께서 이전에 언급하셨던 8,333 PPM보다 10배 더 빠른 수치이며, 이 속도로 동기화가 진행될 경우 NetApp NAS의 128ms Step 임계치는 단 **1.5초 내외**($128ms \div (83.3ms - 0.5ms) \approx 1.54s$)에 돌파하게 됩니다. 따라서 `smoothtime` 등을 통해 이 속도를 강제로 제어하는 것이 조치 계획의 핵심입니다.






# SCP v2 대구 PPP 행정망 NAS 스토리지 NTP 장애 조치 계획서

## 1. 장애 상황 및 기술적 제약 요약

현재 대구 PPP 행정망 내 NetApp NAS 스토리지의 시간이 표준 시간 대비 22분(1,320초) 지연되어 기록되는 현상이 발생하고 있습니다. 이를 해결하기 위해 점진적 동기화(Slew)를 적용해야 하나, 시스템 아키텍처 상 다음과 같은 엄격한 물리적/프로토콜 제약이 존재합니다.

* **Panic Threshold (1,000초):** `ntpd` 및 일반적인 시간 동기화 데몬은 서버와의 시간 오차가 1,000초를 초과할 경우 비정상 상태(Sanity Check Fail)로 간주하여 동기화 자체를 포기합니다. 현재 22분(1,320초) 지연은 이 한계치를 초과한 상태입니다.
* **Step Threshold (128ms):** NetApp ONTAP OS는 NTP 서버와의 시간 오차가 128ms 이하일 때만 커널 수준의 점진적 동기화(Slew)를 수행하며, 이를 초과하는 오차가 발생하면 즉각적으로 시간을 강제로 맞추는 도약 동기화(Step)를 실행합니다.
* **Kernel Slew Rate Limit (500 PPM):** Unix/Linux 커널(ONTAP 포함)이 시간을 Slew 방식으로 당기거나 늦출 수 있는 물리적 최대 한계는 초당 0.0005초(500 PPM)입니다. 

## 2. 기존 제안 시나리오의 위험성 평가

과거 제안되었던 '임시 NTP 2대 캐스케이드 구성 및 3분 수동 점프' 또는 'Chrony 1,000 PPM 가속' 방식은 NetApp ONTAP과 커널의 메커니즘을 고려할 때 치명적인 결함을 내포하고 있습니다.

* **3분 수동 점프 방식의 불가 사유:** 시간 오차를 3분(180,000ms)씩 분할하여 당기더라도, 이는 ONTAP의 Step Threshold인 128ms를 아득히 초과합니다. 따라서 NAS는 이를 Slew로 처리하지 않고 즉시 3분 단위의 Step(강제 시간 도약)을 반복 수행하게 됩니다.
* **1,000 PPM 가속(smoothtime) 방식의 불가 사유:** 임시 Chrony 서버에서 `smoothtime` 디렉티브를 이용해 1,000 PPM(초당 1ms) 속도로 시간을 가속하여 송출할 경우, 클라이언트인 NAS는 커널 물리적 한계인 500 PPM(초당 0.5ms) 속도로밖에 쫓아가지 못합니다. 이로 인해 서버와 NAS 간의 시간 격차가 매초 0.5ms씩 누적되며, 약 256초 후에는 오차가 128ms를 초과하게 되어 NAS 스스로 Slew를 포기하고 Step을 트리거하게 됩니다.

## 3. 클라이언트(DB 등) 영향도 및 리스크 분석

NAS 스토리지에 Step(강제 시간 도약)이 발생할 경우, 연결된 클라이언트 애플리케이션의 특성에 따라 다음과 같은 심각한 장애가 발생할 수 있습니다.

* **Tier 1 (MongoDB 등 분산 데이터베이스):** 가장 치명적인 영향을 받습니다. MongoDB는 Oplog(Operations Log)의 타임스탬프를 기반으로 복제(Replication) 및 데이터 정합성을 유지합니다. 스토리지의 기록 시간이 22분이나 강제로 도약할 경우, 트랜잭션 순서가 역전되거나 Replica Set 노드 간의 Heartbeat/Timeout 메커니즘에 혼란이 발생하여 Leader Election 실패, 스플릿 브레인(Split-Brain) 현상 또는 노드 간 동기화 중단 사태가 발생할 수 있습니다.
* **Tier 2 (백업 및 스케줄러):** 스토리지 스냅샷 및 백업 솔루션이 타임스탬프를 기준으로 증분(Incremental) 백업을 수행할 때, 시간의 연속성이 깨져 특정 구간의 백업이 누락되거나 백업 스케줄 자체가 무효화될 위험이 있습니다.
* **Tier 3 (일반 정적 파일 및 로그):** 애플리케이션 로그의 타임스탬프가 비연속적으로 기록되어 보안 감사(Audit) 및 장애 추적 시스템(Splunk 등)의 정합성이 훼손됩니다.

## 4. [최종 권고] 최적의 해결 방안 및 선정 사유

클라이언트(특히 MongoDB)의 서비스 무중단 및 데이터 정합성 보장을 최우선으로 고려할 때, **Plan A(초장기 Slew 동기화)**를 가장 안전한 최적 방안으로 권고합니다. 만약 38일간의 점진적 시간 보정 기간을 비즈니스적으로 수용할 수 없다면, 반드시 애플리케이션 다운타임을 수반하는 **Plan B(유지보수 윈도우 기반 일괄 Step)**를 채택해야 합니다.

* **[권고] Plan A: 400 PPM 기반 초장기 Slew 동기화 (Client 무중단)**
    * **방식:** NAS 전용 임시 Chrony 서버를 구성하고, `smoothtime 400` (400 PPM)으로 설정하여 초당 0.0004초씩 안전하게 시간을 당깁니다.
    * **선정 사유:** NAS 커널 한계(500 PPM)를 넘지 않아 Step을 원천 차단하며, 128ms 임계치 이내에서 완벽한 Slew를 보장합니다.
    * **소요 시간:** 1,320초 / 0.0004 = 약 3,300,000초 (약 38.2일 소요)

* **[대안] Plan B: DB 정지 후 일괄 Step 동기화 (Client 다운타임 필요)**
    * **방식:** 사전 협의된 심야 점검 시간에 MongoDB 등 시간 민감형 애플리케이션을 모두 Shutdown 한 후, NAS 시스템에서 강제로 정상 시간으로 일괄 Step(22분 점프)을 수행합니다.
    * **선정 사유:** 38일의 시간 불일치 상태를 유지하는 것이 비즈니스 룰에 위배될 경우 선택하는 최후의 수단입니다. 애플리케이션 정지가 전제되므로 데이터 정합성 훼손은 막을 수 있습니다.

## 5. 단계별 실행 절차 (Action Plan) - Plan A 기준

**Phase 1: 사전 준비 및 임시 NTP 인프라 구성**
1.  대구 PPP 행정망 내에 NAS와 통신 가능한 임시 리눅스 가상 머신(Temp-NTP)을 프로비저닝합니다.
2.  해당 VM에 `chrony` 패키지를 설치하고, 최상위 표준 타임 서버(Stratum 1/2)와 동기화되도록 설정합니다.

**Phase 2: 임시 Chrony 서버 설정 (안전 한계치 적용)**
1.  Temp-NTP 서버의 `/etc/chrony.conf` (또는 해당 구성 파일)에 아래 디렉티브를 추가하여 서버 자체의 송출 시간을 NAS의 현재 시간에 맞춘 후 초당 400 PPM의 속도로만 당기도록 구성합니다.
    ```text
    # 400 PPM 속도로 1초당 0.0004초씩만 시간 가속
    smoothtime 400 0.001
    ```
2.  `systemctl restart chronyd` 명령으로 서비스를 재시작하여 설정을 반영합니다.

**Phase 3: NAS NTP Target 변경**
1.  NetApp ONTAP CLI에 클러스터 관리자 권한으로 접속합니다.
2.  현재 연결된 NTP 서버 구성을 제거하고, Temp-NTP 서버를 새로운 타겟으로 등록합니다.
    ```text
    cluster time-service ntp server create -server [Temp-NTP_IP_Address]
    ```

**Phase 4: 상태 모니터링 및 완료 판정**
1.  **초기 확인:** 적용 후 Temp-NTP 서버에서 `chronyc clients` 및 ONTAP 측에서 `cluster time-service ntp server show` 명령을 통해 동기화 상태(Reach)를 확인합니다.
2.  **일일 모니터링:** 오프셋이 정상적으로 감소하는지 일일 단위로 모니터링합니다. (하루 약 34초씩 오차 감소 확인)
3.  **최종 복원:** 약 38일 후, NAS의 시간이 표준 시간과 완전히 일치하게 되면 Temp-NTP를 제거하고 원래의 표준 NTP 서버 인프라로 설정을 원복합니다.

## 참고
**smoothtime의 실제 동작 원리 (Server Time Smoothing)**
작업을 수행하고 나면, 임시 NTP 서버의 OS 자체 시간과 NAS에게 쏴주는(송출하는) 시간이 다르게 동작합니다.

OS 시간은 즉시 정상화 (Step): 임시 서버에서 chronyd 데몬을 기동하는 순간, 임시 서버는 표준 NTP를 바라보고 있으므로 자신의 OS 시간을 즉시 현재 표준 시간(+22분 점프)으로 맞춰버립니다. (chrony.conf의 기본 설정인 makestep 때문)

송출 시간은 천천히 가속 (Slew): 하지만 smoothtime 400 0.001 설정이 켜져 있기 때문에, Chrony 데몬은 "내 OS 시간이 방금 22분 점프했으니, 클라이언트(NAS)에게는 이 점프를 숨기고 원래 시간(22분 늦은 시간)부터 초당 400 PPM 속도로 천천히 가속해서 보내줘야겠다"라고 판단합니다.
