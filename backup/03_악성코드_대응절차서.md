## 1. 목적

SCPv2 컴퓨트 노드(Ubuntu KVM 하이퍼바이저)에서 백신 또는 보완통제(AppArmor sVirt, FIM 등)에 의해 악성코드 의심 활동이 탐지되었을 때, 클라우드 서비스 가용성에 미치는 영향을 최소화하면서 신속하게 격리·복구·근본원인 분석을 수행하기 위한 표준 절차를 정의한다.

---

## 2. 적용 범위

### 2.1 적용 대상 사건
1. 백신 솔루션의 악성코드 시그니처 매칭 탐지
2. AppArmor `DENIED` 이벤트의 비정상 패턴 (예: qemu-kvm이 자신의 도메인 외 파일 접근 시도)
3. FIM(AIDE)에 의한 핵심 시스템 파일 무단 변조 탐지
4. auditd 감사 로그상 비정상 syscall 패턴
5. 외부(SOC, 위협 인텔리전스)로부터 받은 침해지표(IoC) 적중

### 2.2 적용 제외
- 알려진 정상 운영 활동에 의한 오탐 (예: nova-compute의 정기 작업)
- 보안팀 사전 승인된 PoC/테스트 활동

---

## 3. 단계별 절차 개요 (5단계)

```
[1] 탐지 (Detection)
       │
       ▼
[2] 초동 분류 (Triage) — 5분 이내
       │
       ├── 오탐(False Positive) → §4.5 화이트리스트 등록 후 종료
       │
       └── 진성 의심 (True Positive Suspected)
              │
              ▼
       [3] 격리 (Containment) — 30분 이내
              │
              ▼
       [4] 분석 및 근본원인 (Investigation) — 24~72시간
              │
              ▼
       [5] 복구 및 재발방지 (Recovery & Lessons Learned)
```

## 4. 단계별 상세 절차

### 4.1 [1단계] 탐지 (Detection)

#### 4.1.1 자동 탐지 채널
| 출처 | 알림 경로 | 임계 |
|------|----------|------|
| 백신 솔루션 | Slack #scp-secalert + 이메일 | 즉시 |
| AppArmor DENIED | journald → 로그 수집기 → SIEM | 임계패턴 적중 시 |
| AIDE 무결성 검사 | 일 1회 자동 실행 결과 비교 | 변경 발생 시 즉시 |
| auditd | SIEM 룰 | 즉시 |

#### 4.1.2 수동 탐지 채널
- 운영자 일상 점검 중 발견
- 외부 SOC/CERT/벤더로부터 IoC 통보
- VM 고객의 신고 (호스트 침해가 의심되는 게스트 이상 동작)

#### 4.1.3 알림 수신자
- 1차: SCPv2 보안팀 당직자 (Slack #scp-secalert)
- 2차 (15분 미응답 시): SCPv2 운영팀 리더, CI-TEC 보안 담당
- 3차 (1시간 미응답 시): CISO, SCPv2 사업 책임자

---

### 4.2 [2단계] 초동 분류 (Triage) — 5분 이내

#### 4.2.1 1차 확인 항목 (Severity 판정)
당직자는 다음 명령으로 1차 정보를 수집한다.

```bash
# (1) 탐지 대상 식별
NODE=<탐지된 노드명>
DETECT_PATH=<탐지된 경로 또는 PID>

# (2) 노드 기본 상태
ssh $NODE "uptime && uname -a && systemctl status nova-compute libvirtd"

# (3) AppArmor 상태
ssh $NODE "sudo aa-status | head -20"

# (4) 비정상 프로세스 확인
ssh $NODE "ps auxf | grep -vE '\[' | head -50"

# (5) 네트워크 비정상 연결 확인
ssh $NODE "sudo ss -tnp state established | head -30"

# (6) 최근 로그인
ssh $NODE "last -n 20 && sudo journalctl _COMM=sshd --since '24 hours ago'"
```

#### 4.2.2 Severity 분류

| 등급 | 판단 기준 | 대응 시간 |
|------|----------|----------|
| **S1 — Critical** | 호스트 root shell 탈취 정황 / 다수 노드 동시 탐지 / VM 격리 위반 정황 | 즉시 |
| **S2 — High** | 단일 노드의 신뢰도 높은 백신 탐지 / AppArmor sVirt 정책 위반 | 30분 |
| **S3 — Medium** | FIM 변경 탐지 (변경 사유 미확인) / 의심스러운 AppArmor 차단 | 2시간 |
| **S4 — Low** | 단순 패턴 매칭 (오탐 가능성 높음) | 영업일 내 |

#### 4.2.3 오탐 판정 기준
다음에 모두 해당하면 오탐으로 판정 가능:
- 탐지 시그니처가 알려진 false positive 사례에 해당 (벤더 KB 확인)
- 탐지 경로가 §4.5 정상 운영 활동 경로
- 동일 시점 다른 보완통제(AppArmor, FIM)에서 이상 없음

### 4.3 [3단계] 격리 (Containment) — 30분 이내

격리는 **"VM 워크로드 보호" → "노드 격리" → "포렌식 보존"** 순으로 진행한다.

#### 4.3.1 VM 워크로드 보호 (라이브 마이그레이션)
> 공식 OpenStack 라이브 마이그레이션 메커니즘을 활용하여 무중단 소개

```bash
# (1) 해당 노드의 VM 목록 확인
openstack server list --host $NODE --all-projects

# (2) 노드 disable (신규 VM 스케줄 차단)
openstack compute service set --disable \
  --disable-reason "security_incident_$(date +%Y%m%d_%H%M)" \
  $NODE nova-compute

# (3) 모든 VM 라이브 마이그레이션
for VM in $(openstack server list --host $NODE --all-projects -c ID -f value); do
  openstack server migrate --live-migration --wait $VM
  echo "Migrated: $VM"
done

# (4) 마이그레이션 완료 확인
openstack server list --host $NODE --all-projects
```

> **공식 근거**: OpenStack Nova는 nova-compute 서비스 disable 후 VM을 다른 노드로 이동하는 패턴을 컴퓨트 노드 제거 절차로 명시함. (출처: docs.oracle.com/cd/E96260_01/E96263/html/nova.html — Oracle OpenStack Nova Compute Service)

#### 4.3.2 마이그레이션 불가 시 대응
다음 사유로 라이브 마이그레이션이 실패하는 VM이 있을 수 있음:
- CPU mode가 host-passthrough로 설정된 VM (호환 호스트 제한)
- PCI passthrough 디바이스 보유 VM
- 활성 SR-IOV 디바이스 보유 VM

→ 처리 방침:
1. 고객 영향이 큰 VM은 "탐지된 위협의 명확한 영향 범위"를 우선 분석
2. 위협이 호스트 OS 수준에 국한되고 VM 내부 데이터에 위험 없음이 확인되면, VM은 유지하고 호스트만 격리 모드 운영
3. 위협이 VM에도 영향 가능 시, 고객 통보 후 VM 정지(soft shutdown → migrate via shared storage → 재기동) 검토

#### 4.3.3 노드 네트워크 격리
```bash
# 관리망만 유지, 데이터망/스토리지망 차단 (예시 - 실제 NIC 명에 맞게 조정)
ssh $NODE "sudo ip link set eth1 down && sudo ip link set eth2 down"

# 또는 방화벽으로 전체 외부 통신 차단 (관리망 SSH는 운영 IP만 허용)
ssh $NODE "sudo iptables -I INPUT 1 -s <운영자IP> -p tcp --dport 22 -j ACCEPT && \
           sudo iptables -I INPUT 2 -j DROP && \
           sudo iptables -I OUTPUT 1 -d <SOC수집서버> -j ACCEPT && \
           sudo iptables -I OUTPUT 2 -j DROP"
```

#### 4.3.4 메모리·디스크 포렌식 보존
**주의**: 노드를 즉시 재부팅하지 말 것 (메모리 증거 휘발).

```bash
# (1) 메모리 덤프 (LiME 또는 avml 사용)
# 사전 설치 권장: apt install lime-forensics-dkms
ssh $NODE "sudo insmod /usr/lib/lime/lime.ko \
  path=/mnt/forensic/${NODE}_$(date +%s).lime format=lime"

# (2) 디스크 이미지 보존 (가능 시)
# 운영 중 dd는 위험 - 별도 forensic 스토리지로 rsync로 핵심 영역만 복사
ssh $NODE "sudo tar czf /mnt/forensic/${NODE}_critical_$(date +%s).tar.gz \
  /var/log /etc /var/lib/libvirt/qemu 2>/dev/null"

# (3) 휘발성 데이터 수집
ssh $NODE "sudo bash -c '
  echo === ps === > /mnt/forensic/${NODE}_volatile.txt
  ps auxf >> /mnt/forensic/${NODE}_volatile.txt
  echo === netstat === >> /mnt/forensic/${NODE}_volatile.txt
  ss -tnpa >> /mnt/forensic/${NODE}_volatile.txt
  echo === lsmod === >> /mnt/forensic/${NODE}_volatile.txt
  lsmod >> /mnt/forensic/${NODE}_volatile.txt
  echo === lsof === >> /mnt/forensic/${NODE}_volatile.txt
  lsof -nP >> /mnt/forensic/${NODE}_volatile.txt
'"
```

#### 4.3.5 격리 완료 보고
- 격리 완료 시점 및 산출물(포렌식 파일 위치) Slack #scp-secalert 보고
- S1/S2 사건은 즉시 CISO·법무·홍보 대응팀 통보

### 4.4 [4단계] 분석 및 근본원인 (Investigation) — 24~72시간

#### 4.4.1 분석 항목

| 항목 | 도구 / 방법 |
|------|-----------|
| 백신 탐지 시그니처의 진위 | 벤더 KB, VirusTotal 멀티엔진 검증 |
| 침투 경로 (Initial Access) | 인증 로그, 패치 미적용 CVE 매칭 |
| 권한 상승 (Privilege Escalation) | sudo 로그, capability 변화 |
| 측면 이동 (Lateral Movement) | SSH 로그, libvirt API 호출 이력 |
| 영향 범위 (Impact) | FIM diff, 동일 노드의 VM 게스트 영향 분석 |
| C2 통신 | 네트워크 플로우 로그, DNS 질의 로그 |

#### 4.4.2 영향 범위 판정 매트릭스

| 영향 영역 | 판정 방법 | 영향 시 추가 조치 |
|----------|----------|-----------------|
| 호스트 OS만 | FIM diff에서 VM 디렉터리 변조 없음, AppArmor 위반 없음 | 노드 재설치 |
| libvirt/qemu 프로세스 | qemu 메모리 덤프에서 이상, AppArmor 위반 발생 | 모든 활성 VM 재기동 검토 |
| VM 게스트 침해 가능성 | qcow2 이미지 hash 변경, 게스트 내부 의심 활동 | 고객 통보, VM 스냅샷 격리 |
| 다중 노드 확산 | SIEM에서 동일 IoC 다른 노드 적중 | 클러스터 전체 비상대응 |

#### 4.4.3 KISA 침해사고 신고
- 정보통신기반보호법·개인정보보호법상 신고 의무 발생 시 기준에 따라 KISA 침해사고대응본부에 신고
- 신고 기준 판단은 법무팀 및 CISO 협의

### 4.5 [5단계] 복구 및 재발방지 (Recovery & Lessons Learned)

#### 4.5.1 노드 복구 결정 트리
```
근본원인 분석 완료
       │
       ├── 침해 미확인 (오탐 확정) ─→ 화이트리스트 등록 → 정상 운영 복귀
       │
       ├── 침해 확인 / 호스트 OS만 ─→ 노드 OS 완전 재설치 → 클린 베이스라인
       │
       └── 침해 확인 / 광범위        ─→ 노드 폐기, 신규 노드로 교체
```

#### 4.5.2 노드 재투입 절차
```bash
# (1) OS 재설치 (PXE 자동화 또는 수동)
# (2) SCPv2 표준 베이스라인 적용 (Ansible/Salt)
# (3) AppArmor sVirt 정상 작동 검증 (02_보완통제_증적수집_가이드.md §1)
# (4) 백신 설치 및 정책 적용 (01_운영정책서_하이퍼바이저_백신.md §2~5)
# (5) 단일 테스트 VM으로 동작 검증
# (6) nova-compute enable
openstack compute service set --enable $NODE nova-compute
# (7) 점진적 워크로드 재배치
```

#### 4.5.3 재발방지 (Post-Incident Review)
사건 종료 후 **5영업일 이내** 다음 산출물을 작성:
1. 사건 보고서 (Incident Report)
   - 타임라인
   - 근본원인 (5-Why 분석)
   - 영향 범위
   - 비용 (작업시간, 다운타임)
2. 재발방지 액션 (Action Items)
   - 탐지 룰 보강
   - 운영 절차 보완
   - 본 절차서 개정 필요 사항
3. 정책 갱신
   - 본 절차서, 운영정책서 개정 반영

#### 4.5.4 정상 운영 활동 화이트리스트 등록
오탐 확정 시:
1. 탐지 패턴(시그니처, 파일 경로, 프로세스명) 기록
2. 백신 솔루션의 화이트리스트(예외)에 추가
3. 본 절차서 부록에 사례 기록

## 5. 역할 및 책임 (RACI)

| 활동 | 보안팀 당직자 | SCPv2 운영팀 | CI-TEC | CISO | 법무 |
|------|:---:|:---:|:---:|:---:|:---:|
| 탐지·초동분류 | R/A | C | C | I | I |
| 격리(VM 마이그레이션·노드분리) | A | R | C | I | I |
| 포렌식 수집 | C | R | A | I | I |
| 분석·RCA | C | C | R/A | I | C |
| 복구·재투입 | C | R/A | C | I | I |
| 외부 신고(KISA 등) | C | C | C | A | R |
| Post-Incident Review | R | C | C | A | I |

R: Responsible, A: Accountable, C: Consulted, I: Informed

## 6. 핵심 명령어 빠른참조 (Cheat Sheet)

```bash
# ─── 격리 ───
openstack compute service set --disable --disable-reason "incident_$(date +%s)" <NODE> nova-compute
openstack server list --host <NODE> --all-projects
for VM in $(openstack server list --host <NODE> --all-projects -c ID -f value); do
  openstack server migrate --live-migration --wait $VM
done

# ─── 노드 상태 신속 확인 ───
ssh <NODE> "sudo aa-status | head; sudo virsh list; sudo systemctl status libvirtd nova-compute"

# ─── 비정상 프로세스 ───
ssh <NODE> "ps auxf | grep -vE 'libvirtd|qemu|nova|ovs|systemd|kworker'"

# ─── AppArmor 차단 이력 ───
ssh <NODE> "sudo journalctl -k --since '24 hours ago' | grep -i 'apparmor.*denied'"

# ─── FIM 최신 diff ───
ssh <NODE> "sudo aide --check 2>&1 | head -100"

# ─── 노드 재투입 ───
openstack compute service set --enable <NODE> nova-compute
```
