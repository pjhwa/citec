# 리눅스 네트워크 스택에서 TSO, RRO의 상호 작용 이해

## 질문
환경: Ubuntu 22.04, PowerEdge XE9680, BCM57504 NetXtreme-E 10Gb/25Gb/40Gb/50Gb/100Gb/200Gb Ethernet [14e4:1751] (rev 12)
리눅스 네트워크 스택에서 TSO (TCP Segmentation Offload)가 활성화된 상태에서 GRO (Generic Receive Offload)를 활성화하면, 패킷 덤프 도구 (예: tcpdump 또는 Wireshark)에서 TSO가 적용되지 않은 것처럼 패킷이 작은 크기로 분할되어 보이는 현상이 발생하는데, 이는 왜 그런가? TSO와 GRO의 상호 작용, NIC (Network Interface Card)의 offload 처리 과정, 패킷 캡처 지점 (예: 네트워크 스택 내 위치)에 따른 차이, 그리고 관련 ethtool 설정이나 커널 동작을 고려하여 사실 기반으로 자세히 설명해 주세요. 비판적 검증을 통해 가능한 오해 (예: TSO가 실제로 비활성화되는지 여부)를 명확히 하고, 예시나 참조 자료를 포함하세요.

## 답변

리눅스 네트워크 스택에서 TSO (TCP Segmentation Offload)와 GRO (Generic Receive Offload)의 상호 작용을 이해하려면, 먼저 각 기능의 역할을 명확히 해야 합니다. TSO는 송신 측(TX)에서 큰 TCP 세그먼트를 NIC (Network Interface Card)가 하드웨어 수준에서 MTU (Maximum Transmission Unit, 일반적으로 1500바이트) 크기의 작은 패킷으로 분할하는 offload 기능입니다. 반대로 GRO는 수신 측(RX)에서 여러 작은 패킷을 하나의 큰 skb (socket buffer)로 병합하여 CPU 부하를 줄이는 기능입니다. 이 두 기능은 원칙적으로 독립적이지만, 패킷 캡처 도구(tcpdump나 Wireshark)에서 관찰되는 현상은 네트워크 스택 내 캡처 지점과 offload 처리 과정에 따라 다르게 나타날 수 있습니다.

사용자가 관찰한 현상—"TSO가 활성화된 상태에서 GRO를 활성화하면 패킷 덤프에서 TSO가 적용되지 않은 것처럼 패킷이 작은 크기로 분할되어 보임"—은 일반적인 오해에서 비롯될 수 있습니다. TSO가 실제로 비활성화되는 것은 아니지만, 캡처 지점과 offload 설정의 상호 작용으로 인해 그런 착각이 생길 수 있습니다. 아래에서 사실 기반으로 자세히 설명하겠습니다. 설명은 리눅스 커널 문서, ethtool 동작, 그리고 BCM57504 NIC (Broadcom bnxt_en 드라이버)를 사용하는 Ubuntu 22.04 환경을 고려합니다. 비판적 검증을 위해 여러 소스(커널 문서, Broadcom 지식 베이스, 커뮤니티 포럼)를 교차 확인하였으며, 가능한 오해를 명확히 하겠습니다.

### 1. TSO와 GRO의 기본 동작과 상호 작용
- **TSO의 처리 과정**:
  - TSO 비활성화 시: 애플리케이션이 보낸 큰 데이터가 리눅스 TCP/IP 스택에서 MTU 크기로 분할됩니다. 이후 qdisc (queueing discipline)를 거쳐 NIC driver로 전달되어 와이어로 전송됩니다.
  - TSO (Generic Segmentation Offload, TSO의 소프트웨어 버전)와 TSO 활성화 시: TCP 스택이 세그먼션을 미루고 큰 skb (예: 64KB 이하)를 NIC에 넘깁니다. NIC가 하드웨어에서 이를 MTU 크기의 패킷으로 분할합니다. 이는 CPU 부하를 줄여 throughput을 높입니다.
  - 상호 작용: GRO와 TSO/TSO는 설계상 대칭적입니다. GRO로 병합된 프레임은 GSO로 다시 분할될 수 있도록 보장되며 (IPv4 ID 필드 예외), 이는 터널링(예: Generic Routing Encapsulation) 환경에서도 적용됩니다. 하지만 TSO는 TX 중심, GRO는 RX 중심이므로 직접적으로 TSO를 비활성화하지 않습니다.<grok:render card_id="93d6b8" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">20</argument></grok:render> 이는 커널 문서에서 명확히 확인되며, TSO는 TX checksum offload와 scatter-gather가 선행되어야 합니다.<grok:render card_id="506aee" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">7</argument></grok:render>

- **GRO의 처리 과정**:
  - GRO 비활성화 시: NIC가 수신한 작은 패킷들이 개별로 스택으로 전달됩니다.
  - GRO 활성화 시: NAPI (New API) poll에서 driver가 패킷을 병합하여 큰 skb로 만들어 스택으로 넘깁니다. 이는 receive path에서 효율적입니다.
  - TSO와의 상호 작용: GRO는 receive offload이므로 송신 측 TSO에 직접 영향을 주지 않습니다. 그러나 패킷 forwarding이나 bridge 설정에서 GRO가 병합한 패킷을 TSO가 재사용할 수 있어 간접적으로 효율을 높일 수 있습니다.<grok:render card_id="eec370" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">28</argument></grok:render> 일부 NIC에서 GRO/LRO (Large Receive Offload, GRO의 이전 버전)가 활성화되면 TCP window 크기가 줄어 성능 저하가 발생할 수 있지만, 이는 TSO 비활성화가 아니라 MSS (Maximum Segment Size) 계산 오류 때문입니다.<grok:render card_id="755fcc" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">25</argument></grok:render>

- **NIC offload 처리 (BCM57504 경우)**:
  - BCM57504는 Broadcom NetXtreme-E 시리즈로, bnxt_en 드라이버를 사용합니다. 이 드라이버는 TSO, GSO, GRO를 모두 지원하며, ethtool로 설정합니다.
  - bnxt_en에서 GRO 활성화는 TSO를 자동 비활성화하지 않습니다.<grok:render card_id="843a15" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">30</argument></grok:render> 그러나 특정 커널 버전(예: 6.8)에서 네트워크 성능 이슈가 보고되었으나, TSO/GSO/GRO disable로 해결되지 않았습니다 (다른 드라이버 문제 가능).<grok:render card_id="9ff186" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">37</argument></grok:render> Broadcom 문서에 따르면, TSO는 outbound packet을 처리하며, GRO는 RX입니다.<grok:render card_id="99d509" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">8</argument></grok:render>

### 2. 패킷 캡처 지점에 따른 차이
이 현상의 핵심은 tcpdump나 Wireshark의 캡처 지점입니다. 리눅스에서 패킷 캡처는 packet socket을 사용하며, 스택 내 특정 위치에서 발생합니다. 이는 offload가 적용되기 전/후에 따라 보이는 패킷이 다릅니다.

- **송신 측 (TX) 캡처** (dev_queue_xmit 지점):
  - TSO 활성화: 캡처는 NIC 세그먼테이션 전에 일어나므로, 큰 패킷 (MTU 초과, 예: 1804바이트)이 보입니다.<grok:render card_id="51c33c" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">2</argument></grok:render><grok:render card_id="2b604b" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">3</argument></grok:render><grok:render card_id="6417ad" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">9</argument></grok:render>
  - TSO 비활성화: 작은 MTU 크기 패킷이 보입니다.
  - GRO 활성화 영향: GRO는 RX이므로 TX 캡처에 영향을 주지 않습니다. 만약 GRO 활성화 후 작은 패킷이 보인다면, TSO가 실제 비활성화된 것일 수 있습니다 (사용자 설정 오류나 드라이버 버그).

- **수신 측 (RX) 캡처** (__netif_receive_skb_core 지점):
  - GRO 비활성화: 와이어에서 온 작은 패킷 (MTU 크기)이 그대로 보입니다.
  - GRO 활성화: driver가 병합 후 캡처되므로 큰 패킷이 보입니다 (napi_gro_receive 호출 후).<grok:render card_id="4cfb95" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">27</argument></grok:render><grok:render card_id="052bae" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">18</argument></grok:render>
  - TSO와의 연계: 송신 측 TSO on으로 와이어에 작은 패킷이 도착하면, GRO on에서 병합되어 큰 패킷으로 보입니다. 이는 "TSO가 적용되지 않은 것처럼" (작은 패킷 기대하지만 큰 패킷 보임) 착각을 일으킬 수 있지만, 반대입니다. 실제 와이어는 항상 작은 패킷입니다 (TSO on/off 모두).

- **비판적 검증: 오해 명확히**:
  - 오해 1: TSO on 시 와이어/덤프에서 큰 패킷 보임? → No, 와이어는 항상 작은 패킷 (MTU). 덤프는 TX에서 큰 (TSO on), RX에서 병합 (GRO on)만 봅니다.<grok:render card_id="f6189f" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">4</argument></grok:render> TSO는 세그먼테이션을 offload할 뿐, 와이어 패킷 크기를 바꾸지 않습니다.
  - 오해 2: GRO 활성화 시 TSO 비활성화? → No, ethtool -K로 GRO on만 설정되며 TSO는 그대로입니다.<grok:render card_id="999f54" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">10</argument></grok:render> 하지만 일부 도구(ntopng)나 분석에서 GRO off 권장, 왜냐하면 wire-as-is 패킷을 기대하기 때문.<grok:render card_id="8a5bcd" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">1</argument></grok:render> bnxt_en에서 확인된 이슈는 별도입니다.
  - 오해 3: 패킷이 "작게 분할되어 보임" = TSO off? → Yes, but if capturing on TX and seeing small, TSO off; on RX and small, GRO off or merge 실패.
  - 검증 방법: ethtool -k eth0 명령으로 설정 확인 (tx-tcp-segmentation: on, rx-gro: on). 만약 GRO on 후 tx-tso off라면, ethtool -K eth0 tso on으로 재설정.

### 3. ethtool 설정과 커널 동작
- **설정 명령**:
  - 확인: `ethtool -k eth0` (tx-tcp-segmentation, rx-gro 확인).
  - GRO 활성화: `ethtool -K eth0 gro on`
  - TSO 활성화: `ethtool -K eth0 tso on` (gso on도 함께, scatter-gather on 확인).
  - 모든 offload disable (테스트용): `ethtool -K eth0 tso off gso off gro off`<grok:render card_id="4eadb5" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">14</argument></grok:render>
  - Ubuntu 22.04에서 bnxt_en은 기본 on, /etc/network/interfaces나 systemd-networkd에 추가해 재부팅 시 유지.

- **커널 동작 고려**:
  - Ubuntu 22.04 (kernel 5.15+): GRO는 NAPI에서 처리, TSO는 NETIF_F_TSO 플래그.<grok:render card_id="4988c0" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">12</argument></grok:render>
  - PowerEdge XE9680 BCM57504: RoCE 지원하지만, Ethernet 모드에서 GRO on 시 성능 저하 보고 (TCP window clamp).<grok:render card_id="444fd2" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">11</argument></grok:render> Disable로 테스트 추천.

### 4. 예시와 참조 자료
- **예시**:
  - 송신 측 TSO on, tcpdump: 큰 패킷 보임 (e.g., length 1804). 와이어 모니터링(Wireshark on switch): 작은 MTU 패킷.
  - GRO on 추가, RX 캡처: 병합된 큰 패킷 (e.g., 64KB). 작은 패킷 보이면 GRO merge 실패 (timeout or header mismatch).
  - 명령 예: `tcpdump -i eth0 -nn -vv tcp`로 캡처, TSO on 시 "tcp len" 큰 값 확인.
