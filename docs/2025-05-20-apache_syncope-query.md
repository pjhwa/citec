## 장애 개요 및 발생 과정

### **장애 현상**
- **환경**: MFA 서비스 SingleID DB (MariaDB 10.6.14, 이중화 구조)
- **문제**: 임시 디스크(56GB)가 가득 차면서 데이터베이스 Hang 발생
- **발생 시점**: 4개의 장시간 실행 쿼리가 대규모 정렬(sort)을 유발하며 임시 디스크를 모두 소진
  - 예: `thread ID: 337423063, RUNTIME: 3046 sec` 등, 4개 쿼리 모두 약 2900초 이상 실행

### **발생 과정**
1. **비효율적인 쿼리 실행**: 조회 조건에 해당하는 컬럼에 인덱스가 없어 쿼리 실행 계획이 비효율적으로 변경됨. 이는 MariaDB의 통계 정보 자동 업데이트로 인해 발생.
2. **임시 디스크 소진**: 대규모 정렬 작업으로 인해 임시 디스크(56GB)가 완전히 가득 참.
3. **데이터베이스 Hang**: MariaDB 10.6.14 버전에서는 임시 디스크가 가득 차면 공간이 확보될 때까지 최대 60초 간격으로 대기하며, 이로 인해 모든 세션이 중단됨.

---

## MariaDB 버그(MDEV-33813) 분석

### **버그 내용**
- **설명**: MDEV-33813은 임시 디스크가 가득 찬 후 공간이 확보되었음에도 후속 쿼리가 "no space left on device" 오류로 실패하는 문제를 다룸.
- **재현 조건**: Aria 스토리지 엔진에서 테스트됨. 임시 디스크가 100% 가득 찼다가 공간이 확보된 상황에서 오류 발생.
- **수정**: MariaDB 10.6.18에서 패치됨. 디스크가 가득 차면 해당 쿼리를 에러 처리하고 사용 중이던 임시 파일을 삭제하여 공간을 확보하도록 변경.
  - 패치 내용: [GitHub Commit 3655cefc42](https://github.com/MariaDB/server/commit/3655cefc42)
  - 결과: 다른 세션에 영향을 주지 않고 해당 쿼리만 종료.

### **SingleID 장애와의 연관성**
- **차이점**:
  - **MDEV-33813**: 디스크가 가득 찼다가 공간이 확보된 후에도 오류가 지속됨.
  - **SingleID**: 디스크가 100% 가득 찬 상태가 지속됨. 공간이 확보되지 않았음.
- **스토리지 엔진**:
  - SingleID는 데이터 테이블에 InnoDB를 사용하지만, 기본 설정(`internal_tmp_disk_storage_engine=Aria`)에 따라 임시 테이블은 Aria를 사용. 따라서 버그가 완전히 무관하지는 않음.
- **결론**: SingleID의 경우, 임시 디스크가 가득 찬 상태가 유지되었기 때문에 버그의 "공간 확보 후 오류 지속" 시나리오와는 다름. 디스크가 가득 찬 상태에서 "no space left on device" 오류가 발생한 것은 자연스러운 현상이며, 버그가 없더라도 발생했을 가능성이 높음.

## 각 주장의 타당성 분석

### **애플리케이션 담당자의 주장**
1. **쿼리 생성**: Apache Syncope 오픈소스 솔루션에서 JPA로 자동 생성된 쿼리이며, 장애 전까지 문제없던 쿼리였다고 주장.
2. **버그 연관성**: 임시 디스크가 가득 찼을 때 DB Hang이 발생한 것은 MDEV-33813 버그 때문이며, 버그가 수정되었다면 다른 세션에 영향을 주지 않았을 것.
3. **영향 범위**: 버그가 없었다면 단순 관리용 쿼리로 인한 느린 성능 문제로 끝났을 것.

### **데이터베이스 지원 전문가의 주장**
1. **근본 원인**: 인덱스 부재로 쿼리가 비효율적으로 실행되며 임시 디스크를 소진한 것이 주원인.
2. **버그와의 차이**: MDEV-33813은 디스크가 가득 찼다가 공간이 확보된 경우를 다루지만, SingleID는 디스크가 계속 100% 유지됨. 이는 버그와 무관한 자연스러운 결과.
3. **스토리지 엔진**: 버그는 Aria에서 테스트되었으나, SingleID는 InnoDB를 사용하며 상황이 다르다고 주장(단, 임시 테이블은 Aria 사용).

### **장애 등급 판정 위원회 의견**
- 데이터 증가와 통계 정보 변화로 쿼리 실행 계획이 변하며 성능 문제가 발생할 수 있음.
- 운영 중 장시간 실행 쿼리에 대한 지속적인 관리와 개선 필요성을 강조.

## 판단 근거

### **근본 원인: 비효율적인 쿼리**
- **증거**: 4개의 쿼리가 각각 2900초 이상 실행되며 대규모 정렬을 유발, 임시 디스크를 모두 소진.
- **분석**: 인덱스가 없었다면 쿼리는 전체 테이블 스캔(full table scan) 또는 비효율적인 정렬을 수행했을 가능성이 높음. 이는 MariaDB 통계 정보 업데이트로 실행 계획이 변경되면서 악화됨.
- **결론**: 임시 디스크가 가득 찬 것은 버그가 아니라 쿼리의 비효율성 때문.

### **버그의 역할**
- **한계**: MDEV-33813은 디스크가 가득 찼다가 공간이 확보된 후의 문제를 다룸. SingleID에서는 공간이 확보되지 않았으므로 버그의 직접적인 영향은 미미함.
- **MariaDB 10.6.14 동작**: 디스크가 가득 차면 대기 상태에 들어가며, 이는 버그가 아니라 당시 버전의 기본 동작. 패치(10.6.18) 이후에는 에러 처리로 변경되었으나, 근본적으로 디스크 소진을 막지 못함.

### **운영 관리의 중요성**
- 위원회의 의견처럼, 데이터 증가와 실행 계획 변화는 피할 수 없는 현상. 이를 관리하지 않으면 성능 문제가 반복될 가능성이 높음.

**결론**: 데이터베이스 지원 전문가의 주장이 더 타당함. 버그는 상황을 악화시켰을 수 있으나, 근본 원인은 인덱스 없는 쿼리의 비효율성에 있음.

## 애플리케이션 담당자를 논리적으로 설득하는 반박

### **반박 1: 쿼리의 비효율성이 주원인**
- **당신의 주장**: 쿼리는 Apache Syncope에서 생성된 것이며, 장애 전까지 문제없었다.
- **반박**: 쿼리가 자동 생성되었다 하더라도, 인덱스 부재로 인해 대규모 정렬이 발생하며 56GB 임시 디스크를 소진한 것은 부인할 수 없는 사실입니다. 3046초, 2985초 등 실행 시간이 비정상적으로 길어진 것은 실행 계획이 비효율적으로 변경된 결과이며, 이는 통계 정보 업데이트와 데이터 증가로 예견된 문제입니다. 버그가 없었다 하더라도 디스크 소진은 피할 수 없었을 것입니다.

### **반박 2: 버그의 직접적 연관성 부족**
- **당신의 주장**: MDEV-33813이 DB Hang을 유발했으며, 패치되었다면 다른 세션에 영향을 주지 않았을 것.
- **반박**: 버그는 디스크가 가득 찼다가 공간이 확보된 후에도 오류가 발생하는 경우를 다룹니다. 하지만 SingleID에서는 4개 쿼리가 계속 실행 중이어서 디스크가 100% 유지되었습니다. 이는 버그의 재현 조건과 다르며, 10.6.14의 기본 동작(대기)으로 충분히 설명됩니다. 패치(10.6.18)는 디스크 소진 후 에러 처리를 추가하지만, 쿼리가 디스크를 소진하는 것을 막지 못합니다. 따라서 버그는 결과에 영향을 주었을 뿐, 원인이 아닙니다.

### **반박 3: 스토리지 엔진 논쟁**
- **당신의 주장**: Aria와 InnoDB를 동시에 사용하는 환경이므로 버그가 관련 있다.
- **반박**: 임시 테이블이 Aria를 사용하더라도, 버그의 핵심은 "공간 확보 후 오류"입니다. SingleID에서는 공간이 확보되지 않았으므로 버그의 영향을 받았다고 보기 어렵습니다. 디스크가 가득 찬 상태에서의 오류는 Aria나 InnoDB와 관계없이 발생하는 일반적인 현상입니다.

### **반박 4: 예방 가능성**
- **당신의 주장**: 버그가 없었다면 단순 느린 쿼리로 끝났을 것.
- **반박**: 2900초 이상 실행된 쿼리가 "단순 느린 쿼리"로 간주될 수 없습니다. 이는 시스템 전체를 마비시킬 정도로 자원을 소진했으며, 버그 없이도 심각한 장애로 이어졌을 가능성이 높습니다. 인덱스와 쿼리 최적화로 디스크 소진을 막았다면 장애 자체가 발생하지 않았을 것입니다.

### **제안**
- **행동 방안**: 인덱스를 추가하고, 쿼리 실행 계획을 정기적으로 모니터링하며, 데이터 증가를 고려한 최적화를 수행하세요. 버전 업그레이드(10.6.18 이상)는 보조적 조치일 뿐, 근본 해결책은 쿼리 관리에 있습니다.

---

## 결론

데이터베이스 지원 전문가의 입장이 옳으며, 장애의 근본 원인은 인덱스 없는 비효율적인 쿼리입니다. MariaDB 버그(MDEV-33813)는 상황을 악화시켰을 수 있으나, 디스크 소진을 초래한 주원인은 아닙니다. 지속적인 쿼리 관리와 최적화가 재발 방지의 핵심입니다.
