## 구성

| 점검내용 | 점검방법 | 점검기준 | 취약시 문제점 | 개선방안 | 참고 |
|----------|----------|----------|--------------|----------|------|
| **LB 인스턴스 타입 설정** | SCP 콘솔에서 LB 인스턴스 설정 확인 | 정상: 애플리케이션 요구사항에 맞는 인스턴스 타입 설정 (예: ALB/NLB) <br> 취약: 부적절한 인스턴스 타입 선택 | 부적절한 인스턴스 타입으로 인해 성능 저하 또는 불필요한 비용 발생 | 애플리케이션 요구사항(HTTP/HTTPS, TCP/UDP 등)에 따라 ALB 또는 NLB 선택, 트래픽 패턴 분석 후 적합한 타입 설정 | 설정방법: SCP 콘솔 > Load Balancer > 인스턴스 설정 <br> References: [AWS Elastic Load Balancing Documentation](https://docs.aws.amazon.com/elasticloadbalancing/) |[](https://aws.amazon.com/blogs/industries/fsi-services-spotlight-featuring-elastic-load-balancing-elb-2/)
| **리스너 프로토콜 설정** | SCP 콘솔에서 LB 리스너 설정 확인 | 정상: HTTPS 또는 TLS 사용 <br> 취약: HTTP 또는 비보안 프로토콜 사용 | 비보안 프로토콜 사용 시 데이터 유출 위험 증가 | HTTPS/TLS 리스너 설정, SSL 인증서 적용 | 설정방법: SCP 콘솔 > Load Balancer > Listeners > Add Listener <br> References: [AWS ELB Listeners](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/create-https-listener.html) |
| **SSL/TLS 인증서 유효성** | SCP 콘솔 또는 CLI에서 인증서 만료일 확인 (`aws acm describe-certificate`) | 정상: 인증서 유효기간 30일 이상 남음 <br> 취약: 인증서 만료 또는 30일 이내 만료 예정 | 인증서 만료 시 HTTPS 연결 실패, 서비스 중단 발생 | ACM에서 인증서 갱신, 자동 갱신 설정 활성화 | 설정방법: SCP 콘솔 > Certificate Manager > Renew Certificate <br> References: [AWS ACM Documentation](https://docs.aws.amazon.com/acm/) |
| **헬스 체크 설정** | SCP 콘솔에서 헬스 체크 경로 및 임계값 확인 | 정상: 헬스 체크 경로 및 임계값이 애플리케이션 상태를 정확히 반영 <br> 취약: 잘못된 경로 또는 느슨한 임계값 설정 | 부정확한 헬스 체크로 인해 비정상 타겟으로 트래픽 라우팅 | 헬스 체크 경로를 애플리케이션의 실제 상태 반영 경로로 설정, 임계값 조정 (예: 3회 실패, 5초 간격) | 설정방법: SCP 콘솔 > Load Balancer > Health Checks <br> References: [AWS ELB Health Checks](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html) |[](https://www.loadbalancer.org/blog/load-balancing-dicom-pacs-health-check/)
| **타겟 그룹 설정** | SCP 콘솔에서 타겟 그룹 확인 | 정상: 타겟 그룹에 올바른 인스턴스/컨테이너 등록 <br> 취약: 잘못된 타겟 등록 또는 누락 | 잘못된 타겟으로 트래픽 라우팅, 서비스 장애 발생 | 타겟 그룹에 적절한 EC2 인스턴스 또는 ECS 서비스 등록 확인 및 수정 | 설정방법: SCP 콘솔 > Load Balancer > Target Groups <br> References: [AWS Target Groups](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group.html) |
| **보안 그룹 설정** | SCP 콘솔에서 LB 및 타겟 인스턴스의 보안 그룹 확인 | 정상: 필요한 포트만 허용 (예: 80, 443) <br> 취약: 불필요한 포트 오픈 | 불필요한 포트 오픈으로 보안 취약점 발생 | 보안 그룹에서 최소 권한 원칙 적용, 불필요한 포트 차단 | 설정방법: SCP 콘솔 > Security Groups > Edit Inbound Rules <br> References: [AWS Security Groups](https://docs.aws.amazon.com/vpc/latest/userguide/VPC_SecurityGroups.html) |
| **서브넷 설정** | SCP 콘솔에서 LB 서브넷 설정 확인 | 정상: 최소 2개 이상의 가용 영역(AZ)에 서브넷 설정 <br> 취약: 단일 AZ에만 서브넷 설정 | 단일 AZ 장애 시 LB 서비스 중단 | 다중 AZ에 서브넷 설정, 트래픽 분산 확인 | 설정방법: SCP 콘솔 > Load Balancer > Subnets <br> References: [AWS ELB Subnets](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-subnets.html) |
| **크로스 존 로드 밸런싱** | SCP 콘솔에서 크로스 존 로드 밸런싱 설정 확인 | 정상: 크로스 존 로드 밸런싱 활성화 <br> 취약: 비활성화 상태 | 특정 AZ로 트래픽 편중, 성능 저하 발생 | 크로스 존 로드 밸런싱 활성화로 트래픽 균등 분배 | 설정방법: SCP 콘솔 > Load Balancer > Attributes > Cross-Zone Load Balancing <br> References: [AWS Cross-Zone Load Balancing](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-subnets.html#cross-zone-load-balancing) |
| **액세스 로그 활성화** | SCP 콘솔에서 액세스 로그 설정 확인 | 정상: 액세스 로그 활성화 및 S3 버킷 지정 <br> 취약: 액세스 로그 비활성화 | 트래픽 패턴 분석 및 문제 진단 어려움 | 액세스 로그 활성화, S3 버킷에 로그 저장 설정 | 설정방법: SCP 콘솔 > Load Balancer > Attributes > Access Logs <br> References: [AWS ELB Access Logs](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-access-logs.html) |[](https://aws.amazon.com/blogs/industries/fsi-services-spotlight-featuring-elastic-load-balancing-elb-2/)
| **WAF 연동** | SCP 콘솔에서 WAF 연동 여부 확인 | 정상: WAF 활성화 및 적절한 규칙 설정 <br> 취약: WAF 미연동 | 웹 공격(예: SQL 인젝션, XSS)에 취약 | WAF 활성화, OWASP Top 10 규칙 적용 | 설정방법: SCP 콘솔 > WAF > Web ACLs <br> References: [AWS WAF Documentation](https://docs.aws.amazon.com/waf/) |
| **타임아웃 설정** | SCP 콘솔에서 Idle Timeout 설정 확인 | 정상: 애플리케이션 요구사항에 맞는 타임아웃 설정 <br> 취약: 지나치게 짧거나 긴 타임아웃 | 짧은 타임아웃으로 연결 끊김, 긴 타임아웃으로 리소스 낭비 | 일반적으로 60~300초로 설정, 애플리케이션 특성에 맞게 조정 | 설정방법: SCP 콘솔 > Load Balancer > Attributes > Idle Timeout <br> References: [AWS ELB Timeouts](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/application-load-balancers.html#connection-idle-timeout) |
| **스티키 세션 설정** | SCP 콘솔에서 스티키 세션 설정 확인 | 정상: 애플리케이션 요구사항에 따라 적절히 설정 <br> 취약: 불필요한 스티키 세션 활성화 | 불필요한 스티키 세션으로 인해 트래픽 불균형 | 애플리케이션 상태 유지 필요 여부 확인 후 설정 | 설정방법: SCP 콘솔 > Target Groups > Attributes > Stickiness <br> References: [AWS Sticky Sessions](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/sticky-sessions.html) |
| **헬스 체크 프로토콜** | SCP 콘솔에서 헬스 체크 프로토콜 확인 | 정상: HTTP/HTTPS 사용 <br> 취약: TCP 또는 부적절한 프로토콜 사용 | 부정확한 헬스 체크로 비정상 타겟 라우팅 | HTTP/HTTPS로 헬스 체크 설정, 응답 코드 200 확인 | 설정방법: SCP 콘솔 > Target Groups > Health Checks <br> References: [AWS Health Check Protocols](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html) |
| **로드 밸런싱 알고리즘** | SCP 콘솔에서 로드 밸런싱 알고리즘 확인 | 정상: Round Robin 또는 Least Connection 사용 <br> 취약: 부적절한 알고리즘 선택 | 트래픽 불균형으로 인해 특정 서버 과부하 | 애플리케이션 특성에 따라 적절한 알고리즘 선택 | 설정방법: SCP 콘솔 > Target Groups > Attributes <br> References: [Load Balancing Algorithms](https://www.cloudflare.com/learning/performance/types-of-load-balancing-algorithms/) |[](https://www.cloudflare.com/learning/performance/types-of-load-balancing-algorithms/)
| **DNS 설정** | SCP 콘솔 또는 Route 53에서 DNS 레코드 확인 | 정상: LB CNAME 또는 Alias 레코드 설정 <br> 취약: 잘못된 DNS 설정 | 클라이언트가 LB에 접근하지 못함 | Route 53에서 LB에 대한 CNAME/Alias 레코드 설정 확인 | 설정방법: SCP 콘솔 > Route 53 > Hosted Zones <br> References: [AWS Route 53 Documentation](https://docs.aws.amazon.com/Route53/) |
| **헬스 체크 간격** | SCP 콘솔에서 헬스 체크 간격 확인 | 정상: 5~30초 간격 설정 <br> 취약: 지나치게 짧거나 긴 간격 | 짧은 간격은 서버 부하 증가, 긴 간격은 장애 감지 지연 | 일반적으로 10초 간격 설정, 애플리케이션 요구사항에 맞게 조정 | 설정방법: SCP 콘솔 > Target Groups > Health Checks <br> References: [AWS Health Check Intervals](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html) |
| **헬스 체크 타임아웃** | SCP 콘솔에서 헬스 체크 타임아웃 확인 | 정상: 2~10초 설정 <br> 취약: 부적절한 타임아웃 설정 | 타임아웃이 짧아 정상 타겟 제외, 길면 장애 감지 지연 | 일반적으로 5초로 설정, 애플리케이션 응답 시간 고려 | 설정방법: SCP 콘솔 > Target Groups > Health Checks <br> References: [AWS Health Check Timeouts](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html) |
| **포트 매핑 설정** | SCP 콘솔에서 리스너 및 타겟 그룹 포트 확인 | 정상: 리스너와 타겟 그룹 포트 일치 <br> 취약: 포트 불일치 | 트래픽 라우팅 실패, 서비스 장애 발생 | 리스너와 타겟 그룹 포트 설정 일치 확인 | 설정방법: SCP 콘솔 > Load Balancer > Listeners <br> References: [AWS Listener Rules](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/listener-update-rules.html) |
| **VPC 설정** | SCP 콘솔에서 LB의 VPC 설정 확인 | 정상: LB와 타겟이 동일 VPC 내 위치 <br> 취약: 서로 다른 VPC | 타겟 접근 불가, 트래픽 라우팅 실패 | LB와 타겟을 동일 VPC 내에 배치 | 설정방법: SCP 콘솔 > Load Balancer > VPC Settings <br> References: [AWS VPC Documentation](https://docs.aws.amazon.com/vpc/) |
| **태그 설정** | SCP 콘솔에서 LB 및 타겟 그룹의 태그 확인 | 정상: 리소스 관리 및 모니터링을 위한 태그 설정 <br> 취약: 태그 누락 | 리소스 관리 및 비용 추적 어려움 | 환경, 애플리케이션, 소유자 등 태그 추가 | 설정방법: SCP 콘솔 > Load Balancer > Tags <br> References: [AWS Tagging Best Practices](https://docs.aws.amazon.com/general/latest/gr/aws_tagging.html) |
| **IAM 정책 설정** | SCP 콘솔 또는 CLI에서 IAM 정책 확인 (`aws iam list-policies`) | 정상: LB 관리에 필요한 최소 권한 설정 <br> 취약: 과도한 권한 부여 | 보안 취약점 발생, 권한 오용 가능성 | 최소 권한 원칙에 따라 IAM 정책 설정 | 설정방법: SCP 콘솔 > IAM > Policies <br> References: [AWS IAM Policies](https://docs.aws.amazon.com/IAM/latest/UserGuide/access_policies.html) |[](https://docs.aws.amazon.com/elasticloadbalancing/latest/userguide/load-balancer-authentication-access-control.html)
| **헬스 체크 응답 코드** | SCP 콘솔에서 헬스 체크 응답 코드 확인 | 정상: 200 OK 응답 코드 설정 <br> 취약: 부적절한 응답 코드 | 비정상 타겟으로 트래픽 라우팅 | 헬스 체크 응답 코드를 200으로 설정 | 설정방법: SCP 콘솔 > Target Groups > Health Checks <br> References: [AWS Health Check Response Codes](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/target-group-health-checks.html) |
| **Connection Draining** | SCP 콘솔에서 Connection Draining 설정 확인 | 정상: 적절한 드레이닝 시간 설정 (예: 300초) <br> 취약: 드레이닝 비활성화 또는 짧은 시간 | 타겟 해제 시 연결 끊김, 사용자 경험 저하 | Connection Draining 활성화, 적절한 시간 설정 | 설정방법: SCP 콘솔 > Target Groups > Attributes <br> References: [AWS Connection Draining](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/deregistration-delay.html) |
| **리스너 규칙 우선순위** | SCP 콘솔에서 리스너 규칙 우선순위 확인 | 정상: 명확한 우선순위 설정 <br> 취약: 중복 또는 모호한 규칙 | 트래픽 라우팅 오류 발생 | 리스너 규칙에 고유한 우선순위 설정 | 설정방법: SCP 콘솔 > Load Balancer > Listeners > Rules <br> References: [AWS Listener Rules](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/listener-update-rules.html) |
| **백업 설정** | SCP 콘솔 또는 CLI에서 LB 구성 백업 확인 | 정상: 주기적인 백업 설정 및 저장소 지정 <br> 취약: 백업 미설정 | 구성 손실 시 복구 불가 | LB 구성 백업을 S3 또는 외부 저장소에 주기적으로 저장 | 설정방법: CLI로 `aws elb describe-load-balancers` 출력 저장 <br> References: [AWS Backup Documentation](https://docs.aws.amazon.com/aws-backup/) |[](https://gandibleux.eu/blog/nsx/nsx-advanced-load-balancer-avi-vantage-upgrade-checklist/)

## 결함 및 오류

| 점검내용 | 점검방법 | 점검기준 | 취약시 문제점 | 개선방안 | 참고 |
|----------|----------|----------|--------------|----------|------|
