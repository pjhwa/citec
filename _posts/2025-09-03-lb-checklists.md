## 구성

| 점검 내용 | 점검 방법 | 점검 기준 | 취약 시 문제점 | 개선 방안 | 참고 |
|-----------|-----------|-----------|---------------|-----------|-------|
| **로드밸런서 유형 설정** | SCP 콘솔에서 LB 설정 확인 또는 `kubectl describe svc <service-name>` 실행 | 정상: Application LB 또는 Network LB가 워크로드에 맞게 설정됨<br>취약: 워크로드 요구사항과 맞지 않는 LB 유형 사용 | 부적절한 LB 유형 사용 시 성능 저하 또는 트래픽 처리 실패 발생 가능 | 워크로드 특성에 따라 ALB 또는 NLB 선택, HTTP 기반은 ALB, TCP/UDP는 NLB 권장 | * 설정방법: SCP 콘솔에서 LB 유형 변경 또는 `service.spec.type=LoadBalancer` 설정<br>* References: [Samsung Cloud Platform LB Overview](https://www.samsungsds.com/kr/cloud/service/scp_loadbalancer.html) |
| **헬스 체크 설정** | SCP 콘솔에서 헬스 체크 경로 및 포트 확인 또는 `kubectl describe svc`로 어노테이션 확인 | 정상: 헬스 체크 경로 및 포트가 백엔드 애플리케이션 상태를 정확히 반영<br>취약: 헬스 체크 경로 또는 포트 미설정 또는 부정확 | 잘못된 헬스 체크로 인해 정상 백엔드가 비정상으로 판단되거나, 비정상 백엔드가 트래픽 수신 가능 | 헬스 체크 경로를 애플리케이션의 상태 엔드포인트(예: `/health`)로 설정, 포트 일치 확인 | * 확인방법: `kubectl get svc -o yaml`로 `healthCheckNodePort` 확인<br>* References: [Kubernetes Service Health Check](https://kubernetes.io/docs/tasks/access-application-cluster/create-external-load-balancer/) |
| **포트 매핑 설정** | SCP 콘솔 또는 `kubectl get svc -o yaml`로 포트 설정 확인 | 정상: LB 포트와 백엔드 포트가 애플리케이션 요구사항에 맞게 매핑됨<br>취약: 포트 불일치 또는 불필요한 포트 오픈 | 포트 불일치 시 트래픽 전달 실패, 불필요한 포트 오픈 시 보안 위험 증가 | LB 포트와 백엔드 포트를 일치시키고, 최소한의 포트만 오픈 | * 설정방법: `service.spec.ports`에 `port`와 `targetPort` 명시<br>* References: [Kubernetes Service Ports](https://kubernetes.io/docs/concepts/services-networking/service/#defining-a-service) |
| **세션 어피니티 설정** | SCP 콘솔에서 세션 유지 설정 확인 또는 `kubectl get svc -o yaml`로 `sessionAffinity` 확인 | 정상: `sessionAffinity=ClientIP` 또는 애플리케이션 요구사항에 맞는 설정<br>취약: 세션 어피니티 미설정으로 세션 불일치 발생 | 세션 불일치로 인해 사용자 경험이 저하되거나 애플리케이션 오류 발생 | 애플리케이션 요구사항에 따라 `sessionAffinity=ClientIP` 설정 또는 쿠키 기반 세션 유지 활성화 | * 설정방법: `service.spec.sessionAffinity=ClientIP`<br>* References: [Kubernetes Session Affinity](https://kubernetes.io/docs/concepts/services-networking/service/#service-session-affinity) |
| **외부 트래픽 정책** | `kubectl get svc -o yaml`로 `externalTrafficPolicy` 확인 | 정상: `externalTrafficPolicy=Local` 또는 `Cluster`가 요구사항에 맞게 설정<br>취약: 부적절한 트래픽 정책으로 성능 저하 | `Cluster` 모드에서 불필요한 노드 간 트래픽 발생, `Local` 모드에서 헬스 체크 실패 시 트래픽 손실 | 워크로드 특성에 따라 `Local` 또는 `Cluster` 선택, `Local` 사용 시 헬스 체크 강화 | * 설정방법: `service.spec.externalTrafficPolicy=Local`<br>* References: [Kubernetes External Traffic Policy](https://kubernetes.io/docs/tasks/access-application-cluster/create-external-load-balancer/#preserving-the-client-source-ip) |
| **로드밸런서 보안 그룹 설정** | SCP 콘솔에서 LB와 연결된 보안 그룹 확인 | 정상: LB 보안 그룹이 허용된 IP/포트만 오픈<br>취약: 모든 IP(0.0.0.0/0)로 포트 오픈 | 불필요한 포트 오픈으로 보안 취약점 노출 | 최소 권한 원칙에 따라 보안 그룹 설정, 특정 CIDR 및 포트만 허용 | * 설정방법: SCP 콘솔에서 보안 그룹 규칙 수정<br>* References: [SCP Network Security](https://www.samsungsds.com/kr/cloud/security.html) |
| **SSL/TLS 인증서 설정** | SCP 콘솔에서 LB의 SSL 인증서 확인 | 정상: 유효한 SSL/TLS 인증서가 LB에 적용됨<br>취약: 인증서 만료 또는 미적용 | 인증서 문제로 HTTPS 연결 실패 또는 보안 취 # | 유효한 SSL 인증서를 적용하고, 만료일 모니터링 설정 | * 설정방법: SCP 콘솔에서 인증서 업로드 및 연결<br>* References: [SCP SSL Configuration](https://www.samsungsds.com/kr/cloud/service/scp_loadbalancer.html) |
| **프로토콜 설정** | SCP 콘솔 또는 `kubectl get svc -o yaml`로 프로토콜 확인 | 정상: HTTP/HTTPS 또는 TCP/UDP가 애플리케이션에 맞게 설정<br>취약: 잘못된 프로토콜 사용 | 잘못된 프로토콜로 인해 트래픽 처리 실패 | 애플리케이션 요구사항에 따라 적절한 프로토콜 선택 | * 설정방법: `service.spec.ports.protocol=TCP`<br>* References: [Kubernetes Service Protocols](https://kubernetes.io/docs/concepts/services-networking/service/#protocol-support) |
| **백엔드 인스턴스 등록** | SCP 콘솔 또는 `kubectl describe svc`로 백엔드 상태 확인 | 정상: 모든 백엔드 인스턴스가 정상 등록됨<br>취약: 등록되지 않은 인스턴스 존재 | 등록되지 않은 인스턴스로 인해 트래픽 분배 실패 | `selector` 설정 확인 및 일치하는 파드 레이블 설정 | * 확인방법: `kubectl get endpoints`로 백엔드 확인<br>* References: [Kubernetes Endpoints](https://kubernetes.io/docs/concepts/services-networking/service/#services-without-selectors) |
| **헬스 체크 주기 및 임계값** | SCP 콘솔에서 헬스 체크 주기 및 임계값 확인 | 정상: 주기(예: 10초)와 임계값(예: 3회)이 적절히 설정<br>취약: 지나치게 짧거나 긴 주기 설정 | 부정확한 헬스 체크로 정상 인스턴스 제외 또는 비정상 인스턴스 포함 | 주기를 5~10초, 임계값을 2~3회로 설정 | * 설정방법: SCP 콘솔에서 헬스 체크 설정 수정<br>* References: [Azure Load Balancer Health Checks](https://learn.microsoft.com/ko-kr/azure/load-balancer/load-balancer-custom-probe-overview) |[](https://learn.microsoft.com/en-us/azure/well-architected/service-guides/azure-load-balancer)
| **로드밸런서 리전 설정** | SCP 콘솔에서 LB 리전 확인 | 정상: LB가 애플리케이션 리전과 동일<br>취약: 다른 리전 설정 | 리전 불일치로 인해 지연 시간 증가 | LB를 애플리케이션과 동일 리전에 배포 | * 설정방법: SCP 콘솔에서 LB 리전 설정<br>* References: [SCP Region Configuration](https://www.samsungsds.com/kr/cloud/service/scp_loadbalancer.html) |
| **트래픽 분배 알고리즘** | SCP 콘솔 또는 어노테이션으로 알고리즘 확인 | 정상: Round Robin 또는 Least Connection 설정<br>취약: 부적절한 알고리즘 사용 | 트래픽 불균형으로 특정 인스턴스 과부하 | 워크로드 특성에 맞는 알고리즘 선택 | * 확인방법: SCP 콘솔에서 LB 설정 확인<br>* References: [Load Balancing Algorithms](https://www.cloudflare.com/learning/performance/types-of-load-balancing-algorithms/) |[](https://www.cloudflare.com/learning/performance/types-of-load-balancing-algorithms/)
| **CORS 설정** | SCP 콘솔에서 CORS 정책 확인 | 정상: 필요한 도메인만 CORS 허용<br>취약: 모든 도메인(*) 허용 | 보안 취약점으로 인해 악의적인 요청 허용 가능 | 특정 도메인만 허용하도록 CORS 설정 | * 설정방법: SCP 콘솔에서 CORS 정책 수정<br>* References: [CORS Configuration](https://developer.mozilla.org/ko/docs/Web/HTTP/CORS) |
| **WAF(Web Application Firewall) 통합** | SCP 콘솔에서 WAF 설정 확인 | 정상: WAF가 활성화되고 규칙 설정됨<br>취약: WAF 미적용 | 웹 애플리케이션 공격(예: SQL 인젝션)에 취약 | WAF 활성화 및 OWASP Top 10 규칙 적용 | * 설정방법: SCP 콘솔에서 WAF 설정<br>* References: [SCP WAF](https://www.samsungsds.com/kr/cloud/security.html) |
| **타임아웃 설정** | SCP 콘솔에서 연결 및 유휴 타임아웃 확인 | 정상: 타임아웃 값이 애플리케이션 요구사항에 맞게 설정<br>취약: 지나치게 짧거나 긴 타임아웃 | 연결 끊김 또는 리소스 낭비 발생 | 연결 타임아웃 60초, 유휴 타임아웃 300초로 설정 | * 설정방법: SCP 콘솔에서 타임아웃 설정<br>* References: [AWS ALB Timeout](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/application-load-balancers.html#connection-idle-timeout) |
| **로그 활성화** | SCP 콘솔에서 LB 액세스 로그 설정 확인 | 정상: 액세스 로그가 활성화됨<br>취약: 로그 비활성화 | 문제 진단 및 보안 감사 불가 | LB 액세스 로그 활성화 및 로그 저장소 설정 | * 설정방법: SCP 콘솔에서 로그 설정<br>* References: [AWS ELB Logging](https://docs.aws.amazon.com/elasticloadbalancing/latest/application/load-balancer-access-logs.html) |
| **DNS 설정** | SCP 콘솔 또는 DNS 관리 툴에서 CNAME/A 레코드 확인 | 정상: LB의 DNS 레코드가 정확히 설정됨<br>취약: DNS 레코드 불일치 | 트래픽이 LB로 전달되지 않음 | CNAME 또는 A 레코드를 LB의 공인 IP 또는 도메인에 맞게 설정 | * 확인방법: `nslookup <lb-domain>`<br>* References: [Route 53 Configuration](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/dns-configuring.html) |
| **IP 화이트리스트** | SCP 콘솔에서 LB 접근 허용 IP 확인 | 정상: 특정 IP 또는 CIDR만 허용<br>취약: 모든 IP(0.0.0.0/0) 허용 | 보안 취약점으로 무단 접근 가능 | 특정 IP/CIDR만 허용하도록 설정 | * 설정방법: SCP 콘솔에서 보안 그룹 수정<br>* References: [SCP Network Security](https://www.samsungsds.com/kr/cloud/security.html) |
| **헬스 체크 프로토콜** | SCP 콘솔에서 헬스 체크 프로토콜 확인 | 정상: HTTP/TCP가 백엔드와 일치<br>취약: 프로토콜 불일치 | 헬스 체크 실패로 정상 백엔드 제외 | 백엔드 애플리케이션 프로토콜과 일치하도록 설정 | * 설정방법: SCP 콘솔에서 헬스 체크 프로토콜 설정<br>* References: [Azure LB Health Probes](https://learn.microsoft.com/ko-kr/azure/load-balancer/load-balancer-custom-probe-overview) |[](https://learn.microsoft.com/en-us/azure/well-architected/service-guides/azure-load-balancer)
| **스케일링 정책** | SCP 콘솔 또는 `kubectl get hpa`로 스케일링 설정 확인 | 정상: 오토스케일링 정책이 LB와 연동됨<br>취약: 스케일링 정책 미설정 | 트래픽 급증 시 성능 저하 | Horizontal Pod Autoscaler(HPA) 설정 및 LB 연동 | * 설정방법: `kubectl autoscale deployment <name> --min=2 --max=10`<br>* References: [Kubernetes HPA](https://kubernetes.io/docs/tasks/run-application/horizontal-pod-autoscale/) |
| **리소스 태그 설정** | SCP 콘솔에서 LB 리소스 태그 확인 | 정상: 환경, 소유자 등 태그가 적용됨<br>취약: 태그 미적용 | 리소스 관리 및 비용 추적 어려움 | 환경, 팀, 애플리케이션 이름 등 태그 적용 | * 설정방법: SCP 콘솔에서 태그 추가<br>* References: [AWS Tagging Best Practices](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_tagging.html) |[](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps_examples_tagging.html)
| **백엔드 풀 크기** | SCP 콘솔 또는 `kubectl get pods`로 백엔드 확인 | 정상: 최소 2개 이상의 백엔드 인스턴스<br>취약: 단일 인스턴스 | 단일 장애점으로 서비스 중단 위험 | 최소 2개 이상의 파드 배포 | * 확인방법: `kubectl get pods -l app=<app-name>`<br>* References: [Kubernetes High Availability](https://kubernetes.io/docs/setup/production-environment/) |
| **헬스 체크 응답 코드** | SCP 콘솔에서 헬스 체크 응답 코드 확인 | 정상: 200 또는 애플리케이션에 맞는 코드<br>취약: 부정확한 응답 코드 | 헬스 체크 실패로 정상 백엔드 제외 | 애플리케이션의 정상 응답 코드(예: 200) 설정 | * 설정방법: SCP 콘솔에서 헬스 체크 응답 코드 설정<br>* References: [Azure LB Health Probes](https://learn.microsoft.com/ko-kr/azure/load-balancer/load-balancer-custom-probe-overview) |[](https://learn.microsoft.com/en-us/azure/well-architected/service-guides/azure-load-balancer)
| **로드밸런서 이름 규칙** | SCP 콘솔에서 LB 이름 확인 | 정상: 일관된 명명 규칙 준수<br>취약: 비표준 이름 사용 | 리소스 관리 및 식별 어려움 | 환경-애플리케이션-역할 형식의 명명 규칙 적용 | * 설정방법: SCP 콘솔에서 LB 이름 수정<br>* References: [AWS Naming Best Practices](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_tagging.html) |
| **모니터링 통합** | SCP 콘솔에서 모니터링 설정 확인 | 정상: SCP 모니터링 또는 외부 모니터링 도구 통합<br>취약: 모니터링 미설정 | LB 상태 및 성능 문제 파악 불가 | SCP 모니터링 또는 Prometheus 통합 설정 | * 설정방법: SCP 콘솔에서 모니터링 활성화<br>* References: [Prometheus Monitoring](https://prometheus.io/docs/introduction/overview/) |
