| 점검내용 | 점검방법 | 점검기준 | 취약시 문제점 | 개선방안 | 참고 |
|----------|----------|----------|----------|--------------|----------|------|
| Kubernetes 클러스터 API 서버의 인증 모드 설정 | kubectl get configmap -n kube-system kube-apiserver -o yaml | 정상: 인증 모드가 Node, RBAC 등 활성화됨<br>취약: 인증 모드가 Anonymous 또는 AlwaysAllow로 설정됨 | 인증 모드가 취약하게 설정되어 무단 접근이 가능해 클러스터 전체가 노출되며, 악의적 행위자가 클러스터 리소스를 조작하거나 데이터를 유출할 수 있음 | API 서버의 인증 모드를 Node와 RBAC로 설정하여 인증을 강화함. kube-apiserver 설정에서 --authorization-mode=Node,RBAC 옵션을 추가함 | * 설정방법<br>kubectl edit configmap -n kube-system kube-apiserver<br>--authorization-mode=Node,RBAC 추가<br>* References<br>- https://kubernetes.io/docs/reference/access-authn-authz/authorization/ |
| RBAC (Role-Based Access Control) 활성화 여부 | kubectl get clusterrolebindings | 정상: 기본 RBAC 정책이 적용되고 최소 권한 원칙 준수됨<br>취약: RBAC가 비활성화되거나 과도한 권한 부여됨 | RBAC가 취약하면 사용자나 서비스 어카운트가 불필요한 권한을 가져 클러스터 내 리소스에 무단 접근이 발생하며, 보안 침해가 확대됨 | RBAC를 활성화하고, Role과 ClusterRole을 정의하여 최소 권한만 부여함. 불필요한 cluster-admin 권한을 제거함 | * 설정방법<br>kubectl create rolebinding ...<br>* References<br>- https://kubernetes.io/docs/reference/access-authn-authz/rbac/ |
| 네트워크 정책 (NetworkPolicy) 적용 여부 | kubectl get networkpolicies --all-namespaces | 정상: 모든 네임스페이스에 네트워크 정책이 정의됨<br>취약: 네트워크 정책이 없거나 기본 허용 정책임 | 네트워크 정책이 없으면 Pod 간 트래픽이 무제한으로 허용되어 보안 취약점이 발생하며, 공격자가 내부 네트워크를 횡단 이동할 수 있음 | NetworkPolicy를 생성하여 ingress/egress 트래픽을 제한함. 기본 deny 정책을 적용함 | * 설정방법<br>kubectl apply -f networkpolicy.yaml<br>* References<br>- https://kubernetes.io/docs/concepts/services-networking/network-policies/ |
| Pod Security Admission (PSA) 또는 Pod Security Policies (PSP) 설정 | kubectl get podsecuritypolicies | 정상: Restricted 또는 Baseline 정책 적용됨<br>취약: Privileged 모드 허용 또는 PSP 비활성화됨 | PSP가 취약하면 Pod가 루트 권한으로 실행되어 호스트 시스템에 접근 가능하며, 컨테이너 탈출 공격이 발생할 수 있음 | PSA를 활성화하고, Restricted 정책을 네임스페이스에 적용함. Privileged 컨테이너를 금지함 | * 설정방법<br>kubectl label --overwrite ns <namespace> pod-security.kubernetes.io/enforce=restricted<br>* References<br>- https://kubernetes.io/docs/concepts/security/pod-security-admission/ |
| 리소스 쿼터 (ResourceQuota) 설정 | kubectl get resourcequotas --all-namespaces | 정상: CPU, 메모리 쿼터가 네임스페이스별로 설정됨<br>취약: 쿼터가 없어 리소스 과다 사용 가능 | 쿼터 미설정 시 하나의 워크로드가 클러스터 리소스를 독점하여 다른 Pod의 안정성을 저해하며, DoS 공격 유발 가능 | ResourceQuota를 생성하여 네임스페이스별 리소스 한도를 설정함 | * 설정방법<br>kubectl apply -f resourcequota.yaml<br>* References<br>- https://kubernetes.io/docs/concepts/policy/resource-quotas/ |
| 리밋 레인지 (LimitRange) 적용 | kubectl get limitranges --all-namespaces | 정상: Pod/Container별 리소스 리밋과 리퀘스트 설정됨<br>취약: 리밋이 없어 과도한 리소스 할당 가능 | 리밋 미설정 시 Pod가 무제한 리소스를 요청하여 노드 불안정성을 초래하며, 클러스터 전체 성능 저하됨 | LimitRange를 네임스페이스에 적용하여 기본 리밋과 리퀘스트를 정의함 | * 설정방법<br>kubectl apply -f limitrange.yaml<br>* References<br>- https://kubernetes.io/docs/concepts/policy/limit-range/ |
| 노드 셀렉터 (NodeSelector) 및 어피니티 (Affinity) 설정 | kubectl describe deployments | 정상: 워크로드가 특정 노드에 배치되도록 설정됨<br>취약: 노드 셀렉터 미설정으로 임의 배치됨 | 미설정 시 워크로드가 부적합한 노드에 배치되어 성능 저하나 장애 발생 가능 | Deployment spec에 nodeSelector 또는 affinity 규칙을 추가함 | * 설정방법<br>kubectl edit deployment <name><br>* References<br>- https://kubernetes.io/docs/concepts/scheduling-eviction/assign-pod-node/ |
| 테인트와 톨러레이션 (Taints and Tolerations) 사용 | kubectl describe nodes | 정상: 노드에 테인트 적용하고 Pod에 톨러레이션 설정됨<br>취약: 테인트 미적용으로 무단 Pod 배치 | 테인트 미사용 시 중요한 노드에 일반 워크로드가 배치되어 리소스 경쟁 발생 | 노드에 taint를 추가하고 Pod spec에 tolerations를 설정함 | * 설정방법<br>kubectl taint nodes <node> key=value:NoSchedule<br>* References<br>- https://kubernetes.io/docs/concepts/scheduling-eviction/taint-and-toleration/ |
| 서비스 어카운트 (ServiceAccount) 관리 | kubectl get serviceaccounts --all-namespaces | 정상: 기본 SA 사용 금지, 커스텀 SA 사용<br>취약: default SA에 과도한 권한 부여 | default SA 사용 시 불필요한 권한이 부여되어 보안 취약점 발생 | 커스텀 ServiceAccount를 생성하고 RBAC 바인딩 | * 설정방법<br>kubectl create serviceaccount <name><br>* References<br>- https://kubernetes.io/docs/tasks/configure-pod-container/configure-service-account/ |
| 시크릿 (Secrets) 암호화 설정 | kubectl get secrets --all-namespaces | 정상: etcd에 Secrets가 암호화되어 저장됨<br>취약: Secrets가 평문으로 저장됨 | 평문 저장 시 etcd 접근 시 민감 데이터 유출 가능 | EncryptionConfiguration을 API 서버에 적용하여 Secrets 암호화 | * 설정방법<br>kube-apiserver --encryption-provider-config 설정<br>* References<br>- https://kubernetes.io/docs/tasks/administer-cluster/encrypt-data/ |
| 컨피그맵 (ConfigMaps) 사용 | kubectl get configmaps --all-namespaces | 정상: 민감 데이터는 Secrets로 이동<br>취약: ConfigMap에 민감 정보 포함 | 민감 정보 포함 시 ConfigMap 접근으로 데이터 유출 | 민감 데이터를 Secrets로 분리하고 ConfigMap은 설정값만 사용 | * 설정방법<br>kubectl create secret generic <name><br>* References<br>- https://kubernetes.io/docs/concepts/configuration/configmap/ |
| 인그레스 (Ingress) 컨트롤러 설정 | kubectl get ingress --all-namespaces | 정상: TLS 종료와 인증 적용됨<br>취약: HTTP만 사용 또는 인증 미적용 | HTTP 사용 시 트래픽 도청 가능, 인증 미적용으로 무단 접근 | Ingress에 TLS secret과 annotation으로 인증 추가 | * 설정방법<br>kubectl apply -f ingress.yaml<br>* References<br>- https://kubernetes.io/docs/concepts/services-networking/ingress/ |
| 로드 밸런서 (LoadBalancer) 타입 서비스 설정 | kubectl get services --all-namespaces | 정상: SCP LB와 연동되어 보안 그룹 적용됨<br>취약: 공용 IP 직접 노출 | 공용 노출 시 DDoS 공격 취약 | Service type을 LoadBalancer로 설정하고 SCP LB 보안 규칙 적용 | * 설정방법<br>kubectl expose deployment <name> --type=LoadBalancer<br>* References<br>- https://kubernetes.io/docs/concepts/services-networking/service/#loadbalancer |
| 방화벽 규칙 (Firewall Rules) 적용 | SCP 콘솔에서 네트워크 방화벽 확인 | 정상: 인바운드/아웃바운드 규칙이 최소 허용<br>취약: 모든 포트 개방 | 모든 포트 개방 시 무단 접근 가능 | SCP 방화벽에서 필요 포트만 허용 규칙 추가 | * 설정방법<br>SCP 콘솔 > Network > Firewall > 규칙 추가<br>* References<br>- https://www.samsungcloud.com/ (SCP 문서 가정, 실제 확인 필요; 일반: https://cloud.google.com/vpc/docs/firewalls) |
| 클러스터 오토스케일링 (Cluster Autoscaler) 설정 | kubectl get configmaps -n kube-system cluster-autoscaler-status | 정상: 노드 풀에 autoscaler 활성화됨<br>취약: autoscaler 미설정으로 수동 스케일링 | 미설정 시 리소스 부족 시 장애 발생 | ClusterAutoscaler를 배포하고 노드 풀에 min/max 설정 | * 설정방법<br>kubectl apply -f cluster-autoscaler.yaml<br>* References<br>- https://kubernetes.io/docs/tasks/administer-cluster/cluster-management/#automatic-scaling |
| 노드 풀 (Node Pool) 구성 | kubectl get nodes | 정상: 다중 노드 풀로 워크로드 분리<br>취약: 단일 노드 풀 사용 | 단일 풀 사용 시 장애 시 전체 영향 | 여러 노드 풀 생성하고 라벨링 | * 설정방법<br>SCP 콘솔 > SKE > Node Pool 추가<br>* References<br>- https://kubernetes.io/docs/concepts/architecture/nodes/ |
| 클렛 (Kubelet) 설정 | kubectl get daemonsets -n kube-system kubelet | 정상: --anonymous-auth=false 등 보안 옵션 적용<br>취약: 기본 설정으로 익명 접근 허용 | 익명 접근 시 보안 취약 | Kubelet config 파일에 보안 옵션 추가 | * 설정방법<br>/var/lib/kubelet/config.yaml 편집<br>* References<br>- https://kubernetes.io/docs/reference/command-line-tools-reference/kubelet/ |
| 컨트롤 플레인 고가용성 (High Availability) | kubectl get nodes -l node-role.kubernetes.io/master | 정상: 3개 이상 마스터 노드<br>취약: 단일 마스터 | 단일 마스터 시 장애 발생으로 클러스터 다운 | HA 클러스터로 구성 | * 설정방법<br>SCP SKE에서 HA 옵션 선택<br>* References<br>- https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/high-availability/ |
| 모니터링 통합 (Monitoring Integration) | kubectl get deployments -n monitoring | 정상: Prometheus 등 모니터링 도구 배포됨<br>취약: 모니터링 미설정 | 미설정 시 문제 조기 발견 어려움 | Prometheus와 Grafana 배포 | * 설정방법<br>helm install prometheus<br>* References<br>- https://kubernetes.io/docs/tasks/debug/debug-cluster/resource-metrics-pipeline/ |
| 로깅 설정 (Logging) | kubectl get daemonsets -n kube-system fluentd | 정상: 중앙 로깅 시스템 연결됨<br>취약: 로컬 로깅만 | 로컬만 시 로그 손실 가능 | Fluentd 또는 ELK 스택 배포 | * 설정방법<br>kubectl apply -f fluentd.yaml<br>* References<br>- https://kubernetes.io/docs/concepts/cluster-administration/logging/ |
| 백업 정책 (Backup Policies) | kubectl get cronjobs | 정상: etcd 백업 크론잡 설정됨<br>취약: 백업 미설정 | 미설정 시 데이터 손실 | Velero 등 백업 도구 설치 | * 설정방법<br>kubectl apply -f backup-cronjob.yaml<br>* References<br>- https://kubernetes.io/docs/tasks/administer-cluster/configure-upgrade-etcd/#backing-up-an-etcd-cluster |
| 업그레이드 정책 (Upgrade Policies) | SCP 콘솔에서 클러스터 버전 확인 | 정상: 자동 업그레이드 활성화됨<br>취약: 수동 업그레이드 | 수동 시 보안 패치 지연 | SKE에서 자동 업그레이드 설정 | * 설정방법<br>SCP 콘솔 > SKE > Upgrade 설정<br>* References<br>- https://kubernetes.io/docs/tasks/administer-cluster/cluster-upgrade/ |
| 네트워크 CIDR 설정 | kubectl cluster-info dump | grep podCIDR | 정상: 충분한 CIDR 범위 할당됨<br>취약: CIDR 부족 | 부족 시 Pod IP 고갈 | 클러스터 생성 시 넓은 CIDR 지정 | * 설정방법<br>SCP SKE 생성 시 CIDR 설정<br>* References<br>- https://kubernetes.io/docs/concepts/networking/networking-overview/ |
| 애드온 (Addons) 관리 | kubectl get deployments -n kube-system | 정상: 필수 애드온만 활성화<br>취약: 불필요 애드온 설치 | 불필요 시 리소스 낭비 및 취약점 | SKE 애드온 관리에서 선택 | * 설정방법<br>SCP 콘솔 > SKE > Addons<br>* References<br>- https://kubernetes.io/docs/concepts/cluster-administration/addons/ |
| 어드미션 플러긴 (Admission Plugins) 활성화 | kubectl get configmap -n kube-system kube-apiserver | 정상: PodSecurityPolicy, NodeRestriction 등 활성화<br>취약: 기본 플러긴 only | 미활성화 시 보안 제어 부족 | --enable-admission-plugins 옵션 추가 | * 설정방법<br>kube-apiserver --enable-admission-plugins=NodeRestriction,PodSecurityPolicy<br>* References<br>- https://kubernetes.io/docs/reference/access-authn-authz/admission-controllers/
